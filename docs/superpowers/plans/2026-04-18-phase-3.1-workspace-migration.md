# Phase 3.1: App Insights Workspace Migration

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate App Insights from its managed workspace to `shared-services-logs` so Cosmos diagnostics and App Insights telemetry share one user-controlled workspace. Remove the `COSMOS_DIAGNOSTICS_WORKSPACE_ID` workaround.

**Background:** Phase 3 discovered that the App Insights managed workspace (`managed-second-brain-insights-ws`) has a deny assignment blocking `Microsoft.OperationalInsights/workspaces/sharedKeys/action`, which prevents routing Cosmos diagnostic settings to it. This is a documented Microsoft platform constraint. As a workaround, Cosmos diagnostic logs were routed to `shared-services-logs` with a separate workspace ID env var. This phase unifies everything onto one workspace.

**Risk:** Historical App Insights data remains in the old managed workspace. New telemetry flows to `shared-services-logs` after migration. Queries against recent data work immediately; historical queries need the old workspace until retention expires.

**Spec reference:** `docs/superpowers/specs/2026-04-14-per-segment-observability-design.md`

**Phase 3 dependency:** All Phase 3 tasks must be complete and deployed.

## Task 1: Pre-migration validation

- [ ] **Step 1: Check shared-services-logs workspace configuration**

```bash
az monitor log-analytics workspace show -g shared-services-rg -n shared-services-logs \
  --query "{sku: sku.name, retentionDays: retentionInDays, dailyCapGb: workspaceCapping.dailyQuotaGb}" \
  -o json
```

Verify retention and caps are acceptable. Compare with current App Insights settings:

```bash
az monitor app-insights component show -g shared-services-rg --app second-brain-insights \
  --query "{retentionDays: RetentionInDays, dailyCapGb: Cap.Cap}" \

```

If `shared-services-logs` has lower retention or tighter caps than needed, adjust before proceeding.

- [ ] **Step 2: Verify shared-services-logs workspace ID**

```bash

  --query customerId -o tsv
```

Expected: `572d91c2-3209-4b92-b431-5ffb7e8ce4ad`

- [ ] **Step 3: Snapshot current App Insights workspace ID for rollback**


Save this value — needed for rollback if migration fails.

## Task 2: Migrate App Insights to shared-services-logs

- [ ] **Step 1: Re-point App Insights**

```bash
WS_RESOURCE_ID=$(az monitor log-analytics workspace show -g shared-services-rg -n shared-services-logs --query id -o tsv)

az monitor app-insights component update \
  -g shared-services-rg \
  --app second-brain-insights \
  --workspace "$WS_RESOURCE_ID" \
  -o table
```

- [ ] **Step 2: Verify migration**

```bash
az monitor app-insights component show -g shared-services-rg --app second-brain-insights \
  --query workspaceResourceId -o tsv
```

Expected: should now point to `shared-services-logs`.

- [ ] **Step 3: Wait for telemetry to flow**

Wait 5 minutes, then verify new data appears in the workspace:

```bash
az monitor log-analytics query \
  --workspace "572d91c2-3209-4b92-b431-5ffb7e8ce4ad" \
  --analytics-query "AppTraces | summarize count() by bin(TimeGenerated, 1m) | order by TimeGenerated desc | take 5" \
  --timespan PT15M -o json
```

Expected: non-zero counts in recent time bins.

## Rollback

If queries break after migration:

```bash
# Re-point App Insights back to managed workspace
az monitor app-insights component update \
  -g shared-services-rg \
  --app second-brain-insights \
  --workspace "<saved-managed-workspace-resource-id>"

# Restore Container App env vars
az containerapp update -n second-brain-api -g shared-services-rg \
  --set-env-vars \
    "LOG_ANALYTICS_WORKSPACE_ID=<old-workspace-customer-id>" \
    "COSMOS_DIAGNOSTICS_WORKSPACE_ID=572d91c2-3209-4b92-b431-5ffb7e8ce4ad"
```
