# Session Handoff — MCP Cutover + Parity Cleanup

Today's session finished the Phase 4 loose ends and made a strategic pivot on the MCP tool architecture. **Parity was deleted.** The legacy-vs-spine parity workflow (`.github/workflows/mcp-parity.yml`), the `parity_check.py` script, the `mcp_parity/` runner module, canonical queries, history directory, and all 4 runner tests are gone (commit `1aeea10`). The underlying reason: the 7-day-green gate was guarding a cutover that you decided to make by fiat instead of by comparison — exact-value parity between independently-populated data sources was unreachable anyway, and `system_health` in particular answers a different question on each path (ops metrics vs. traffic lights).

**MCP tool cutover shipped** (commit `7450e07`): `recent_errors`, `trace_lifecycle`, and `admin_audit` now read spine directly — no feature flag, no fallback. `_spine_enabled_for`, the three `_legacy_*` helpers, and the now-unused legacy query imports are all gone. `system_health` is pinned to legacy permanently (documented in its docstring — the spine `/status` endpoint tracks traffic lights, not latency histograms). `usage_patterns` and `run_kql` remain legacy-only for the same reason (spine has no usage aggregation or raw-KQL endpoint). Live-verified the spine code paths return real data from the deployed backend: `recent_errors` saw 2 exceptions, `admin_audit` 0 agent runs. 336 tests pass.

**Along the way, several infrastructure items got fixed.** The CI service principal (`oid=9727a024-ec7e-4cdf-ae7b-822a1ded819d`) was granted `Key Vault Secrets User` on `wkm-shared-kv` and `Log Analytics Reader` on `shared-services-logs` — it had neither before, which is why the parity workflow was failing on `Forbidden`. The workflow itself also had a silent-failure bug (`tee` was masking the script's non-zero exit) which was fixed with `set -euo pipefail` before the workflow was deleted — worth knowing if you ever reconstruct similar CI plumbing. The MCP server's `backend/.env` had a stale `LOG_ANALYTICS_WORKSPACE_ID` from before the Phase 3.1 workspace unification; fixed locally (uncommitted — `.env` is gitignored). `/api/spine/status` confirmed 9 segments live and green, including the new `mobile_ui` and `mobile_capture` from Phase 4. Also committed the uncommitted web-layer correlation wiring from the handoff (commit `2ac45ec`): `web/app/segment/[id]/page.tsx`, `web/app/correlation/[kind]/[id]/page.tsx` (new route), and both renderers now thread `correlation_kind` / `correlation_id` through to the detail fetch and expose trace/thread links — the UI can now actually use the transaction-scoped correlation the backend already supported.
