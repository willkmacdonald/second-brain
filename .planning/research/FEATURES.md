# Feature Landscape: v3.1 Observability & Evals

**Domain:** AI agent observability, natural language investigation over telemetry, LLM evaluation frameworks
**Researched:** 2026-04-04
**Overall confidence:** MEDIUM-HIGH (Azure AI Foundry eval SDK verified via official docs, MCP/agent patterns well-documented, mobile UX patterns are standard)

---

## Context: What Exists Today

The Second Brain app already has substantial observability infrastructure built in v3.0 Phase 14:

- **Per-capture trace IDs** generated by the mobile app, propagated via `X-Trace-Id` header through the full backend pipeline into App Insights custom dimensions
- **Structured logging** with component tags (`classifier`, `admin_agent`, `admin_handoff`, `mobile`) and severity levels
- **Four KQL queries** in `backend/queries/`: capture-trace, recent-failures, system-health, admin-agent-audit
- **Azure Monitor alerts** for API error spikes, capture processing failures, and health checks
- **Mobile trace ID display** with tap-to-copy on capture detail cards
- **Telemetry proxy endpoint** (`POST /api/telemetry`) for mobile client error reporting
- **App Insights** workspace-based instance (`second-brain-insights`) with OTel spans and token usage tracking

What does NOT exist:
- Any way to query this data without opening the Azure Portal and writing KQL manually
- Any measurement of agent quality (classifier accuracy, admin agent correctness)
- Any feedback mechanism from the user about whether classifications or errand routing were correct
- Any automated evaluation pipeline
- Any alerting on quality degradation (only infrastructure health alerts exist today)

v3.1 adds three capabilities: (1) a conversational investigation agent that translates natural language into KQL queries over App Insights, (2) an eval framework that measures and tracks agent quality, and (3) a self-monitoring loop that alerts when quality degrades.

## Table Stakes

Features users expect from an observability and evaluation system for an AI-powered personal app. Missing any of these means the milestone feels incomplete relative to its stated goal: "the system watches itself."

### TS-1: Natural Language Investigation Agent (Backend)

| Attribute | Detail |
|-----------|--------|
| **Why expected** | Core premise of v3.1 -- ability to ask "what happened to my milk capture?" without writing KQL |
| **Complexity** | Medium-High |
| **Dependencies** | Existing App Insights workspace, Azure OpenAI GPT-4o, existing KQL queries as templates |
| **Notes** | The investigation agent translates natural language questions into KQL, executes them against the Log Analytics workspace, and returns human-readable summaries. Two implementation paths: (A) Azure MCP Server's built-in `query_workspace_logs` tool -- requires workspace ID, resource group, table name, and KQL as parameters; (B) Custom agent with `azure-monitor-query` Python SDK executing KQL directly. Path B is preferred because it gives full control over the KQL generation prompt, can inject schema context (table names, custom dimension fields like `capture_trace_id`, `component`), and avoids the overhead of configuring the Azure MCP Server for a single-purpose use case. The agent needs a system prompt loaded with: (1) available tables (traces, requests, dependencies, exceptions), (2) custom dimension fields used by Second Brain, (3) the four existing KQL query patterns as few-shot examples, and (4) knowledge of the trace ID propagation flow. |

**Expected user behavior:** Will asks "show me recent errors" and gets a summary of error-level logs from the last 24h with trace IDs and component attribution. Will asks "what happened to trace abc-123?" and gets the full lifecycle of that capture. Will asks "how many captures happened today?" and gets a count with hourly breakdown.

**Critical design decision:** The agent generates AND executes KQL -- it does not just show the query and make the user run it. The response is a natural language summary of the results, not raw JSON.

### TS-2: Mobile Chat Interface for Investigation Agent

| Attribute | Detail |
|-----------|--------|
| **Why expected** | Will's primary device is his phone; the investigation agent must be accessible from the mobile app |
| **Complexity** | Medium |
| **Dependencies** | TS-1 (backend agent), existing AG-UI SSE infrastructure |
| **Notes** | A new screen (not a tab -- accessible from the Status screen or a header button) with a chat-style interface. Uses the same SSE streaming infrastructure as the capture flow. The key difference from capture: this is a multi-turn conversation, not a one-shot capture. The agent maintains context across turns ("show me more detail on the third one" should work). Thread management can reuse the existing Foundry thread pattern. |

**Expected user behavior:** Will taps an "Investigate" button (magnifying glass icon in the Status screen header). A chat screen opens. Will types or speaks a question. The agent streams a response showing the investigation results. Will can ask follow-up questions in the same thread. The conversation persists within the session but does not need to survive app restarts.

**UX patterns:**
- Chat bubbles: user messages right-aligned, agent responses left-aligned
- Agent responses include structured data when relevant (tables, counts, timestamps)
- "Thinking..." indicator while KQL is being generated and executed
- Error states: "I couldn't find any data matching that query" (not raw KQL errors)

### TS-3: Claude Code MCP Tool for Investigation

| Attribute | Detail |
|-----------|--------|
| **Why expected** | Will uses Claude Code for development; investigation should be available during coding sessions without context-switching to mobile or Azure Portal |
| **Complexity** | Medium |
| **Dependencies** | TS-1 (backend agent or direct KQL execution), MCP server infrastructure |
| **Notes** | Two viable approaches: (A) Use the official Azure MCP Server which already has `query_workspace_logs` and `query_resource_logs` tools -- this requires configuring the Azure MCP Server in Claude Code's MCP settings with proper auth (DefaultAzureCredential or service principal); (B) Build a custom lightweight MCP server (using FastMCP) that wraps the same investigation agent from TS-1 and exposes it as a tool. **Recommendation: Start with approach (A)** -- the Azure MCP Server already exists, supports KQL queries with workspace/table/query parameters, and avoids building custom infrastructure. If the DX is insufficient (e.g., Claude Code needs more schema context to write good KQL), then layer a custom MCP server with pre-configured prompts on top. |

**Expected user behavior:** During a Claude Code session, Will asks "check App Insights for recent errors in the second brain backend." Claude Code invokes the MCP tool, runs a KQL query, and presents the results in the conversation. Will can refine: "show me the trace for capture ID xyz."

### TS-4: Classifier Accuracy Eval with Golden Dataset

| Attribute | Detail |
|-----------|--------|
| **Why expected** | The Classifier agent is the core routing engine -- quality degradation means items go to the wrong bucket silently |
| **Complexity** | Medium |
| **Dependencies** | Existing Classifier agent, Azure AI Foundry eval SDK (`azure-ai-evaluation` package) |
| **Notes** | A golden dataset of ~50-100 captures with known-correct bucket labels. Each entry: `{"text": "need cat litter", "expected_bucket": "Admin", "expected_confidence_min": 0.7}`. The eval pipeline sends each test case through the Classifier and checks: (1) correct bucket assignment, (2) confidence above threshold, (3) multi-bucket splits detected when expected. Azure AI Foundry's evaluation SDK supports custom code-based evaluators where a `grade()` function returns 0.0-1.0. The golden dataset can be seeded from the existing Cosmos DB Inbox collection (captures Will has already validated by not recategorizing). |

**Golden dataset categories to cover:**
- Unambiguous single-bucket (Admin: "buy milk", People: "call mom", Projects: "update portfolio", Ideas: "app for tracking sleep")
- Multi-bucket triggers ("buy milk and call the vet" -> Admin + Admin, or Admin + People)
- Edge cases (URLs, recipe links, ambiguous items like "check on the project timeline")
- Low-confidence triggers (vague captures like "thing for later")
- Voice transcription artifacts (run-on sentences, missing punctuation)

**Expected output:** Classification accuracy percentage, per-bucket precision/recall, confidence calibration (are 0.9 confidence items actually right 90% of the time?), and a list of failures for manual review.

### TS-5: Admin Agent Quality Eval

| Attribute | Detail |
|-----------|--------|
| **Why expected** | The Admin Agent handles errand routing, recipe extraction, and affinity rules -- quality issues mean items go to the wrong destination or get lost |
| **Complexity** | Medium-High |
| **Dependencies** | Existing Admin Agent, Azure AI Foundry eval SDK, golden dataset |
| **Notes** | Admin Agent eval is harder than Classifier eval because correctness depends on the destination routing context (affinity rules, available destinations). The eval needs: (1) test captures with expected routing outcomes (e.g., "buy milk" -> destination: "jewel-osco"), (2) expected tool call sequences (add_errand_items should be called, not manage_destination), (3) recipe extraction quality (given a URL, were the right ingredients extracted?). Azure AI Foundry provides a **Task Adherence** evaluator that checks if the agent followed its system instructions -- this maps directly to "did it route to the correct destination?" Use LLM-as-judge with a rubric: "Given the input '{text}' and the routing context '{rules}', the agent should route to '{expected_destination}'. Score 1.0 if correct, 0.5 if reasonable alternative, 0.0 if wrong." |

**Expected output:** Task adherence score, routing accuracy by destination, recipe extraction completeness (ingredients found vs. expected), and tool usage correctness (right tools called in right order).

### TS-6: Eval Score Tracking Over Time

| Attribute | Detail |
|-----------|--------|
| **Why expected** | A single eval run is a snapshot; the value is in tracking trends and detecting regressions |
| **Complexity** | Low-Medium |
| **Dependencies** | TS-4, TS-5 (eval pipelines), Cosmos DB or App Insights for storage |
| **Notes** | After each eval run, store the results: timestamp, overall accuracy, per-bucket scores, per-evaluator scores. Two storage options: (A) Cosmos DB `EvalResults` container -- simple, queryable, fits existing pattern; (B) App Insights custom events -- queryable via KQL, visible in the investigation agent. **Recommendation: Both.** Store detailed results in Cosmos for the eval dashboard, and emit summary metrics to App Insights for alerting and investigation agent visibility. The Azure AI Foundry portal also stores eval run results with comparison views -- leverage this for manual deep-dives. |

### TS-7: Alert on Eval Score Degradation

| Attribute | Detail |
|-----------|--------|
| **Why expected** | The self-monitoring loop -- "the system watches itself" -- requires automated alerting when quality drops |
| **Complexity** | Low-Medium |
| **Dependencies** | TS-6 (score tracking), existing Azure Monitor alert infrastructure |
| **Notes** | Define thresholds: Classifier accuracy < 85% triggers warning, < 75% triggers critical. Admin Agent task adherence < 80% triggers warning. These alerts use the same Azure Monitor scheduled query rules pattern from Phase 14 (KQL over App Insights custom events). The eval pipeline runs on a schedule (weekly recommended for a single-user hobby project) or on-demand via a CLI command. |

**Expected user behavior:** If classifier quality degrades (perhaps due to an Azure OpenAI model update or instruction drift), Will gets a push notification via Azure Monitor: "Classifier accuracy dropped to 72% (threshold: 85%)." Will can then investigate via the mobile investigation agent or Claude Code MCP tool.

## Differentiators

Features that elevate this beyond basic monitoring into a self-improving system.

### D-1: Implicit Quality Signals from User Behavior

| Attribute | Detail |
|-----------|--------|
| **Value proposition** | Every recategorization, HITL intervention, and errand re-routing is a signal about agent quality -- capture these without extra user effort |
| **Complexity** | Low-Medium |
| **Dependencies** | Existing recategorize flow, HITL bucket selection, errand routing corrections |
| **Notes** | The app already has implicit feedback signals: (1) **Recategorize** in inbox = classifier was wrong about the bucket, (2) **HITL bucket selection** on low-confidence = classifier couldn't decide, (3) **Errand re-routing** = admin agent picked wrong destination. Today these are logged but not aggregated. Adding a `quality_signals` container in Cosmos that records each correction event (original classification, user's correction, timestamp) creates a dataset that can be used for: eval set expansion, instruction tuning evidence, and quality trend dashboards. This is the highest-value differentiator because it requires zero additional user effort -- the feedback is already happening, it just needs to be captured and analyzed. |

**Expected user behavior:** Will's existing workflow doesn't change. When he recategorizes an item or selects a bucket during HITL, the system silently records this as a quality signal. Over time, these accumulate into a rich dataset showing where the agents struggle. The investigation agent can answer "what are the most common misclassifications?" by querying these signals.

### D-2: Mobile Observability Dashboard (Summary View)

| Attribute | Detail |
|-----------|--------|
| **Value proposition** | At-a-glance system health without opening a chat or the Azure Portal |
| **Complexity** | Medium |
| **Dependencies** | Backend API endpoints for aggregated metrics |
| **Notes** | A lightweight dashboard section at the top of the Status screen (above errands) showing: capture count today, success rate, last error (if any), and latest eval scores. This is NOT a full Grafana-style dashboard -- it's 3-5 key metrics rendered as simple cards. The data comes from a new `GET /api/observability/summary` endpoint that runs pre-built KQL queries and returns structured results. Refreshes on screen open, not real-time. |

**Cards to show:**
- "12 captures today, 100% classified" (green) or "11/12 captures classified, 1 error" (amber)
- "Classifier: 92% accuracy" (from last eval run)
- "Admin Agent: 88% task adherence" (from last eval run)
- "Last error: 2h ago - recipe fetch timeout" (tappable -> jumps to investigation chat with pre-filled query)

### D-3: Eval-Driven Golden Dataset Evolution

| Attribute | Detail |
|-----------|--------|
| **Value proposition** | The golden dataset grows organically from production data rather than being manually curated once |
| **Complexity** | Medium |
| **Dependencies** | D-1 (implicit signals), TS-4 (classifier eval) |
| **Notes** | When implicit signals identify a misclassification, the corrected capture can be automatically added to the golden dataset (after human review). Over time, the eval dataset becomes increasingly representative of real-world usage patterns, including edge cases that manual curation would miss. A CLI command or API endpoint promotes a quality signal to a golden dataset entry after Will confirms the correction is the ground truth. |

### D-4: On-Demand Eval Trigger from Mobile

| Attribute | Detail |
|-----------|--------|
| **Value proposition** | "Run evals" from the phone after making agent instruction changes |
| **Complexity** | Low |
| **Dependencies** | TS-4, TS-5 (eval pipelines), backend endpoint |
| **Notes** | A button in the observability dashboard or investigation chat that triggers an eval run. The endpoint kicks off an async eval pipeline (not blocking -- eval runs take minutes). Results are available via the investigation agent once complete. Could also be triggered from Claude Code via MCP. |

### D-5: Investigation Agent Pre-Built Queries (Quick Actions)

| Attribute | Detail |
|-----------|--------|
| **Value proposition** | One-tap access to common investigations without typing |
| **Complexity** | Low |
| **Dependencies** | TS-2 (mobile chat interface) |
| **Notes** | The investigation chat shows 3-4 quick action chips above the input: "Recent errors", "Today's captures", "System health", "Last eval results". Tapping a chip sends the pre-built question to the investigation agent. These map directly to the four existing KQL query templates. Reduces friction for the most common investigations. |

## Anti-Features

Features to explicitly NOT build in v3.1. Each has a clear rationale.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **Full Grafana/dashboard visualization** | Scope creep; mobile screen is too small for complex dashboards; single-user doesn't need dashboard infra | Simple metric cards (D-2) + natural language investigation (TS-1) |
| **Real-time streaming metrics** | Over-engineered for single-user; polling on screen open is sufficient | `GET /api/observability/summary` returns snapshot |
| **Automated instruction tuning** | Dangerous for production; agent instructions should be changed deliberately | Surface quality signals, let Will decide when/how to adjust instructions |
| **Fine-tuning / model retraining** | Azure OpenAI GPT-4o doesn't support fine-tuning for agent use; wrong approach for this use case | Prompt/instruction engineering based on eval results |
| **Third-party eval platforms (Langfuse, Confident AI, etc.)** | Adds external dependency and cost for a single-user hobby project; Azure AI Foundry has native eval support | Use `azure-ai-evaluation` SDK + Foundry portal for eval results |
| **A/B testing of agent configurations** | Single-user system; no statistical significance possible | Run eval before/after instruction changes, compare scores manually |
| **Custom KQL query builder UI** | Building a query builder is a product in itself | Natural language agent generates KQL; Will can copy queries for manual refinement |
| **Anomaly detection on metrics** | Overkill for single-user volume (~10-20 captures/day) | Simple threshold alerts on eval scores |
| **Historical trend charts on mobile** | Complex charting libraries for React Native add bloat for marginal value | Use Foundry portal or App Insights for visual analytics; mobile shows latest scores only |
| **Multi-agent eval orchestration** | Evaluating the Classifier -> Admin handoff as a single pipeline adds complexity | Evaluate each agent independently first; pipeline eval is future work |
| **Continuous production evaluation (every request)** | Cost-prohibitive (LLM-as-judge on every capture); unnecessary at low volume | Scheduled batch eval (weekly) + implicit signal collection (continuous) |

## Expected User Behaviors & Scenarios

### Scenario 1: Post-Capture Debugging (weekly, ~40% of investigation usage)

1. Will captures "add chicken thighs and rice to the grocery list" via voice
2. Opens Status screen, doesn't see the items on the errand list
3. Taps the "Investigate" button (magnifying glass icon)
4. Types: "what happened to my last capture?"
5. Agent runs KQL, shows: "Your last capture (trace-id abc-123) was classified as Admin with 0.91 confidence. Admin Agent processing failed with: 'ConnectionError fetching recipe URL.' The capture is in pending state."
6. Will now knows the issue and can re-trigger processing or capture again

### Scenario 2: Morning Health Check (daily, ~30% of investigation usage)

1. Will opens the Status screen
2. Sees the dashboard cards: "8 captures yesterday, 100% success, Classifier 94%"
3. Everything is green -- no investigation needed
4. If a card shows amber/red, taps it to jump to investigation chat with the relevant question pre-filled

### Scenario 3: Claude Code Debugging Session (monthly, ~20% of investigation usage)

1. Will is working on the codebase in Claude Code
2. Asks: "check the second brain App Insights for any errors in the last 24 hours"
3. Claude Code uses the MCP tool to run a KQL query
4. Shows 2 errors: a recipe fetch timeout and a Cosmos throttle
5. Will asks: "show me the full trace for the recipe fetch timeout"
6. Claude Code drills into the trace ID and shows the full lifecycle

### Scenario 4: Post-Instruction-Change Eval (monthly, ~10% of eval usage)

1. Will updates the Classifier agent's instructions in the Foundry portal
2. Opens the mobile investigation chat
3. Taps "Run evals" quick action
4. Waits ~2 minutes, then asks "show me the latest eval results"
5. Agent reports: "Classifier accuracy: 96% (up from 94%). 2 failures: 'check on project timeline' misclassified as Admin (expected: Projects), 'thing for later' went to Ideas (expected: Admin)."
6. Will reviews the failures and decides whether to adjust instructions further

### Scenario 5: Quality Signal Accumulation (continuous, passive)

1. Will captures "remind me to check on the Johnson project"
2. Classifier routes to People (because "Johnson" sounds like a person name)
3. Will notices it in the Inbox, recategorizes to Projects
4. The recategorization is silently logged as a quality signal
5. Over weeks, several "project-related with person names" misclassifications accumulate
6. During the next eval, these appear in the failure analysis
7. Will adjusts the Classifier instructions to handle this edge case

## MVP Recommendation

Prioritize features in this order, based on dependency chain and immediate value:

### Phase 1: Investigation Agent Foundation
1. **TS-1: Investigation agent backend** -- NL -> KQL -> execute -> summarize pipeline
2. **TS-3: Claude Code MCP tool** -- Configure Azure MCP Server for immediate dev-time value
   - Lowest cost path: just configure the existing Azure MCP Server, no custom code

### Phase 2: Mobile Investigation
3. **TS-2: Mobile chat interface** -- Conversation screen with SSE streaming
4. **D-5: Quick action chips** -- Pre-built queries for common investigations
5. **D-2: Observability dashboard cards** -- Summary metrics on Status screen

### Phase 3: Eval Foundation
6. **TS-4: Classifier golden dataset + eval pipeline** -- Build dataset, write evaluators, run first eval
7. **D-1: Implicit quality signal capture** -- Record recategorizations and HITL corrections
8. **TS-5: Admin Agent eval** -- Task adherence and routing accuracy evaluators

### Phase 4: Self-Monitoring Loop
9. **TS-6: Eval score tracking** -- Store results in Cosmos + App Insights
10. **TS-7: Degradation alerts** -- Azure Monitor rules on eval score custom events
11. **D-3: Golden dataset evolution** -- Promote quality signals to eval dataset
12. **D-4: On-demand eval trigger** -- Mobile button + backend endpoint

**Defer (even within v3.1):**
- **D-2 dashboard charts** -- Start with text-only metric cards, add charts only if the text format feels insufficient
- **D-3 automatic promotion** -- Start with manual review of quality signals before auto-adding to golden dataset
