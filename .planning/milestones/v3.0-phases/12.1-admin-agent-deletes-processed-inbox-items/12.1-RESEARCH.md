# Phase 12.1: Admin Agent Deletes Processed Inbox Items - Research

**Researched:** 2026-03-02
**Domain:** Cosmos DB document deletion in async background processing pipeline
**Confidence:** HIGH

## Summary

Phase 12.1 is a surgical backend-only change. When the Admin Agent finishes processing an inbox item (status transitions to "processed"), the item should be deleted from the Inbox Cosmos container rather than remaining with a "processed" flag. This keeps the Inbox clean -- users only see unprocessed items.

The change is confined to a single file: `backend/src/second_brain/processing/admin_handoff.py`. After the Admin Agent successfully completes its work and the status would be set to "processed", instead call `container.delete_item()` to remove the inbox document. The Cosmos DB `delete_item` API is already used elsewhere in the codebase (in `inbox.py` for user-initiated swipe-to-delete and in `shopping_lists.py` for shopping item deletion), so the pattern is well-established. The Inbox container uses `/userId` as partition key with `"will"` as the value -- identical to the existing `delete_inbox_item` endpoint.

The mobile Inbox screen (`inbox.tsx`) already handles items disappearing gracefully -- it re-fetches data on every focus via `useFocusEffect`, and optimistic deletes are already part of the UX. No mobile changes are needed. The only consideration is whether to also cascade-delete the associated bucket document (like the API endpoint does), but for Admin-classified items, the `filedRecordId` points to an Admin container document that should be preserved -- the inbox item is just the audit trail entry that is no longer needed.

**Primary recommendation:** Delete the inbox item directly inside `process_admin_capture` after successful processing. Do NOT cascade-delete the Admin container bucket document. Handle delete failures as non-fatal (log and continue) since the Admin Agent's primary job (routing items to shopping lists) already succeeded.

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Immediate delete after processing | Separate cleanup job / TTL | Immediate delete is simpler -- no need for a scheduled job or TTL policy. TTL adds configuration complexity and delay. The Admin Agent already has the item ID and container reference in scope. |
| Delete in `process_admin_capture` | Delete in a new post-processing step | Extra abstraction with no benefit. The delete is a single SDK call. Factoring it out would add indirection without improving testability or separation of concerns. |
| Hard delete | Soft delete (keep with "deleted" flag) | Hard delete is correct here. The inbox item was a temporary audit trail for the capture flow. Once processed, the shopping list items are the durable output. Keeping deleted flags wastes storage and complicates queries. |

### Pattern 1: Delete After Successful Processing
**What:** After the Admin Agent processes an inbox item successfully, delete the inbox document from Cosmos instead of setting `adminProcessingStatus = "processed"`.
**When to use:** When the inbox item is a transient record that served its purpose (routing the capture to the Admin Agent).
**Example:**
```python
# In process_admin_capture, replace the "processed" upsert with delete
# CURRENT (lines 76-80 of admin_handoff.py):
doc = await inbox_container.read_item(
    item=inbox_item_id, partition_key="will"
)
doc["adminProcessingStatus"] = "processed"
await inbox_container.upsert_item(body=doc)

# NEW:
await inbox_container.delete_item(
    item=inbox_item_id, partition_key="will"
)
```

### Pattern 2: Non-Fatal Delete Failure
**What:** If the delete fails after processing succeeded, log the error but do NOT re-raise. The Admin Agent's primary job (adding shopping list items) already completed successfully.
**When to use:** When the delete is a cleanup step, not the primary operation.
**Example:**
```python
try:
    await inbox_container.delete_item(
        item=inbox_item_id, partition_key="will"
    )
    logger.info("Deleted processed inbox item %s", inbox_item_id)
except Exception as del_exc:
    # Non-fatal: items were already added to shopping lists
    logger.warning(
        "Failed to delete processed inbox item %s: %s",
        inbox_item_id,
        del_exc,
    )
```

### Pattern 3: Preserve Failed Items
**What:** When the Admin Agent fails, keep the inbox item with `adminProcessingStatus = "failed"`. Only delete on success.
**When to use:** Always. Failed items should remain visible in the Inbox so the user can see that processing failed and potentially retry or manually handle the capture.
**Example:**
```python
# Success path: delete the inbox item
# Failure path: keep the item, set status to "failed" (existing behavior, unchanged)
```

### Anti-Patterns to Avoid
- **Cascade-deleting the Admin bucket document:** The `delete_inbox_item` API endpoint cascade-deletes the associated bucket document. For Admin Agent processing, the bucket document in the Admin container should NOT be deleted -- it may serve as an audit trail. The shopping list items are the primary output and live in a separate container.
- **Deleting before processing completes:** The delete must happen AFTER the Admin Agent has finished its work and shopping list items are written. Never delete the inbox item optimistically before the agent call.
- **Batched/deferred deletion:** A separate cleanup job or TTL-based expiration adds unnecessary complexity. The Admin Agent already has the item ID and container reference. Delete immediately.
- **Re-reading the document before delete:** The `delete_item` API only needs the item ID and partition key. No need to `read_item` first just to delete it. The existing code already read the document at the start of `process_admin_capture` to set "pending" status, so we know the item exists.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Document deletion | Custom Cosmos query with DELETE | `container.delete_item(item=id, partition_key="will")` | SDK method handles retry, error codes, and partition routing. Already used in 2 other files. |
| Handling already-deleted items | Custom existence check before delete | Catch `CosmosResourceNotFoundError` | Race condition where user swipe-deleted the item while Admin was processing. SDK raises the correct exception. |
| Scheduled cleanup job | Cron/timer to find and delete processed items | Inline delete in `process_admin_capture` | The delete point is obvious (right after successful processing). A separate job adds latency, complexity, and a new infrastructure component for zero benefit. |

**Key insight:** This phase requires modifying ~10 lines in one file. The Cosmos SDK `delete_item` call is the same 2-parameter call used throughout the codebase. The only design decision is WHERE to place it (after successful processing) and HOW to handle failures (non-fatal).

## Common Pitfalls

### Pitfall 1: Deleting Before Admin Agent Completes
**What goes wrong:** The inbox item is deleted before the Admin Agent finishes writing shopping list items. If the agent call fails, the inbox item is gone and the capture is lost.
**Why it happens:** Premature optimization -- trying to clean up early.
**How to avoid:** The delete MUST happen after `admin_client.get_response()` returns successfully. The current code already has the correct structure: set pending -> call agent -> update status. Replace the final "update status to processed" with "delete item".
**Warning signs:** Shopping list items missing but inbox item also gone.

### Pitfall 2: Race Condition with User Swipe-to-Delete
**What goes wrong:** The user swipe-deletes an inbox item from the mobile Inbox screen while the Admin Agent is still processing it. The Admin Agent then tries to delete an already-deleted item and gets a 404.
**Why it happens:** The Admin Agent runs as a fire-and-forget background task. The user can interact with the Inbox screen concurrently.
**How to avoid:** Catch `CosmosResourceNotFoundError` on the delete call and treat it as a no-op. The item is already gone -- that is the desired end state.
**Warning signs:** Error logs showing 404 on inbox item delete after Admin processing.

### Pitfall 3: Breaking the "Pending" Status Update
**What goes wrong:** The initial "pending" status update (lines 49-55 of `admin_handoff.py`) is accidentally removed or moved after the delete logic, so the inbox item never shows "pending" status during processing.
**Why it happens:** Over-aggressive refactoring when adding the delete logic.
**How to avoid:** Only modify the SUCCESS path (the "processed" upsert). The "pending" upsert at the start and the "failed" upsert in the except block remain unchanged.
**Warning signs:** Inbox items jump from None to deleted without ever showing "pending".

### Pitfall 4: Forgetting to Update Tests
**What goes wrong:** Existing tests assert `adminProcessingStatus == "processed"` after successful processing. These assertions will fail since the item is now deleted instead of updated.
**Why it happens:** Only modifying the source code without updating `test_admin_handoff.py`.
**How to avoid:** Update `test_sets_status_to_pending_then_processed` to assert `delete_item` is called instead of the second `upsert_item`. The test should verify: (1) pending upsert still happens, (2) `delete_item` is called with correct ID and partition key, (3) no second upsert happens on the success path.
**Warning signs:** Test failures in CI after deploying the change.

### Pitfall 5: Not Handling the Batch Path
**What goes wrong:** `process_admin_captures_batch` calls `process_admin_capture` for each item. If the delete is only added in a wrapper but not in `process_admin_capture` itself, batch processing misses the delete.
**Why it happens:** Misunderstanding the code structure.
**How to avoid:** The delete goes inside `process_admin_capture`, which is called by both the single-item path and the batch path. No changes needed in `process_admin_captures_batch`.
**Warning signs:** Single captures get cleaned up but multi-split captures leave inbox items behind.

## Open Questions

1. **Should the Admin container bucket document also be deleted?**
   - What we know: The `delete_inbox_item` API endpoint cascade-deletes the bucket document. For Admin-classified items, the bucket document is in the Admin container.
   - What's unclear: Whether anyone references the Admin container document after processing. Currently nothing queries the Admin container for user-facing data.
   - Recommendation: Do NOT cascade-delete the Admin bucket document in Phase 12.1. The Admin container document may serve as an audit trail. If cleanup is needed later, it can be added as a separate concern. Keep this phase minimal -- delete only the Inbox item.

2. **Should failed items be retried or surfaced to the user?**
   - What we know: Failed items remain in the Inbox with `adminProcessingStatus = "failed"`. The Inbox screen shows them but does not highlight the failed status.
   - What's unclear: Whether users even notice or care about failed Admin items.
   - Recommendation: Out of scope for Phase 12.1. Failed item handling is a separate UX concern. This phase only changes the success path.
