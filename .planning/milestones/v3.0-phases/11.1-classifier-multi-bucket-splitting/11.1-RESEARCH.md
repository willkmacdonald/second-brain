# Phase 11.1: Classifier Multi-Bucket Splitting - Research

**Researched:** 2026-03-01
**Domain:** Foundry Agent Service streaming, multi-tool-call detection, SSE event aggregation, Classifier instruction design
**Confidence:** HIGH

## Summary

Phase 11.1 enables the Classifier agent to call `file_capture` multiple times in a single agent run when a capture contains intents targeting different buckets. Today, the Classifier calls `file_capture` once per capture, and the adapter tracks only one `file_capture` outcome (single `detected_tool` / `detected_tool_args` / `tool_result` variables). This phase requires three coordinated changes:

1. **Classifier Foundry instructions** -- updated to detect multi-intent captures and call `file_capture` once per distinct-bucket intent
2. **Streaming adapter** -- refactored to collect multiple `file_capture` calls/results from the SSE stream, then emit an aggregated result event
3. **Mobile SSE client** -- updated to display multi-bucket confirmation ("Filed to Admin, Ideas") instead of single-bucket format

The Foundry Agent Service streaming API naturally supports multiple tool calls per agent run. Each `function_call` and `function_result` content item arrives as a separate streaming chunk. The adapter's current single-variable tracking is the only limitation -- it is a code design choice, not a platform constraint.

**Primary recommendation:** Refactor the adapter to collect `file_capture` results into a list rather than a single variable. After the stream completes, emit a single aggregated SSE event containing all buckets. The mobile client adapts the toast format based on the number of buckets returned.

### Pattern 1: Multi-Result Collection in Streaming Adapter
**What:** Replace single `detected_tool` / `detected_tool_args` / `tool_result` variables with a list that accumulates all `file_capture` call/result pairs during a single agent run.
**When to use:** In `stream_text_capture()` and `stream_voice_capture()` in adapter.py.

The current adapter code:
```python
# Current: single tracking
detected_tool: str | None = None
detected_tool_args: dict = {}
tool_result: dict | None = None
```

The refactored approach:
```python
# Phase 11.1: multi-result tracking
file_capture_results: list[dict] = []  # Accumulates {bucket, confidence, item_id, status, ...}
current_fc_args: dict = {}  # Temp: args for the function_call currently in progress
current_fc_name: str | None = None  # Temp: name of function_call currently in progress
```

**How it works in the streaming loop:**

The Foundry streaming API emits content items in order within each update chunk. For a multi-tool-call run, the sequence is:

```
[text reasoning] -> [function_call: file_capture #1] -> [function_result #1]
                 -> [function_call: file_capture #2] -> [function_result #2]
                 -> ... (agent may emit more reasoning between calls)
```

Each `function_call` content has `content.type == "function_call"` with `content.name` and `content.arguments`. Each `function_result` has `content.type == "function_result"` with `content.result`.

The adapter must:
1. When seeing `function_call` with `name == "file_capture"`: store args in `current_fc_args`
2. When seeing `function_result` after a `file_capture` call: parse the result, merge with `current_fc_args`, append to `file_capture_results`
3. After the stream completes: use `file_capture_results` to determine the aggregated event

**Critical detail:** The agent may also call `transcribe_audio` (voice captures). The adapter must track which function_call is "current" to correctly pair results with calls. The existing `last_function_call` pattern in `stream_voice_capture` already does this -- extend it to `stream_text_capture` as well.

```python
# Inside the streaming loop:
for content in update.contents or []:
    if content.type == "function_call":
        name = getattr(content, "name", None)
        current_fc_name = name
        if name == "file_capture":
            current_fc_args = _parse_args(getattr(content, "arguments", {}))

    elif content.type == "function_result":
        parsed = _parse_result(getattr(content, "result", None))
        if current_fc_name == "file_capture" and parsed is not None:
            # Merge args with result for complete picture
            merged = {**current_fc_args, **(parsed or {})}
            file_capture_results.append(merged)
        elif current_fc_name == "transcribe_audio" and parsed is not None:
            transcript_text = parsed.get("raw", "")
        current_fc_name = None  # Reset after consuming result

    elif content.type == "text" and getattr(content, "text", None):
        reasoning_buffer += content.text
        # ... logging
```

### Pattern 2: Aggregated SSE Event
**What:** A single SSE event that communicates all filed buckets to the mobile client.
**When to use:** After the streaming loop completes, when `file_capture_results` has one or more entries.

**Option A: Extend existing CLASSIFIED event with a list**

```python
# New event format (backward-compatible when len==1):
{
    "type": "CLASSIFIED",
    "value": {
        "inboxItemId": "first-item-id",  # Primary item (first filed)
        "bucket": "Admin",               # Primary bucket (first filed)
        "confidence": 0.85,              # Primary confidence
        "buckets": ["Admin", "Ideas"],   # ALL buckets filed (new field)
    }
}
```

The mobile client currently reads `value.bucket` for the toast. Adding `value.buckets` (array) lets the client choose: if `buckets.length > 1`, show "Filed to Admin, Ideas"; otherwise fall back to existing behavior.

**Option B: New MULTI_CLASSIFIED event type**

```python
{
    "type": "MULTI_CLASSIFIED",
    "value": {
        "items": [
            {"inboxItemId": "id-1", "bucket": "Admin", "confidence": 0.85},
            {"inboxItemId": "id-2", "bucket": "Ideas", "confidence": 0.78},
        ]
    }
}
```

**Recommendation: Option A** -- extending CLASSIFIED is simpler and backward-compatible. The mobile client already handles CLASSIFIED; it just needs to check for the new `buckets` field. Option B would require a new event type in both backend and mobile, adding to the event type proliferation.

For single-file captures (the common case), `buckets` would be `["Admin"]` (length 1) -- the toast stays "Filed -> Admin (0.85)". For multi-split, `buckets` would be `["Admin", "Ideas"]` -- the toast becomes "Filed to Admin, Ideas".

### Pattern 3: Admin Agent Batching for Multi-Split
**What:** When a multi-split capture produces multiple Admin-classified items, process them in a single Admin Agent background task rather than spawning one per item.
**When to use:** In the adapter, after determining which items are Admin-classified from `file_capture_results`.

**Current single-item Admin trigger:**
```python
if admin_bucket == "Admin" and admin_client is not None:
    task = asyncio.create_task(process_admin_capture(..., inbox_item_id=item_id, raw_text=text))
```

**Batched Admin trigger for multi-split:**
```python
admin_items = [
    r for r in file_capture_results
    if r.get("bucket") == "Admin"
]
if admin_items and admin_client is not None:
    # Option 1: Single task that processes all Admin items sequentially
    task = asyncio.create_task(
        process_admin_captures_batch(
            admin_client=admin_client,
            admin_tools=admin_tools,
            cosmos_manager=cosmos_manager,
            admin_items=admin_items,  # List of {item_id, raw_text, ...}
        )
    )
    background_tasks.add(task)
    task.add_done_callback(background_tasks.discard)
```

**Where does `raw_text` come from for each Admin item?**
Per CONTEXT.md: "Each split item stores only its own text segment." The Classifier's `file_capture` call passes the segment text as the `text` parameter. The adapter can extract this from `current_fc_args["text"]` (the `text` argument to `file_capture`). This is the natural text segment the Classifier extracted for each intent.

However, there is a subtlety: the `file_capture` result (from `function_result`) contains `{bucket, confidence, item_id}` -- it does NOT echo back the `text` argument. The `text` argument is in `current_fc_args` (from the preceding `function_call`). When we merge `{**current_fc_args, **(parsed or {})}`, we get both `text` and the result fields. This merged dict becomes the source for Admin Agent raw_text.

**Batch processing function:**
Create `process_admin_captures_batch` as a new function (or extend `process_admin_capture` to accept a list). Each item in the batch gets sent to the Admin Agent independently but within the same background task coroutine. This means one `asyncio.create_task` call, but potentially multiple `admin_client.get_response` calls within it.

Alternatively, since each Admin-split item is likely a short, single-intent capture (e.g., "need milk" from "need milk and also call the vet"), a single Admin Agent call per item is fine. The "batching" just means using one background task wrapping a loop over items, rather than spawning N tasks.

### Pattern 4: Classifier Instruction Update for Multi-Intent Detection
**What:** Update the Classifier agent's Foundry portal instructions to analyze captures for multiple distinct intents and call `file_capture` once per cross-bucket intent.
**When to use:** Foundry portal update (no code change needed).

**Key instruction additions:**

```
## Multi-Intent Detection

Before classifying, check if the capture contains MULTIPLE distinct intents
that belong to DIFFERENT buckets.

Rules:
1. If all intents belong to the SAME bucket, call file_capture ONCE with the
   full text. Do NOT split within a single bucket.
   Example: "buy milk, eggs, and bread" → single Admin item

2. If intents span DIFFERENT buckets, call file_capture ONCE PER BUCKET with
   only the text segment for that intent.
   Example: "need milk and also remind me to call the vet" →
     file_capture(text="need milk", bucket="Admin", ...)
     file_capture(text="remind me to call the vet", bucket="People", ...)

3. Preserve the user's exact words for each segment. Extract, don't rephrase.

4. Each segment gets its own title appropriate to its content.

5. When in doubt (ambiguous boundary), keep as a single item in the best-fit
   bucket rather than splitting incorrectly.
```

**Important:** The Classifier must still produce valid `status` and `confidence` for each `file_capture` call independently. Each call is a complete classification decision for its segment.

### Anti-Patterns to Avoid

- **Emitting multiple CLASSIFIED events per capture:** The mobile client expects exactly one result event before COMPLETE. Multiple events would trigger `onComplete` multiple times or get lost. Use a single aggregated event.
- **Losing the function_call args when processing function_result:** The `text` argument to `file_capture` is only in the `function_call` content, not in the `function_result`. Must track `current_fc_args` per call and merge when the result arrives.
- **Spawning N Admin Agent background tasks for N Admin-split items:** Per CONTEXT.md, batch into a single task. One `create_task` call, one `add_done_callback`.
- **Breaking the safety-net logic:** The safety net fires when NO `file_capture` call was detected. With multi-result tracking, the safety net checks `len(file_capture_results) == 0` instead of `detected_tool != "file_capture"`.
- **Modifying follow-up flow:** Per CONTEXT.md, follow-ups keep the single-item flow. Only `stream_text_capture` and `stream_voice_capture` get the multi-result logic. `stream_follow_up_capture` is untouched.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multi-intent detection | Custom NLP parser to split text | Classifier agent instructions in Foundry portal | The LLM does the intent analysis; instructions define split rules |
| Multiple tool call tracking | Custom state machine for tool call sequencing | Simple list accumulation in the streaming loop | Tool calls arrive in order; no complex state machine needed |
| Multi-bucket toast formatting | Complex event type hierarchy | Single CLASSIFIED event with `buckets` array field | Backward-compatible; minimal mobile change |
| Admin batching | Task queue or job system | Single `asyncio.create_task` wrapping a loop | Simple; one task processes all Admin items sequentially |

## Common Pitfalls

### Pitfall 1: function_result Content Not Paired with Correct function_call
**What goes wrong:** The adapter sees a `function_result` but doesn't know which `function_call` it belongs to, especially when `transcribe_audio` (voice) and `file_capture` calls are interleaved.
**Why it happens:** The streaming API does not explicitly link results to calls via a `call_id` in the content items. The pairing is implicit by order.
**How to avoid:** Track `current_fc_name` -- set it on each `function_call`, consume it on the next `function_result`, then reset. This matches the existing `last_function_call` pattern in `stream_voice_capture`.
**Warning signs:** Transcript text ending up as a `file_capture` result, or `file_capture` results being lost.

### Pitfall 2: Overwriting current_fc_args on Rapid Tool Calls
**What goes wrong:** If the streaming API emits two `function_call` chunks rapidly without an intervening `function_result`, the second call's args overwrite the first's before the first result is processed.
**Why it happens:** This should NOT happen in practice -- Foundry Agent Service executes tool calls sequentially (call -> execute -> result -> next call). But defensive coding is warranted.
**How to avoid:** Only overwrite `current_fc_args` when a new `function_call` with `name == "file_capture"` is seen. If a result was expected but not received, log a warning.
**Warning signs:** Missing items in multi-split results; `file_capture_results` has fewer entries than expected.

### Pitfall 3: Safety Net Triggering on Partial Multi-Split
**What goes wrong:** The Classifier calls `file_capture` for the first intent but the agent run times out or errors before the second call. The safety net should NOT re-file the capture as misunderstood because the first item was already filed.
**Why it happens:** The safety-net currently checks `detected_tool != "file_capture"`. With multi-result tracking, it would check `len(file_capture_results) == 0`.
**How to avoid:** Per CONTEXT.md: "Partial success is acceptable -- successfully filed items stay filed." Only trigger the safety net when `file_capture_results` is empty. If at least one `file_capture` succeeded, emit the aggregated event with whatever results were collected.
**Warning signs:** Misunderstood events fired alongside valid CLASSIFIED events for the same capture.

### Pitfall 4: Mobile Client Crashes on New Event Format
**What goes wrong:** Adding `buckets` array to the CLASSIFIED event breaks the mobile client parsing if it expects only `bucket` (string).
**Why it happens:** TypeScript is lenient about extra fields in parsed JSON, so adding `buckets` should not crash. But if the toast logic reads `value.buckets` on an old backend version that doesn't send it, it gets `undefined`.
**How to avoid:** Mobile code checks `value.buckets?.length > 1` first; falls back to existing `value.bucket` for the toast text. This makes the mobile code backward-compatible with both old (no `buckets`) and new (with `buckets`) backend versions.
**Warning signs:** Toast showing "undefined" or "Filed -> undefined (0.85)".

### Pitfall 5: Admin Agent Receiving Full Original Text Instead of Segment
**What goes wrong:** The Admin Agent receives "need milk and also remind me to call the vet" instead of just "need milk", because the adapter passes `user_text` (full capture) instead of the segment text.
**Why it happens:** The Phase 11 trigger passes `raw_text=user_text` (the full capture). For multi-split, each Admin item should receive only its segment.
**How to avoid:** Extract the segment text from the merged `file_capture_results` entry. The `text` field in each entry comes from the Classifier's `file_capture(text=...)` call, which contains only the segment per CONTEXT.md.
**Warning signs:** Admin Agent creating duplicate shopping items or items from the non-Admin segment.

### Pitfall 6: Empty `status` or `confidence` on Multi-Split file_capture Calls
**What goes wrong:** The Classifier calls `file_capture` for a split segment but omits `status` or `confidence`, resulting in undefined behavior in `_emit_result_event`.
**Why it happens:** The LLM may not consistently provide all parameters for every call in a multi-call run.
**How to avoid:** Add explicit defaults in the merged result dict. If `status` is missing, default to "classified". If `confidence` is missing, default to 0.75 (matching the existing zero-confidence fallback pattern in `file_capture`).
**Warning signs:** UNRESOLVED events instead of CLASSIFIED for multi-split captures.

### Aggregated Result Event Emission
```python
if file_capture_results:
    # Use first result as the primary (backward-compatible fields)
    primary = file_capture_results[0]
    all_buckets = [r.get("bucket", "?") for r in file_capture_results]

    # Determine worst-case status (any misunderstood -> misunderstood)
    statuses = [r.get("status", "classified") for r in file_capture_results]

    # Check if any result is misunderstood or pending
    has_misunderstood = "misunderstood" in statuses
    has_pending = "pending" in statuses

    if has_misunderstood:
        # If any split is misunderstood, treat as misunderstood
        # (This is unlikely in multi-split but handle defensively)
        yield encode_sse(misunderstood_event(...))
    elif has_pending:
        # If any split is low-confidence, show low-confidence for the whole capture
        yield encode_sse(low_confidence_event(
            primary.get("item_id", ""),
            primary.get("bucket", "?"),
            primary.get("confidence", 0.0),
        ))
    else:
        # All classified -- emit aggregated CLASSIFIED
        yield encode_sse(classified_event_multi(
            primary_item_id=primary.get("item_id", ""),
            primary_bucket=primary.get("bucket", "?"),
            primary_confidence=primary.get("confidence", 0.0),

    # OTel attributes
    span.set_attribute("capture.outcome", "classified")
    span.set_attribute("capture.split_count", len(file_capture_results))
    span.set_attribute("capture.buckets", ",".join(all_buckets))

    # Admin Agent batching
    admin_results = [r for r in file_capture_results if r.get("bucket") == "Admin"]
    if admin_results and admin_client is not None and background_tasks is not None:
        task = asyncio.create_task(
            process_admin_captures_batch(

                admin_tools=admin_tools or [],
                cosmos_manager=cosmos_manager,

                    {"inbox_item_id": r.get("item_id", ""), "raw_text": r.get("text", "")}
                    for r in admin_results
                ],
            )
        )

        task.add_done_callback(background_tasks.discard)
```

## Open Questions

1. **Should misunderstood/pending splits affect the entire capture or just the individual segment?**
   - What we know: CONTEXT.md says "Partial success is acceptable -- successfully filed items stay filed, failed ones hit the existing safety-net."
   - What's unclear: If the Classifier calls `file_capture` three times and the second call returns `status=misunderstood`, should the user see MISUNDERSTOOD for the whole capture, or should the two classified items still emit a CLASSIFIED event?
   - Recommendation: Treat each split independently. If some succeed and one is misunderstood, emit CLASSIFIED for the successful ones. The misunderstood segment is already filed as misunderstood by `file_capture` (it creates an inbox doc with `status=misunderstood`). The safety-net is only for when NO `file_capture` calls happen at all. In practice, the Classifier is unlikely to produce a misunderstood segment in a multi-split scenario -- if it can split intents, it can classify them.

2. **How does `_emit_result_event` work with multi-split?**
   - What we know: `_emit_result_event` takes a single `detected_tool_args` and `tool_result` and determines the event type (classified, misunderstood, pending).
   - What's unclear: With multiple results, do we call `_emit_result_event` once per result or once for the aggregate?
   - Recommendation: Replace the call to `_emit_result_event` with inline logic that handles the list. The function's current signature is incompatible with multi-result. Rather than over-abstracting, handle the aggregation directly in the streaming function. The logic is: filter classified results, build the aggregated event, handle edge cases (all misunderstood, mixed statuses).

3. **Should the CLASSIFIED event include all item IDs or just the primary?**
   - What we know: The mobile client currently uses `inboxItemId` only for the toast -- it does not navigate to the item or use it for follow-up.
   - What's unclear: Future phases (inbox display of split items) may need all IDs.
   - Recommendation: Include `inboxItemId` as the primary (first) item ID for backward compatibility, and add `itemIds: [id1, id2, ...]` for future use. Costs nothing to include now; saves a breaking change later.

4. **How does the batched Admin Agent processing handle partial failure?**
   - What we know: `process_admin_capture` currently handles one item with pending -> processed/failed.
   - What's unclear: If the batch has 2 Admin items and the first succeeds but the second fails, what's the status?
   - Recommendation: Process each Admin item independently within the batch function. Each gets its own `adminProcessingStatus` update. One item's failure does not affect the other. This aligns with "split items are fully independent."
