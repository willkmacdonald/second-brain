# Phase 11.1: Classifier Multi-Bucket Splitting - Context

**Gathered:** 2026-03-01
**Status:** Ready for planning

<domain>
## Phase Boundary

When a user captures mixed-content text containing multiple distinct intents targeting different buckets (e.g., "need milk and also remind me to call the vet"), the Classifier splits it into separate inbox items — each classified into the correct bucket. Today the Classifier calls `file_capture` once per capture, producing a single inbox item. This phase enables multiple `file_capture` calls per capture.

</domain>

<decisions>
## Implementation Decisions

### Splitting trigger
- Always analyze every capture for multiple intents — no pre-filtering heuristics
- The Classifier calls `file_capture` once per distinct intent/bucket. Multiple calls from one agent run is the mechanism
- Splitting logic lives entirely in the Classifier's Foundry instructions (portal-managed, no code deploy to tune)
- Partial success is acceptable — successfully filed items stay filed, failed ones hit the existing safety-net
- No maximum split limit — captures are short in practice, unlikely to exceed 3-4 splits
- Multi-split applies to fresh captures only — follow-up reclassification keeps the existing single-item flow

### Split boundaries
- Each intent maps to its best-fit single bucket — no filing the same segment to multiple buckets
- Preserve the user's exact words for each segment — extract, don't rephrase
- Single-intent captures that mention multiple things (e.g., "buy cake, candles, and a card for the party") stay as one item — the Admin Agent already handles multi-item splitting within a single capture
- If all intents map to the same bucket, keep as a single item — only split when intents cross bucket boundaries

### Text and titles
- Each split item stores only its own text segment (no full original text preserved)
- Each split item gets its own segment-specific title — items stand alone independently

### SSE/streaming UX
- Adapter collects all `file_capture` results during the agent run, then emits a single aggregated SSE event at the end
- Confirmation shows bucket names only: "Filed to Admin, Ideas" (no text previews)
- Toast adapts format: "Filed to Admin" for single, "Filed to Admin, Ideas" for multi-split
- Voice captures flow through the same path unchanged — transcription produces text, multi-split applies the same way
- Existing safety-net unchanged: if agent finishes without any `file_capture` call, auto-file as misunderstood

### Admin Agent handoff
- When a capture splits into multiple Admin items, batch them into a single Admin Agent background task (fewer agent invocations)

### Reconciliation
- Split items are fully independent — no shared capture group ID, no sibling references
- No visual difference in the inbox — split items look identical to regular captures
- Follow-up reclassification affects only the selected item, not siblings
- Standard logging only — no special split metrics, App Insights traces naturally show multiple file_capture calls per run

### Claude's Discretion
- Exact wording of the Classifier's Foundry instructions for multi-intent analysis
- How the adapter aggregates multiple file_capture results (internal data structure)
- How to batch multiple Admin-classified items into a single Admin Agent background task
- SSE event format for the aggregated classification result

</decisions>

<specifics>
## Specific Ideas

- The splitting instruction update mirrors the existing pattern: Classifier instructions are managed in the AI Foundry portal, same as store routing rules in the Admin Agent
- Key principle: split by bucket boundaries, not by individual items within a bucket. "Buy milk and eggs" = 1 Admin item. "Buy milk and call the vet" = 2 items (Admin + People/Projects)

</specifics>

<deferred>
## Deferred Ideas

- Follow-up reclassification supporting multi-split — keep simple single-item flow for now
- Split metrics dashboard in App Insights — standard logging sufficient for v1
- Visual indicators for split items in the inbox — items are independent, no indicator needed

</deferred>
