# Phase 12.3: Destination Affinity and Knowledge System for Errand Routing - Context

**Gathered:** 2026-03-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Build a knowledge system that lets the Admin Agent route errand items to the correct destination based on user-defined affinity rules. Replace hardcoded destination logic with dynamic, voice-managed routing rules stored in Cosmos DB. Destinations and rules are both created and managed entirely through voice/text captures.

</domain>

<decisions>
## Implementation Decisions

### Knowledge Storage
- Affinity rules are user-defined, not learned or inferred
- Rules stored in Cosmos DB as data documents (not in agent instructions)
- Rules managed entirely via voice/text captures — no mobile UI for rule management
- Admin Agent distinguishes rule-setting intent ("chicken always goes to Agora") from errand intent ("need chicken") within Admin-classified captures — no new classifier bucket
- Category-level rules supported: "meat goes to Agora" — LLM interprets whether an item falls into a category at routing time
- Compound rules with embedded exceptions: "meat goes to Agora, except fish goes to Nick's" stored as a single rule unit, not separate conflicting rules
- Full CRUD via voice: create, update, delete rules through captures
- Rule conflicts require confirmation: system flags "Chicken currently goes to Agora. Change to Jewel?" before overwriting
- System confirms rule creation/modification back to user (toast or similar feedback)
- Rules are queryable via voice: "where does chicken go?" or "what are my rules?" gets a response

### Routing Logic
- Each item routes independently based on its own affinity — no conditional bundling this phase
- Unmatched items (no affinity rule applies) trigger HITL clarification flow: Admin Agent asks "where should batteries go?"
- HITL routing answers auto-save as permanent rules — every answer trains the system
- User can delete auto-saved rules later if wrong

### Destination Management
- Destinations created via voice captures: "add Costco as a destination"
- Each destination has a type: physical or online
- LLM infers destination type from name (Costco → physical, Amazon → online) and confirms — user corrects if wrong
- Online destinations get a visual indicator on Status screen, but all destinations appear in one flat list (no separate grouping)
- Full CRUD via voice: create, rename, remove destinations through captures
- Removing a destination with active items triggers HITL reassignment: "Gangnam Market has 3 items. Where should they go?"
- Destination metadata is minimal: name + type (physical/online) only — no address, notes, or other fields

### Item Matching
- Entity-specific routing uses natural language rules: "Luna's food goes to PetSmart" — no separate entity registry
- LLM semantic matching at routing time: "chicken thighs" matches a "chicken" rule, LLM judges edge cases like "chicken broth"
- Specific rules beat general category rules: "fish → Nick's" overrides "meat → Agora" for fish items
- Ambiguous entity matches trigger HITL clarification: "cat food" with rules for two cats asks "Cat food for Luna (PetSmart) or Oliver (Chewy)?"

### Claude's Discretion
- Cosmos DB document schema for rules and destinations
- How rule-setting vs errand intent detection is implemented (prompt engineering, tool design)
- HITL clarification flow implementation details (reuse existing patterns or adapt)
- How compound rules with exceptions are modeled in the data layer
- Rule querying response format and delivery mechanism
- Auto-save rule confirmation UX

</decisions>

<specifics>
## Specific Ideas

- Rules should feel like talking to a person: "meat goes to Agora, except fish that I get from Nick's" — natural language, not structured input
- The system should work like a personal shopping assistant that learns your preferences over time through HITL answers
- Existing HITL clarification flow (from Classifier low-confidence) should be reused/adapted for routing clarification
- Current hardcoded destinations (Jewel-Osco, Agora, Gangnam Market, Nick's Fishmarket, PetSmart, Chewy, CVS) should be seeded as initial destinations

</specifics>

<deferred>
## Deferred Ideas

- Conditional bundling ("if meat is on the list, put produce at Agora too") — future enhancement
- Destination metadata (address, notes, hours, preferred visit days) — future enhancement
- Mobile UI for rule management (settings screen) — future phase
- Entity registry (defining pets, people as first-class objects) — not needed with natural language rules

</deferred>
