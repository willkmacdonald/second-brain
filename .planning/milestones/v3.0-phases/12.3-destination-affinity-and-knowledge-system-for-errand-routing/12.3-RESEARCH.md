## Summary

Phase 12.3 replaces the hardcoded destination routing in the Admin Agent with a dynamic, user-managed knowledge system. Currently, destinations are a hardcoded list (`KNOWN_DESTINATIONS = ["jewel", "cvs", "pet_store", "other"]`) and routing logic lives entirely in the Admin Agent's Foundry portal instructions. This phase introduces two new Cosmos DB document types (Destinations and AffinityRules), expands the Admin Agent's tool repertoire from 1 tool to approximately 6-8 tools, and adds a HITL clarification flow within the Admin Agent processing pipeline for unmatched items.

The core architectural challenge is that the Admin Agent currently runs non-streaming in the background (via `process_admin_capture` in `admin_handoff.py`) and produces no user-facing output other than errand items appearing on the Status screen. Phase 12.3 requires the Admin Agent to sometimes need user input (HITL for unmatched items, conflict confirmation for rule changes, destination reassignment). This means the Admin Agent processing model must evolve from pure fire-and-forget to a model that can pause, ask questions, and resume — or alternatively, items that need HITL can be surfaced separately for resolution.

The second significant change is intent detection: the Admin Agent must now distinguish between errand captures ("need chicken"), rule-setting captures ("chicken always goes to Agora"), destination management captures ("add Costco as a destination"), and rule queries ("where does chicken go?"). This is best handled through prompt engineering in the Admin Agent's Foundry instructions combined with distinct tools for each intent type.

**Primary recommendation:** Use Cosmos DB documents with `/userId` partition key for both Destinations and AffinityRules containers. Expand the Admin Agent with separate tools for each operation type (add_errand_items, manage_destination, manage_rule, query_rules, request_routing_clarification). Implement HITL by marking items as "needs_routing" and surfacing them on the Status screen with an inline clarification UI, rather than trying to make the background agent interactive.

### Pattern 1: Cosmos DB Document Schemas

**What:** Two new document types for the knowledge system
**When to use:** All rule and destination CRUD operations

**DestinationDocument schema:**
```python
class DestinationDocument(BaseModel):
    """User-defined errand destination (physical or online store/service).

    Partition key: /userId (consistent with other user-data containers).
    The 'slug' field is the machine-readable identifier used as the
    partition key in the Errands container (e.g., "agora", "costco").
    """
    id: str = Field(default_factory=lambda: str(uuid4()))
    userId: str = "will"
    slug: str          # Lowercase, URL-safe: "agora", "nicks_fishmarket"
    displayName: str   # User-facing: "Agora", "Nick's Fishmarket"
    type: str = "physical"  # "physical" or "online"
    createdAt: datetime = Field(default_factory=lambda: datetime.now(UTC))
```

**AffinityRuleDocument schema:**
```python
class AffinityRuleDocument(BaseModel):
    """User-defined routing rule mapping items/categories to destinations.

    Partition key: /userId.
    Rules are natural language with structured metadata extracted by the LLM.
    Compound rules (with exceptions) are stored as a single document with
    an exceptions list.
    """
    id: str = Field(default_factory=lambda: str(uuid4()))
    userId: str = "will"
    naturalLanguage: str    # "meat goes to Agora, except fish goes to Nick's"
    itemPattern: str        # "meat" — primary match pattern
    destinationSlug: str    # "agora" — primary destination
    ruleType: str           # "item", "category", "entity"
    exceptions: list[dict] = []  # [{"pattern": "fish", "destinationSlug": "nicks_fishmarket"}]
    autoSaved: bool = False      # True if created from HITL answer
    createdAt: datetime = Field(default_factory=lambda: datetime.now(UTC))
    updatedAt: datetime = Field(default_factory=lambda: datetime.now(UTC))
```

**Key design decisions:**
- `slug` on DestinationDocument doubles as the partition key value in the Errands container — replaces the hardcoded `KNOWN_DESTINATIONS` list
- `naturalLanguage` on AffinityRuleDocument preserves the original voice/text input for queryability ("what are my rules?")
- `ruleType` helps the LLM prioritize: specific item > category > entity (specificity hierarchy)
- `exceptions` list models compound rules as a single unit per CONTEXT.md decision

### Pattern 2: Admin Agent Intent Detection via Tool Design

**What:** The Admin Agent distinguishes intent by which tool it calls — no separate classifier needed
**When to use:** Every Admin-classified capture

The current Admin Agent has one tool: `add_errand_items`. Expanding to multiple tools lets the LLM naturally route intent:

| Tool | Intent | Example Input |
|------|--------|---------------|
| `add_errand_items` | Shopping/errand item | "need chicken" |
| `manage_destination` | Destination CRUD | "add Costco as a destination" |
| `manage_affinity_rule` | Rule CRUD | "chicken always goes to Agora" |
| `query_rules` | Rule inquiry | "where does chicken go?" |
| `request_routing_clarification` | Unmatched item | (called by agent when no rule matches) |

The Admin Agent's Foundry portal instructions are updated to:
1. Load all destinations and affinity rules at the start of each processing run (via a read-only tool or system context)
2. For errand items: check rules, route matched items, flag unmatched items
3. For rule-setting: check for conflicts, create/update rules
4. For destination management: create/rename/remove destinations
5. For queries: retrieve and format rule information

**Critical insight:** The rules context must be injected into each Admin Agent call. The current `process_admin_capture` sends only the raw text. It must be modified to also pass current rules and destinations as context (either as system message context or via a tool that the agent calls first).

### Pattern 3: HITL for Unmatched Items (Async via Status Screen)

**What:** Items that don't match any rule are stored with a "needs_routing" status and surfaced on the Status screen for user resolution
**When to use:** When the Admin Agent encounters an item with no matching affinity rule

**Why not real-time HITL:** The Admin Agent runs in the background via `asyncio.create_task` in `process_admin_capture`. Making it interactive would require:
- Changing from non-streaming to streaming
- Creating a new SSE connection from the Status screen
- Managing long-lived agent threads
- Fundamentally changing the capture-to-processing pipeline

Instead, the simpler approach:
1. Admin Agent calls `add_errand_items` with `needs_routing=True` flag for unmatched items
2. These items appear on the Status screen with a special UI (destination picker)
3. User selects destination inline
4. Selection triggers: (a) move item to chosen destination, (b) auto-save affinity rule
5. Future captures of the same item route automatically

This reuses the existing Status screen polling pattern and avoids making `admin_handoff.py` interactive.

**ErrandItem extension for unmatched items:**
```python
class ErrandItem(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid4()))
    destination: str      # "unrouted" for unmatched items
    name: str
    needsRouting: bool = False  # True when no affinity rule matched
```

### Pattern 4: Destination-Driven Errands Query (Dynamic)

**What:** Replace the hardcoded `KNOWN_DESTINATIONS` loop in `errands.py` with a dynamic query from the Destinations container
**When to use:** Every `GET /api/errands` call

Currently `get_errands()` loops over `KNOWN_DESTINATIONS` — a hardcoded Python list. This must change to:
1. Query all DestinationDocuments from the Destinations container
2. For each destination, query the Errands container with that slug as partition key
3. Also query for "unrouted" partition (items needing routing)
4. Return dynamic display names and types from the Destinations documents

### Pattern 5: Rule Conflict Detection and Confirmation

**What:** When a rule-setting capture conflicts with an existing rule, the Admin Agent detects the conflict and asks for confirmation
**When to use:** Rule creation and update operations

The `manage_affinity_rule` tool must:
1. Accept the rule definition from the LLM
2. Query existing rules for overlapping patterns
3. If a conflict exists, return a conflict notification instead of creating the rule
4. The conflict appears as a "pending rule change" that the user confirms or rejects

**Implementation approach:** The tool returns a structured response indicating conflict. The Admin Agent's response text (stored in a new field on the inbox item or a separate pending-actions container) surfaces the confirmation question. The user resolves via the Status screen or a follow-up voice capture.

### Anti-Patterns to Avoid
- **Embedding rules in agent instructions:** Agent instructions are static and shared across all users. Rules must be in the database for per-user, dynamic behavior.
- **Making the Admin Agent interactive/streaming:** The fire-and-forget pattern works. Adding streaming would create complexity disproportionate to the value. Use async HITL via Status screen instead.
- **Separate Cosmos queries per rule for matching:** Load all rules once per processing call and let the LLM match in-context. Rules are expected to be < 100 per user, well within LLM context window.
- **Complex NLP for rule matching:** The LLM is the matcher. Pass items and rules to the LLM and let it determine matches. Don't build custom matching logic in Python.
- **Cross-partition fan-out in Errands:** Continue the per-partition query pattern, but iterate over dynamic destinations from the Destinations container instead of the hardcoded list.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Item-to-rule matching | Custom NLP/fuzzy matching | LLM semantic matching via Admin Agent instructions | The LLM naturally handles "chicken thighs" matching "chicken" rule, category inference ("salmon" is "fish"), and entity disambiguation — all hard to code manually |
| Destination slug generation | Custom slugification | LLM generates slug in `manage_destination` tool, validated by Python | Edge cases like "Nick's Fishmarket" → "nicks_fishmarket" are easier for the LLM |
| Rule conflict detection | Graph-based conflict resolver | LLM with full rule context | Rule conflicts are semantic ("meat" vs "chicken") not just string matches |
| Intent classification | Separate classifier | Multi-tool Admin Agent | The Foundry agent naturally selects the right tool based on intent — this is what LLM tool calling is designed for |

**Key insight:** This phase is fundamentally about leveraging the LLM's natural language understanding for matching, conflict detection, and intent routing. Python code should handle data persistence and UI orchestration, not semantic reasoning.

## Common Pitfalls

### Pitfall 1: Errands Container Partition Key Mismatch
**What goes wrong:** The Errands container uses `/destination` as partition key. New dynamic destinations create new partition key values. If a destination is renamed, existing items have the old partition key value and become orphaned.
**Why it happens:** Cosmos DB doesn't support changing partition key values on existing documents.
**How to avoid:** Destination rename must: (1) query all items in old partition, (2) create copies in new partition, (3) delete old items. Or, use the destination `id` (UUID) instead of `slug` as the Errands partition key — UUIDs never change.
**Warning signs:** Items disappearing after destination rename.
**Recommendation:** Use destination `id` (UUID) as the Errands partition key rather than slug. The slug/displayName can change freely without affecting item storage. This requires a one-time migration of existing items.

### Pitfall 2: Admin Agent Context Window Overload
**What goes wrong:** Passing all rules and destinations as context to every Admin Agent call bloats the context, increasing latency and cost.
**Why it happens:** Rules grow over time. A user with 50+ rules and 15 destinations adds significant context per call.
**How to avoid:** For errand processing: pass only the rules (not full destination list — the LLM just needs the destination slug from the rule). For rule management: pass existing rules for conflict detection. For queries: pass all rules. Tailor context to intent.
**Warning signs:** Admin Agent processing time increasing over months of use.

### Pitfall 3: Race Condition on Rule Auto-Save from HITL
**What goes wrong:** User resolves an unmatched item on the Status screen. The API creates the rule and routes the item. But a second identical unmatched item was queued and gets processed before the rule is saved.
**Why it happens:** Background processing is async; rule creation and item routing happen in separate operations.
**How to avoid:** Rule auto-save should happen synchronously within the HITL resolution API endpoint (not as a background task). The rule is created first, then the item is moved.
**Warning signs:** Same HITL question appearing multiple times for the same item type.

### Pitfall 4: Compound Rule Complexity
**What goes wrong:** Compound rules like "meat goes to Agora, except fish goes to Nick's" are hard to model if split into separate rules. Two rules ("meat → Agora" and "fish → Nick's") conflict — the system can't tell which takes priority.
**Why it happens:** Treating compound rules as independent creates false conflicts.
**How to avoid:** Store compound rules as a single document with exceptions list. The LLM evaluates the full rule including exceptions as a unit. Never split a compound rule into separate documents.
**Warning signs:** Conflict detection firing on rules that the user intended as a single compound rule.

### Pitfall 5: Destination Deletion with Active Items
**What goes wrong:** User says "remove Gangnam Market". Items in that destination become orphaned if not reassigned.
**Why it happens:** Cosmos DB has no referential integrity — deleting a destination doesn't cascade to items.
**How to avoid:** The `manage_destination` tool for deletion must: (1) check if items exist in that destination, (2) if yes, return a reassignment question, (3) user answers where items should go, (4) move items first, then delete destination.
**Warning signs:** Items appearing under "Unknown" destination or disappearing entirely.

### Pitfall 6: Seeding Initial Destinations
**What goes wrong:** Existing errand items reference hardcoded destination slugs ("jewel", "cvs", "pet_store"). If the Destinations container doesn't have matching documents, the UI can't display them correctly.
**Why it happens:** Migration from hardcoded to dynamic destinations requires seeding.
**How to avoid:** Write a migration script (like `migrate_shopping_to_errands.py`) that: (1) creates DestinationDocuments for current hardcoded destinations plus the additional ones from CONTEXT.md (Agora, Gangnam Market, Nick's Fishmarket, PetSmart, Chewy), (2) optionally remaps partition keys if switching to UUID-based keys.
**Warning signs:** Status screen showing raw slugs instead of display names after deployment.

## Open Questions

1. **Errands container partition key: slug vs UUID?**
   - What we know: Currently uses destination slug as partition key (`/destination`). Dynamic destinations mean slugs could change (rename).
   - What's unclear: Whether to migrate to UUID-based partition keys now or accept the rename-requires-item-migration constraint.
   - Recommendation: Use destination `id` (UUID) as partition key. Slugs change; UUIDs don't. Requires a one-time migration of existing errand items to use UUID-based partition keys. This is a one-time cost vs ongoing complexity of handling renames.

2. **Rule confirmation UX for conflicts**
   - What we know: CONTEXT.md says "system flags 'Chicken currently goes to Agora. Change to Jewel?' before overwriting." This implies a confirmation step.
   - What's unclear: How to surface the confirmation. Options: (a) Admin Agent response text stored and shown on Status screen, (b) a toast/alert from the capture flow, (c) a separate pending-actions queue.
   - Recommendation: Use the capture flow's SSE response. When the Admin Agent detects a conflict, the processing result includes a confirmation question. The mobile app shows a confirmation dialog. If the user confirms (via follow-up capture or UI button), the rule is updated.

3. **How to inject rules context without bloating every call**
   - What we know: Rules and destinations must be available to the Admin Agent for matching.
   - What's unclear: Whether to use a "load context" tool the agent calls first, or inject as system message.
   - Recommendation: Use a `get_routing_context` tool that the Admin Agent calls at the start of processing. This is more flexible than system message injection and lets the agent decide when it needs context. The tool returns formatted rules and destinations.

4. **Handling the transition from hardcoded to dynamic**
   - What we know: Existing items use hardcoded slugs ("jewel", "cvs", "pet_store", "other"). CONTEXT.md lists additional destinations to seed (Agora, Gangnam Market, Nick's Fishmarket, PetSmart, Chewy, CVS).
   - What's unclear: Whether to keep backward compatibility with old slugs or force a clean migration.
   - Recommendation: Seed migration script creates DestinationDocuments for all destinations (existing + new). If using UUID partition keys, also migrates existing errand items. The old `KNOWN_DESTINATIONS` constant is removed.

5. **Rule query response delivery**
   - What we know: User can ask "where does chicken go?" or "what are my rules?"
   - What's unclear: How the response reaches the user. Current Admin Agent produces no visible output to the user (runs in background).
   - Recommendation: Rule queries should return results through the capture SSE flow, not the background processing path. The Classifier could route these as Admin, and the Admin Agent's response text (the answer to the query) would need a new SSE event type (e.g., ADMIN_RESPONSE) that the mobile app renders as a toast or inline message. Alternatively, rule queries could be handled synchronously in the capture endpoint before fire-and-forget processing.
