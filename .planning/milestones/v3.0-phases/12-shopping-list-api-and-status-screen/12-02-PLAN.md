    The Status & Priorities screen. Follow the inbox.tsx patterns for data fetching and focus-based refresh.

    Key imports:
    ```typescript
    import { useState, useCallback } from "react";
    import { View, Text, ActivityIndicator, StyleSheet } from "react-native";
    import { SectionList } from "react-native";

    ```

    **Types** (define locally in this file):
    ```typescript
    interface ShoppingItem {
      id: string;
      name: string;
      store: string;
    }

    interface StoreSectionData {
      type: "shopping";
      title: string;       // displayName from API (e.g., "Jewel-Osco")
      count: number;
      data: ShoppingItem[];
      key: string;          // store key (e.g., "jewel")
    }
    ```

    **State:**
    - `sections: StoreSectionData[]` -- the SectionList data
    - `expandedSections: Set<string>` -- which sections are expanded (start EMPTY -- all collapsed per CONTEXT.md locked decision)
    - `loading: boolean` -- true during initial fetch
    - `hasLoaded: boolean` -- tracks if at least one successful fetch happened (for stale data decision)

    **Data fetching (`fetchShoppingLists`):**
    - Call `GET ${API_BASE_URL}/api/shopping-lists` with Authorization header
    - On success: transform API response to StoreSectionData[] (map stores -> sections, using store as key, displayName as title)
    - The API already returns stores sorted by count descending, so no client-side sorting needed
    - Set loading=false, hasLoaded=true
    - On error: if hasLoaded is true, keep stale data (per CONTEXT.md: "show stale data if available, silently retry next focus"). If hasLoaded is false, set loading=false.

    **Focus-based refresh (per CONTEXT.md: "Refresh on tab focus only"):**
    ```typescript
    useFocusEffect(
      useCallback(() => {
        void fetchShoppingLists();
      }, [fetchShoppingLists])
    );
    ```

    NO pull-to-refresh (per CONTEXT.md locked decision).
    NO background polling (per CONTEXT.md locked decision).

    **Expand/collapse:**
    - `toggleSection(key: string)` -- add/remove from expandedSections Set
    - All sections start COLLAPSED (empty Set) per CONTEXT.md locked decision
    - Pass `extraData={expandedSections}` to SectionList to force re-render on toggle

    **Optimistic delete (`handleDeleteItem`):**
    - Immediately remove item from sections state
    - Filter out sections where data becomes empty (per CONTEXT.md: "When last item in a store is deleted, the section disappears")
    - Call `DELETE ${API_BASE_URL}/api/shopping-lists/items/${itemId}?store=${store}`
    - On failure (non-404): refetch from API instead of snapshot rollback (per RESEARCH.md Pitfall 3: avoids stale closure issues with rapid deletes)
    - NO error message shown to user (per CONTEXT.md: "silent rollback")

    **Rendering with SectionList:**
    Use React Native SectionList. The `sections` prop maps directly to StoreSectionData[].

    ```typescript
    <SectionList
      sections={sections}
      extraData={expandedSections}
      keyExtractor={(item) => item.id}
      renderSectionHeader={({ section }) => (
        <StatusSectionRenderer
          section={section}
          isExpanded={expandedSections.has(section.key)}
          onToggle={() => toggleSection(section.key)}
          renderItem={() => null}  // Header-only rendering here
        />
      )}
      renderItem={({ item, section }) => {
        if (!expandedSections.has(section.key)) return null;
        return (
          <ShoppingListRow
            item={item}
            onDelete={handleDeleteItem}
          />
        );
      }}
      ListEmptyComponent={...}
      stickySectionHeadersEnabled={false}
    />
    ```

    IMPORTANT: Re-evaluate whether to use StatusSectionRenderer inside renderSectionHeader vs. making the section header rendering inline. The StatusSectionRenderer was designed as a generic component, but for SectionList integration, the header might work better rendered directly in renderSectionHeader with the StatusSectionRenderer providing just the header UI (not wrapping items). Adjust the integration to make it clean -- the key requirement is that the section renderer is reusable for future section types.

    **Empty state** (per CONTEXT.md: "No items yet" centered):

    ```typescript
    ListEmptyComponent={
      !loading ? (
        <View style={styles.empty}>
          <Text style={styles.emptyText}>No items yet</Text>

    ```

    **Loading spinner** (per CONTEXT.md: "Spinner while loading"):
    Show ActivityIndicator centered on screen while `loading` is true and `!hasLoaded`.

    **No screen header/title** (per CONTEXT.md locked decision):
    Do NOT add a heading like Inbox has. Sections start at the top of the screen.

    - Empty text: centered, color "#666", fontSize 16
