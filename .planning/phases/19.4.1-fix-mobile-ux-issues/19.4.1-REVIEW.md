## Findings

### WR-01: SecureStore.getItemAsync error not caught in ApiKeyContext

**Severity:** warning
**File:** /Users/willmacdonald/Documents/Code/claude/second-brain/mobile/contexts/ApiKeyContext.tsx:35
**Issue:** The `SecureStore.getItemAsync("api_key")` call uses `.then()` without a `.catch()` handler. If SecureStore throws (e.g., Keychain unavailable, device locked with data protection), the promise rejection is unhandled. The `isLoading` state would remain `true` forever, causing the app to show nothing (ApiKeyGate checks `!isLoading` to decide visibility, so neither the gate nor the main app would render).
**Suggestion:** Add a `.catch()` handler that sets `isLoading` to false and leaves `apiKey` empty, which would show the ApiKeyGate modal as a graceful fallback:
```typescript
SecureStore.getItemAsync("api_key")
  .then((stored) => {
    const key = stored ?? "";
    setApiKeyState(key);
    setRuntimeApiKey(key);
  })
  .catch(() => {
    // Keychain unavailable -- show gate to re-enter key
  })
  .finally(() => {
    setIsLoading(false);
  });
```

### WR-02: Settings screen handleSave swallows SecureStore write errors

**Severity:** warning
**File:** /Users/willmacdonald/Documents/Code/claude/second-brain/mobile/app/settings.tsx:42
**Issue:** `handleSave` calls `await setApiKey(trimmed)` which calls `SecureStore.setItemAsync`. If the SecureStore write fails, the `await` throws and the entire handler aborts silently -- the user sees no feedback that the save failed, and the edit state remains frozen (isEditing stays true, but editValue was already cleared on line 48 which runs after the await). Actually, the throw would prevent lines 47-49 from running, leaving the user in edit mode with the trimmed value still in the input, which is acceptable, but there is no error indication.
**Suggestion:** Wrap in try/catch and show an error toast or alert:
```typescript
const handleSave = async () => {
  const trimmed = editValue.trim();
  if (!trimmed) return;
  try {
    await setApiKey(trimmed);
    setIsEditing(false);
    setEditValue("");
    setShowToast(true);
    setTimeout(() => setShowToast(false), 2000);
  } catch {
    Alert.alert("Error", "Failed to save API key. Please try again.");
  }
};
```

### WR-04: `require("react-native")` inside render function in tab layout

**Severity:** warning
**File:** /Users/willmacdonald/Documents/Code/claude/second-brain/mobile/app/(tabs)/_layout.tsx:60
**Issue:** The `TabIcon` component uses `const { Text } = require("react-native")` inline within the component body. This is a dynamic require inside a render function, which runs on every render call. While Metro handles this correctly at runtime, it is an anti-pattern: it bypasses static analysis, tree-shaking, and TypeScript type checking for the `Text` import. It also makes the import invisible to bundle analyzers.
**Suggestion:** Move the import to the top of the file:
```typescript
import { Tabs } from "expo-router";
import { Text } from "react-native";
```
Then simplify `TabIcon`:
```typescript
function TabIcon({ label, color }: { label: string; color: string }) {
  return <Text style={{ fontSize: 20, color }}>{label}</Text>;
}
```
