**1. Install expo-updates:**

Run from the mobile directory:
```bash
cd /Users/willmacdonald/Documents/Code/claude/second-brain/mobile && npx expo install expo-updates
```

This adds `expo-updates` to package.json dependencies with the SDK 54 compatible version. It may also add an `expo-updates` entry to app.json plugins -- verify after installation.

**2. Modify mobile/app.json:**

Add `runtimeVersion` and `updates` configuration to the expo object. These go at the top level of the expo config (sibling to `version`, `slug`, etc.):

```json
"runtimeVersion": {
  "policy": "appVersion"
},
"updates": {
  "url": "https://u.expo.dev/81ec77de-a22f-49c9-aed8-29d81b0ca951"
}
```

The `policy: "appVersion"` ties the runtime version to the `version` field ("1.0.0"). OTA updates only apply to builds with matching app version. When native modules change, bump the version.

The URL uses the existing EAS project ID from `extra.eas.projectId`.

If `npx expo install expo-updates` did NOT auto-add it to plugins, add `"expo-updates"` to the plugins array.

**3. Modify mobile/eas.json:**

Add `channel` field to each build profile for OTA update routing:

```json
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "channel": "development",
      "env": { ... existing ... }
    },
    "preview": {
      "distribution": "internal",
      "channel": "preview",
      "env": { ... existing ... }
    },
    "production": {
      "autoIncrement": true,
      "channel": "production",
      "env": { ... existing ... }
    }
  }
}
```

Note: `EXPO_PUBLIC_API_KEY` is intentionally NOT added to the production env. The API key is prompted on first launch (Plan 01) and stored in SecureStore (D-09).

Verify the production profile does NOT contain `EXPO_PUBLIC_API_KEY`. It should only have `EXPO_PUBLIC_API_URL` and `EXPO_PUBLIC_SENTRY_DSN`.
