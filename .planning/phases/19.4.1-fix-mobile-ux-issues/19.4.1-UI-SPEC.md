# Phase 19.4.1 — UI Design Contract

> Visual and interaction contract for mobile app restructuring: 4-tab layout, Tasks tab, Status simplification, error card enhancement, first-launch API key prompt, and Settings screen.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, `marginTop` on section dividers |
| sm | 8px | Card row gaps, `gap` between dashboard cards, `paddingTop` on screen headers |
| md | 16px | Default horizontal padding (`paddingHorizontal: 16`), card internal padding, tile padding |
| lg | 24px | Not currently used explicitly — available for future section padding |
| xl | 32px | Empty state padding, `padding: 32` on empty views |
| 2xl | 48px | Not currently used |
| 3xl | 64px | Not currently used |

Exceptions: `14px` used for section header `paddingVertical` and row internal `padding` (ErrandRow, TaskRow). `12px` used for card padding, toast padding, processing banner padding. These pre-existing non-multiples-of-4 values are inherited from the existing codebase and are preserved for consistency.

### S-02: Tasks Tab (NEW)

**File:** `mobile/app/(tabs)/tasks.tsx` (new file)

**Purpose:** Displays all actionable items (errands grouped by destination + general tasks). This is the content currently on the Status screen's SectionList, extracted into its own dedicated tab. The Tasks tab takes over as the trigger for Admin Agent processing (D-04).

**Layout structure:**

```
SafeAreaView (flex: 1, bg: #0f0f23)
  |-- Screen Header Row
  |     |-- Text "Tasks" (fontSize: 28, fontWeight: bold, color: #fff)
  |
  |-- SectionList
        |-- [Processing Banner] (conditional, when processingCount > 0)
        |-- [Admin Notifications] (conditional, dismissible)
        |-- Section Headers (collapsed by default, D-13)
        |     |-- StatusSectionRenderer (reused as-is)
        |-- Section Items (when expanded)
        |     |-- ErrandRow (reused as-is, swipe-to-delete)
        |     |-- RoutingPicker (inline below unrouted items, reused as-is)
        |     |-- TaskRow (reused as-is, swipe-to-done)
        |
        |-- [ListEmptyComponent] (when no sections)
```

**Data fetching:**
- On tab focus (`useFocusEffect`): `GET /api/errands` + `GET /api/tasks` (identical to current status.tsx logic)
- Focus polling: 4 extra polls at 3000ms interval after focus (catches Admin Agent processing results)
- Processing polling: continues while `processingCount > 0`
- Admin Agent trigger: `GET /api/errands` on focus triggers processing as side-effect (D-04)

**Section behavior:**
- All sections start COLLAPSED (D-13) — `expandedSections` initialized as empty `Set<string>`
- Tapping section header toggles expand/collapse via `StatusSectionRenderer`
- Errand sections grouped by destination (physical, online, unrouted)
- Task section appears last (key: "tasks")

**Empty state (D-14):**
```
View (alignItems: center, padding: 32)
  Text "No tasks" (fontSize: 16, color: #666)
```

Minimal single-line text matching existing `emptyText` style in status.tsx. No subtitle, no call-to-action.

**Processing banner:** Same as current status.tsx:
```
View (flexDirection: row, bg: #1a1a2e, border: 1px #4a90d9, borderRadius: 10, padding: 12, marginHorizontal: 16, marginTop: 8)
  ActivityIndicator (small, color: #4a90d9)
  Text "Processing N new capture(s)..." (fontSize: 14, color: #4a90d9, fontWeight: 500)
```

**Admin notification banners:** Same as current status.tsx, dismissible, positioned above sections.

**Interactions:**
- Swipe-to-delete on errands (optimistic removal with rollback on failure)
- Swipe-to-done on tasks (optimistic removal with rollback on failure)
- Tap routing chips on unrouted items
- Tap section headers to expand/collapse
- Dismiss notification banners

**Components reused verbatim:** `StatusSectionRenderer`, `ErrandRow`, `TaskRow`, `RoutingPicker`

### S-04: Error Card Enhancement

**File:** `mobile/components/DashboardCards.tsx` (modified)

**Current behavior:**
- Error card shows: "N error(s)" or "None"
- Deep-link sends: `"Show me the recent errors from the last 24 hours with details"`

**New behavior (D-06, D-07):**

**Recency info:** Error card displays both the count AND how recently the last error occurred:
```
"4 errors -- last: 3h ago"
```

This requires either:
- (a) A new `lastErrorAge` field from `GET /api/health-summary` (backend change, Claude's discretion per CONTEXT.md), OR
- (b) Client-side computation if the backend returns a `lastErrorTimestamp`

**Card display logic:**
```
if loading:           "..."
if errorCount == 0:   "None"       (color: #4a90d9)
if errorCount > 0:    "N error(s)  (color: #ff6b6b)
                       last: Xh ago"
```

The recency line is a second line within the card, using a smaller/muted style:
```
Text "last: 3h ago" (fontSize: 11, color: #ff6b6b, opacity: 0.7, marginTop: 2)
```

**Time-ranged deep-link query (D-06):** The investigation query sent on error card tap includes an explicit time window so the investigation agent searches the correct range:

```typescript
onErrorPress={() => {
  router.push({
    pathname: "/investigate",
    params: {
      initialQuery:
        "Show me the errors from the last 24 hours with full details including error messages and affected captures",
    },
  });
}}
```

The exact query wording is at Claude's discretion, but it MUST include "last 24 hours" (or equivalent time range language) to match the dashboard's 24h window. The current query ("Show me the recent errors from the last 24 hours with details") already has this, but should be strengthened to ensure the investigation agent calls `recent_errors` with `hours=24`.

**DashboardData interface update:**
```typescript
export interface DashboardData {
  captureCount: number | null;
  successRate: number | null;
  errorCount: number | null;
  lastErrorAge: string | null;  // NEW: human-readable age, e.g. "3h ago", "12m ago"
  loading: boolean;
}
```

### S-06: First-Launch API Key Prompt

**Purpose:** On first launch (no API key stored), prompt the user to enter their API key before using the app (D-09, D-10).

**Detection logic:**
```typescript
// In root layout or a dedicated auth gate component:
const [apiKey, setApiKey] = useState<string | null>(null);
const [checking, setChecking] = useState(true);

useEffect(() => {
  SecureStore.getItemAsync("api_key").then((key) => {
    setApiKey(key);
    setChecking(false);
  });
}, []);
```

**Approach: Full-screen modal overlay**

When no API key is stored, display a full-screen modal (not a tab screen) that blocks interaction with the rest of the app.

**Layout:**

```
Modal (visible: !apiKey && !checking, animationType: "fade")
  SafeAreaView (flex: 1, bg: #0f0f23)
    |-- View (flex: 1, justifyContent: center, paddingHorizontal: 24)
          |-- Text "Second Brain" (fontSize: 28, fontWeight: bold, color: #fff, textAlign: center)
          |-- Text "Enter your API key to get started"
          |     (fontSize: 15, color: #888, textAlign: center, marginTop: 8, marginBottom: 32)
          |
          |-- Text "API KEY" (label style: fontSize: 11, fontWeight: 600, color: #888, uppercase, marginBottom: 4)
          |-- TextInput
          |     (bg: #1a1a2e, borderRadius: 10, padding: 14, color: #fff, fontSize: 15)
          |     secureTextEntry, placeholder: "Paste your API key", placeholderTextColor: #666
          |     autoFocus: true
          |
          |-- Pressable "Continue"
          |     (bg: #4a90d9, borderRadius: 10, paddingVertical: 14, marginTop: 16, alignItems: center)
          |     disabled when input is empty (opacity: 0.4)
          |     Text (color: #fff, fontSize: 16, fontWeight: 600)
          |
          |-- Text "You can change this later in Settings"
                (fontSize: 12, color: #666, textAlign: center, marginTop: 12)
```

**Continue behavior:**
1. Validate non-empty input
2. Store via `SecureStore.setItemAsync("api_key", value)`
3. Update app-level state so the modal dismisses
4. App loads normally with the stored key

**Dev builds (D-12):** In development, if `EXPO_PUBLIC_API_KEY` env var is set, skip the first-launch prompt entirely and use the env var. The prompt only appears when `EXPO_PUBLIC_API_KEY` is empty AND `SecureStore` has no stored key.

**Config refactoring (`mobile/constants/config.ts`):**

The current `API_KEY` is a synchronous constant. It must be refactored to support async retrieval from `SecureStore`:

```typescript
// Option A: Async getter function
export async function getApiKey(): Promise<string> {
  // Dev override
  const envKey = process.env.EXPO_PUBLIC_API_KEY;
  if (envKey) return envKey;

  // Production: read from secure store
  const stored = await SecureStore.getItemAsync("api_key");
  return stored ?? "";
}

// Option B: React context with cached value (recommended)
// ApiKeyProvider wraps the app, provides { apiKey, setApiKey } via context.
// All fetch calls read from context instead of importing a constant.
```

The implementation approach is at Claude's discretion, but all existing `API_KEY` import sites must continue to work. The synchronous `API_KEY` export should remain available for backward compatibility during transition, falling back to empty string when no env var is set.

## Interaction Patterns

### Tab Navigation
- Tabs are always visible at the bottom (no hide-on-scroll)
- Tab transitions are instant (no animation between tabs)
- Inbox badge count continues to work as-is (derived from pending items)
- Tasks tab has no badge (errands/tasks don't need pending indicators)

### Data Flow on Tab Focus
| Tab | On Focus | Polling |
|-----|----------|---------|
| Capture | Permission check (voice mode) | None |
| Inbox | `GET /api/inbox` | None (pull-to-refresh available) |
| Tasks | `GET /api/errands` + `GET /api/tasks` | 4 focus polls + processing polls |
| Status | `GET /api/health-summary` | None (SpineStatusTile has its own 30s interval) |

### Swipe Gestures
- ErrandRow: Swipe left reveals red "Delete" action (threshold: 80px)
- TaskRow: Swipe left reveals green "Done" action (threshold: 80px)
- Both use optimistic removal with rollback on API failure

### Navigation Map

```
Tab Bar
  |-- Capture (index.tsx)
  |-- Inbox (inbox.tsx)
  |     |-- [Modal] Item detail card
  |-- Tasks (tasks.tsx) [NEW]
  |-- Status (status.tsx) [SIMPLIFIED]
        |-- [Push] /investigate
        |-- [Push] /settings [NEW]
        |-- [External] Spine web links (Linking.openURL)
```
