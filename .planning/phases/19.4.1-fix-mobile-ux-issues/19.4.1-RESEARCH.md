# Phase 19.4.1: Fix Mobile UX Issues - Research

**Researched:** 2026-04-21

## Summary

This phase restructures the mobile app from 3 tabs to 4 tabs (Capture | Inbox | Tasks | Status), extracts errands/tasks into a dedicated Tasks tab, simplifies the Status tab to be purely observational, adds a first-launch API key prompt with SecureStore, enhances the error dashboard card with recency info and time-ranged deep-link, and ships a production EAS build to TestFlight with OTA update capability.

The work is primarily a refactoring and feature enhancement within the existing React Native + Expo Router codebase. No new npm packages are required -- expo-secure-store is already installed and configured. The backend needs a small enhancement to include last-error timestamp in the health-summary endpoint. The EAS Build and Update setup requires configuration changes to eas.json and app.json but no new native modules.

**Primary recommendation:** Execute as 5 work streams: (1) Tab restructuring + Tasks tab extraction, (2) Status tab simplification + error card enhancement, (3) API key management with SecureStore + settings screen, (4) Backend health-summary recency field, (5) EAS production build + Update configuration. Streams 1-2 are tightly coupled; stream 3 touches all API call sites; stream 4 is a small backend change; stream 5 is a configuration/build step done last.

### Pattern 1: React Context for API Key (Recommended)
**What:** Wrap the app in an `ApiKeyProvider` that reads from SecureStore on mount and provides `{ apiKey, setApiKey, isLoading }` to all consumers.
**When to use:** When a value needs to be read asynchronously at startup and consumed synchronously by many components.
**Example:**
```typescript
// Source: Project-specific pattern, verified against expo-secure-store docs
import * as SecureStore from "expo-secure-store";
import { createContext, useContext, useState, useEffect, ReactNode } from "react";

interface ApiKeyContextType {
  apiKey: string;
  setApiKey: (key: string) => Promise<void>;
  isLoading: boolean;
}

const ApiKeyContext = createContext<ApiKeyContextType>({
  apiKey: "",
  setApiKey: async () => {},
  isLoading: true,
});

export function ApiKeyProvider({ children }: { children: ReactNode }) {
  const [apiKey, setApiKeyState] = useState("");
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    // Dev override: use env var if set
    const envKey = process.env.EXPO_PUBLIC_API_KEY;
    if (envKey) {
      setApiKeyState(envKey);
      setIsLoading(false);
      return;
    }
    // Production: read from SecureStore
    SecureStore.getItemAsync("api_key").then((stored) => {
      setApiKeyState(stored ?? "");
      setIsLoading(false);
    });
  }, []);

  const setApiKey = async (key: string) => {
    await SecureStore.setItemAsync("api_key", key);
    setApiKeyState(key);
  };

  return (
    <ApiKeyContext.Provider value={{ apiKey, setApiKey, isLoading }}>
      {children}
    </ApiKeyContext.Provider>
  );
}

export const useApiKey = () => useContext(ApiKeyContext);
```

### Pattern 2: Tab Screen Extraction
**What:** Move errands/tasks state + data fetching from status.tsx into a new tasks.tsx, keeping all component imports and interaction handlers identical.
**When to use:** When splitting a screen into two screens that share components but not state.
**Example:** The current status.tsx has ~560 lines. The Tasks tab will take the SectionList, all errand/task state, fetchData, polling logic, and CRUD handlers. The Status tab keeps only DashboardCards, SpineStatusTile, and fetchDashboardData.

### Pattern 3: Backward-Compatible Config Refactoring
**What:** Keep the synchronous `API_KEY` export from config.ts as a fallback (reads env var, defaults to empty string) while adding the React Context as the primary source. Existing code that uses `API_KEY` directly continues to work in dev builds where env vars are set.
**When to use:** When migrating from synchronous to async config without breaking all call sites at once.

### Anti-Patterns to Avoid
- **Importing SecureStore in every component:** Use a single context provider. Don't scatter SecureStore.getItemAsync calls across 8+ files.
- **Blocking render on SecureStore read:** SecureStore.getItemAsync is async. Show a loading state, don't use the synchronous getItem() which blocks the JS thread.
- **Conditional tab rendering:** Don't hide the Tasks tab based on API key state. All 4 tabs should always be visible; the API key gate is a modal overlay, not a navigation guard.
- **Modifying StatusSectionRenderer/ErrandRow/TaskRow:** These components are reused verbatim on the Tasks tab. Don't modify them; just import them in the new file.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Encrypted storage | Custom encryption | expo-secure-store | Uses iOS Keychain / Android Keystore; handles encryption, key rotation, biometric support |
| OTA updates | Custom update server | EAS Update + expo-updates | Apple-compliant OTA for JS bundles; handles version compatibility, rollbacks |
| Tab navigation | Custom bottom bar | expo-router Tabs | File-based routing handles deep links, back navigation, state persistence |
| Time-ago formatting | Custom date math | Simple inline function | The recency display is "Xh ago" or "Xm ago" -- simple enough to compute inline (divide ms by 3600000). Don't add a date library for this. |

**Key insight:** This phase is almost entirely refactoring and reconfiguration of existing code. The main risk is breaking existing functionality during extraction, not building new complex features.

## Common Pitfalls

### Pitfall 1: API Key Propagation to Utility Modules
**What goes wrong:** The `telemetry.ts` and `spine.ts` modules import `API_KEY` at module scope. After refactoring to use React Context, these non-component files can't use hooks.
**Why it happens:** React Context only works inside the component tree. Utility functions outside components need a different mechanism.
**How to avoid:** Two approaches: (a) Pass apiKey as a parameter to utility functions (cleanest), or (b) Keep the synchronous `API_KEY` export for utility modules and only use Context in components. Since dev builds use env vars and production builds use SecureStore, the synchronous export will be empty string in production -- so utility functions MUST receive the key as a parameter in production.
**Warning signs:** `API_KEY` is empty string at runtime in production builds; telemetry/spine calls silently fail with 401.

### Pitfall 2: Expo Router Tab Ordering
**What goes wrong:** Expo Router uses file-system ordering for tabs by default. Adding `tasks.tsx` might place it alphabetically after `status.tsx`.
**Why it happens:** Without explicit `Tabs.Screen` entries in `_layout.tsx`, expo-router uses alphabetical file order.
**How to avoid:** The existing `_layout.tsx` explicitly defines `Tabs.Screen` entries with `name` props. Add the new `tasks` screen between `inbox` and `status` in the layout. This is already the pattern used.
**Warning signs:** Tasks tab appears in wrong position in the tab bar.

### Pitfall 3: Focus Effect Cleanup on Tab Switch
**What goes wrong:** If the Tasks tab's `useFocusEffect` cleanup doesn't properly clear intervals, polling timers from Tasks tab fire while Status tab is active (or vice versa).
**Why it happens:** The current status.tsx has careful cleanup in useFocusEffect's return function and in useEffect cleanup. When extracting to tasks.tsx, missing any cleanup path leaks timers.
**How to avoid:** Copy the polling logic exactly from status.tsx. The focus polling (focusPollRef) and processing polling (pollingRef) cleanup patterns are battle-tested. Don't refactor the polling approach while extracting.
**Warning signs:** Multiple fetchData calls firing simultaneously, or fetching errands from the Status tab.

### Pitfall 4: SecureStore Async Timing
**What goes wrong:** The app renders before SecureStore.getItemAsync resolves, causing the first API call to use an empty API key.
**Why it happens:** SecureStore is async; React renders synchronously.
**How to avoid:** The ApiKeyProvider sets `isLoading: true` until the key is loaded. The ApiKeyGate modal blocks interaction until loading completes. Components should check `isLoading` before making API calls.
**Warning signs:** Brief flash of empty/error state on first render; 401 errors in logs followed by successful retry.

### Pitfall 5: EAS Update Runtime Version Mismatch
**What goes wrong:** OTA updates fail to apply because the runtime version of the update doesn't match the build.
**Why it happens:** If `runtimeVersion` isn't configured correctly, or if native changes are made after the initial build, the update's runtime version won't match.
**How to avoid:** Use `"runtimeVersion": { "policy": "appVersion" }` in app.json. This ties runtime version to the app version string ("1.0.0"). OTA updates will only apply to builds with the same app version. When native modules change, bump the version.
**Warning signs:** Updates published but not appearing on device; "Update rejected by selection policy" in logs.

### Pitfall 6: EAS Build Missing API Key in Production
**What goes wrong:** Production build ships with `EXPO_PUBLIC_API_KEY` not set in eas.json. App opens, SecureStore is empty, first-launch prompt shows. User enters key. Works correctly. BUT: if any code path still reads the synchronous `API_KEY` constant from config.ts, it will be empty string forever in production.
**Why it happens:** The synchronous constant `process.env.EXPO_PUBLIC_API_KEY ?? ""` is baked at build time. In production profile, this env var is intentionally not set.
**How to avoid:** Ensure ALL fetch calls use the API key from React Context (or passed as parameter), never from the synchronous constant, in production builds. The synchronous constant only works in dev/preview builds where the env var is set.
**Warning signs:** Some API calls work (context-sourced) while others fail (constant-sourced) after entering key on first launch.

### Pitfall 7: Auto-Format Hook Strips Unused Imports
**What goes wrong:** Adding an import in one edit step that's only used in a subsequent edit causes ruff (or ESLint) to strip it before it's referenced.
**Why it happens:** The auto-format PostToolUse hook runs after each file write. Documented in MEMORY.md Phase 17.1 lesson.
**How to avoid:** Do import addition + usage in a single Write that replaces the whole file. Or add the import last, after the usage is in place.
**Warning signs:** `ReferenceError: X is not defined` after editing a file in steps.

### Client-Side Recency Computation
```typescript
// Source: Project-specific pattern
function formatErrorAge(isoTimestamp: string | null): string | null {
  if (!isoTimestamp) return null;
  const diffMs = Date.now() - new Date(isoTimestamp).getTime();
  const diffHours = Math.floor(diffMs / 3_600_000);
  if (diffHours < 1) {
    const diffMins = Math.max(1, Math.floor(diffMs / 60_000));
    return `${diffMins}m ago`;
  }
  if (diffHours < 24) return `${diffHours}h ago`;
  return `${Math.floor(diffHours / 24)}d ago`;
}
```

## Open Questions

1. **EAS Update: expo-updates installation requirement**
   - What we know: EAS Update requires `expo-updates` library to be installed. It's not currently in package.json.
   - What's unclear: Whether `npx expo install expo-updates` requires a native rebuild (yes, it adds a native module). This means the EAS Update setup must happen BEFORE the production build, not after.
   - Recommendation: Install expo-updates as the very first step of the EAS work stream, then build once with all changes.

2. **Apple Developer Account prerequisites for TestFlight**
   - What we know: Submitting to TestFlight requires an Apple Developer Account, distribution certificate, and provisioning profile. The app already has `bundleIdentifier: "com.willkmacdonald.secondbrain"` and `ITSAppUsesNonExemptEncryption: false` configured.
   - What's unclear: Whether Will has an active Apple Developer Program membership and whether `eas credentials` has been configured for this project.
   - Recommendation: Add a prerequisite check task before the TestFlight submission. If credentials aren't set up, `eas credentials --platform ios` will handle the setup interactively.

3. **Health summary last-error time: App Insights vs Cosmos**
   - What we know: The SYSTEM_HEALTH KQL already queries AppTraces with SeverityLevel >= 3. Adding `max(TimeGenerated)` is straightforward.
   - What's unclear: Whether the KQL max() returns an ISO string or datetime object through the Python SDK.
   - Recommendation: Return as ISO string from backend; client computes "Xh ago" display. This avoids timezone issues.

4. **API key migration for existing dev builds**
   - What we know: Dev builds use `EXPO_PUBLIC_API_KEY` env var. This will continue working (D-12).
   - What's unclear: Whether existing preview builds (already installed via EAS) will get the first-launch prompt if rebuilt with the new code but without the env var.
   - Recommendation: Keep `EXPO_PUBLIC_API_KEY` in the preview profile in eas.json to avoid disrupting existing dev workflow. Only the production profile omits it.
