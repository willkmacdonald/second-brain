<objective>
Fix the KQL query projection gap that causes severity-3 exceptions in App
Insights to surface as bare class names (e.g., "HttpResponseError") with no
status code, response body, or inner message. The rich fields are captured
by `configure_azure_monitor` on the AppExceptions table -- our three KQL
templates just throw them away. This phase is additive everywhere (new
optional Pydantic fields, new KQL projection columns, new MCP truncation
helper) and makes no producer-side changes.

Purpose: After shipping, re-running `recent_errors(time_range="7d")`
against deployed App Insights returns the two existing HttpResponseError
rows (classifier 4/13, investigation_agent 4/11) with populated
outer_message, outer_type, innermost_message, and details -- enough
diagnostic signal for Claude Code (or the investigation agent) to
reason about the failure without a follow-up portal dive.

Output:
- Updated KQL in three templates
- Four new optional fields on FailureRecord and TraceRecord
- Parser mappings in three query functions
- Details truncation helper in MCP tools
- Two new test classes; five existing regression tests untouched
</objective>

```python
DETAILS_MAX_CHARS = 2000  # truncate verbose stack-trace JSON in MCP responses


def _apply_details_truncation(event: dict) -> dict:
    """Truncate verbose AppExceptions Details JSON to keep MCP responses compact.

    Mutates the incoming dict in place and returns it. If `details` is
    absent or short enough, the dict is unchanged.
    """
    details = event.get("details")
    if details and isinstance(details, str) and len(details) > DETAILS_MAX_CHARS:
        event["details"] = details[:DETAILS_MAX_CHARS] + "...[truncated]"
        event["details_truncated"] = True
    return event
```
