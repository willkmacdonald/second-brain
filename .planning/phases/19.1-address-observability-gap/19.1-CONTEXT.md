# Phase 19.1: address observability gap - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning
**Source:** Investigation during Phase 19 RETEST UAT (2026-04-13 → 2026-04-14)

<domain>
## Phase Boundary

Fix the KQL query projection gap that causes severity 3 exceptions in App
Insights to appear as bare class names (e.g., "HttpResponseError") with no
status code, response body, or inner message. The exception data IS being
captured correctly by `configure_azure_monitor` — it's our query templates
that are throwing it away.

**In scope:**
- Update KQL query templates in `backend/src/second_brain/observability/kql_templates.py`
  to project `OuterMessage`, `Details`, `OuterType`, `InnermostMessage`, and
  `ResultCode` from `AppExceptions` rows (in addition to existing fields)
- Update Pydantic result models in `backend/src/second_brain/observability/queries.py`
  to carry the new fields
- Update MCP tool return shape in `mcp/server.py` so Claude Code sees
  actionable error detail
- Preserve existing regression tests in `backend/tests/test_investigation_queries.py`
  (severity-level int pinning must not break)

**Out of scope (explicitly):**
- Foundry-agent internal visibility (covered by Phase 17.4)
- Changes to exception logging in the classifier / adapter (existing
  `logger.error(..., exc_info=True)` pattern is correct; the capture
  pipeline works)
- New App Insights custom metrics or alerts
- Mobile UI changes to display new fields
- Changes to the investigation agent's Foundry-portal prompt (the new
  fields will flow through automatically via existing queries; no prompt
  rewrite needed)

</domain>

<decisions>
## Implementation Decisions

### Root cause (locked)
The three KQL templates in `kql_templates.py` project `AppExceptions` rows
with only `Message = coalesce(Message, ExceptionType)`. For
`HttpResponseError` and similar Azure SDK exceptions, the `Message` column
is empty, so only the exception class name is surfaced. The rich fields
(`OuterMessage`, `Details`, `ResultCode`, `OuterType`,
`InnermostMessage`) are present in the underlying Azure Monitor data but
never queried.

### Affected KQL templates (locked)
All three must be updated:
1. `CAPTURE_TRACE` — per-trace event log (used by `trace_lifecycle`)
2. `RECENT_FAILURES` — unfiltered error list
3. `RECENT_FAILURES_FILTERED` — error list with severity + component
   filtering (used by `recent_errors`)

### Field-inclusion policy (locked)
Projections must include, for `AppExceptions` rows:
- `OuterMessage` — human-readable message with status code embedded in prose (e.g., `"(429) Too Many Requests: ..."`)
- `OuterType` — fully-qualified exception class (e.g., `azure.core.exceptions.HttpResponseError`)
- `InnermostMessage` — root-cause message when exception was wrapped
- `Details` — stack frames and structured detail (dynamic type — must be `tostring(Details)` in KQL projection; may be verbose — truncate at MCP layer if >2000 chars)

For `AppTraces` rows, keep existing projection unchanged — `Message` is
already populated correctly by our `logger.error(...)` calls.

### ResultCode NOT projected (locked, revised 2026-04-14)
The `AppExceptions` workspace table has no `ResultCode` column (verified
against Microsoft Learn schema docs, 2026-03-11 revision). Rather than
synthesize it via brittle regex extraction from `OuterMessage`, we accept
that the HTTP status code is visible inline in `OuterMessage` prose
(e.g., `"(429) Too Many Requests: ..."`). Claude Code reads it from
there on demand. If structured filtering/aggregation by status code
becomes important later, that's a separate phase with its own
extraction strategy.

### Backwards compatibility (locked)
The existing `Message` field in query results MUST remain populated so
downstream consumers don't break. New fields are additive, not replacements.
Preferred synthesis:
```
Message = coalesce(OuterMessage, Message, InnermostMessage, ExceptionType, Name)
```
This elevates the rich `OuterMessage` to the `Message` slot so existing
consumers immediately see better data without code changes, and keeps the
raw fields available for structured access. For AppTraces rows (where
`OuterMessage` is null), coalesce falls through to `Message` as before —
no behavior change.

### Result model changes (locked)
`backend/src/second_brain/observability/queries.py` models for
`FailureRecord` and `TraceRecord` get new optional fields:
`outer_message`, `outer_type`, `innermost_message`, `details`. All
`Optional[str] = None`, reusing the existing `_empty_to_none`
`@field_validator(mode="before")` pattern so the new field names just
get added to the existing validator decorator.

### MCP tool return shape (locked)
`mcp/server.py` tools (`recent_errors`, `trace_lifecycle`) return the new
fields in their JSON response. RESULT_LIMIT=20 unchanged.

### `Details` truncation (Claude's Discretion)
The `Details` column on AppExceptions can be very long (full stack
traces). Planner should decide: truncate to N characters in KQL, summarize
to top frame, or return full and let Claude Code handle presentation.
Recommendation: project it raw, let the MCP layer truncate if >2000 chars.

### Test strategy (locked)
The 5 regression tests in `backend/tests/test_investigation_queries.py`
that pin severity-level int mappings MUST continue to pass. Add new
tests for the projection-fields contract (assert new fields appear in
result models, assert `Message` coalesce preserves existing behavior).

</decisions>

<specifics>
## Specific Ideas

### Evidence from RETEST UAT (2026-04-13)
Running `trace_lifecycle(trace_id="219b58c9-bed7-4be6-b115-f43714dc8920")`
returned a single event:
```json
{
  "timestamp": "2026-04-13 19:08:19.297571+00:00",
  "item_type": "Exception",
  "severity_level": 3,
  "message": "HttpResponseError",
  "component": "classifier"
}
```
`message: "HttpResponseError"` is the exception class name from
`ExceptionType`. The actual HTTP status code, response body, and Azure
request ID are in AppExceptions but never queried.

### Second witness
A matching April 11 error from `investigation_agent` shows the same
shape — `message: "HttpResponseError"`, nothing else. This bug predates
the Phase 19 MCP work; Phase 19 just made it visible.

### File surface (3 files to edit)
- `backend/src/second_brain/observability/kql_templates.py` — the three
  KQL templates
- `backend/src/second_brain/observability/queries.py` — result model
  fields
- `mcp/server.py` — tool return shape (thin wrapper, should pick up
  model changes automatically if using Pydantic serialization)

### Verification approach
After shipping, re-run `recent_errors(time_range="7d")` against the
deployed system. The two existing `HttpResponseError` rows in App
Insights should now return with populated `outer_message`, `outer_type`,
and `result_code` fields. No new deploy needed — these are query-side
changes, not producer-side.

### Non-goal: fixing the HttpResponseError itself
The two captured errors (classifier 4/13, investigation_agent 4/11) are
likely transient Azure rate-limit or network hiccups. Once we can
actually read the status code, we'll know. This phase makes them
diagnosable; any follow-up fix is a separate phase.

</specifics>
