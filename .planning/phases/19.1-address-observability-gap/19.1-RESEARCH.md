## User Constraints (from CONTEXT.md)

### Locked Decisions

**Root cause:** The three KQL templates in `kql_templates.py` project `AppExceptions`
rows with only `Message = coalesce(Message, ExceptionType)`. For `HttpResponseError`
and similar Azure SDK exceptions the `Message` column is empty, so only the
exception class name is surfaced. The rich fields (`OuterMessage`, `Details`,
`ResultCode`, `OuterType`, `InnermostMessage`) are present in the underlying Azure
Monitor data but never queried.

**Affected KQL templates (all three must be updated):**
1. `CAPTURE_TRACE` — per-trace event log (used by `trace_lifecycle`)
2. `RECENT_FAILURES` — unfiltered error list
3. `RECENT_FAILURES_FILTERED` — error list with severity + component filtering (used by `recent_errors`)

**Field-inclusion policy:** Projections must include, for `AppExceptions` rows:
- `OuterMessage` — human-readable message with status code and context
- `OuterType` — fully-qualified exception class (e.g., `azure.core.exceptions.HttpResponseError`)
- `InnermostMessage` — root-cause message when exception was wrapped
- `Details` — stack frames and structured detail (may be verbose — truncate or summarize)
- `ResultCode` — HTTP status code when present (for `HttpResponseError`)

For `AppTraces` rows, keep existing projection unchanged.

**Backwards compatibility:** The existing `Message` field in query results MUST
remain populated. New fields are additive. Preferred synthesis:
```
Message = coalesce(OuterMessage, Message, InnermostMessage, ExceptionType, Name)
```

**Result model changes:** `FailureRecord` and `TraceRecord` (in
`backend/src/second_brain/observability/models.py`) get new optional fields:
`outer_message`, `outer_type`, `innermost_message`, `result_code`, `details`.
Optional so existing `AppTraces`-only rows don't break schema validation.

**MCP tool return shape:** `mcp/server.py` tools (`recent_errors`,
`trace_lifecycle`) return the new fields. `RESULT_LIMIT=20` unchanged.

**Test strategy:** The regression tests in
`backend/tests/test_investigation_queries.py` that pin severity-level int
mappings MUST continue to pass. Add new tests for the projection-fields
contract (assert new fields appear in result models, assert `Message`
coalesce preserves existing behavior).

### Claude's Discretion

**`Details` truncation:** The `Details` column on AppExceptions can be very
long (full stack traces). Planner should decide: truncate to N characters in
KQL, summarize to top frame, or return full and let Claude Code handle
presentation. Recommendation: project it raw, let the MCP layer truncate if
>2000 chars.

## Summary

Phase 19.1 fixes a query-side projection gap: rich `AppExceptions` fields
(`OuterMessage`, `OuterType`, `InnermostMessage`, `Details`) are captured by
`configure_azure_monitor` but thrown away by three KQL templates. The scope is
surgical — three KQL templates, two Pydantic models, one MCP server file — and
there are no new dependencies, no deploy-pipeline changes, and no producer-side
(logging) changes. Everything is additive.

There is **one critical correction** to the locked CONTEXT.md field list: the
official Microsoft Learn `AppExceptions` schema (verified 2026-04-14, table
documentation last updated 2026-03-11) has **no `ResultCode` column**. HTTP
status codes from `HttpResponseError` are captured by OpenTelemetry's
`record_exception` inside the serialized `str(exc)`, which lands in
`OuterMessage` and/or `Message` as a substring (e.g., `"Operation returned an
invalid status 'Too Many Requests'\nContent: ..."`), not as a typed integer
column. The planner must either (a) drop `result_code` from the field list, or
(b) synthesize it via `extract()` from `OuterMessage`. Recommendation below.

**Primary recommendation:** Update the three KQL templates to project
`OuterMessage`, `OuterType`, `InnermostMessage`, and `tostring(Details)` from
`AppExceptions` (null on `AppTraces` rows via the `union` semantics); extend
`FailureRecord` and `TraceRecord` with matching optional fields; let Pydantic
`model_dump()` flow the new fields through `mcp/server.py` unchanged. Replace
`result_code: int | None` with a synthesized `result_code: str | None` via
`extract(@'\d{3}', 0, OuterMessage)` OR drop the field entirely and let Claude
Code read the status code from `outer_message` as prose — see Open Questions.

### Pattern 1: KQL union with null-safe projection

**What:** `AppTraces` and `AppExceptions` have disjoint columns. A `union` of
the two, followed by `project`, returns `null` for columns missing in either
source table. The projection can name `OuterMessage` freely — rows from
`AppTraces` will have `OuterMessage = null`, rows from `AppExceptions` will
have it populated.

**When to use:** Every multi-table union in all three templates.

**Example (verified against official AppExceptions schema):**
```kql
// Source: https://learn.microsoft.com/en-us/azure/azure-monitor/reference/tables/appexceptions
union withsource=SourceTable
    (AppTraces | where SeverityLevel >= 3),
    AppExceptions
| project
    timestamp = TimeGenerated,
    ItemType = case(
        SourceTable == "AppTraces", "Log",
        SourceTable == "AppExceptions", "Exception",
        SourceTable
    ),
    severityLevel = SeverityLevel,
    Message = coalesce(OuterMessage, Message, InnermostMessage, ExceptionType, Name),
    OuterMessage,                            // string (null on AppTraces)
    OuterType,                               // string (null on AppTraces)
    InnermostMessage,                        // string (null on AppTraces)
    Details = tostring(Details),             // dynamic -> string JSON for transport
    Component = tostring(Properties.component),
    CaptureTraceId = tostring(Properties.capture_trace_id)
| order by timestamp desc
| take 50
```

### Pattern 2: Additive Pydantic field evolution

**What:** Add `Optional[str] = None` fields to `FailureRecord` / `TraceRecord`.
Existing rows (AppTraces) deserialize cleanly because all new fields default to
None. The existing `_empty_to_none` validator should be extended to cover the
new fields too, since `tostring(null_dynamic)` returns `""` in KQL.

**When to use:** Both `FailureRecord` and `TraceRecord` in `models.py`.

**Example:**
```python
# Source: existing pattern in backend/src/second_brain/observability/models.py
class FailureRecord(BaseModel):
    timestamp: str
    item_type: str
    severity_level: int | None = None
    message: str
    component: str | None = None
    capture_trace_id: str | None = None
    # NEW (all optional, default None — backwards compatible)
    outer_message: str | None = None
    outer_type: str | None = None
    innermost_message: str | None = None
    details: str | None = None
    result_code: str | None = None  # str, not int — see Open Questions

    @field_validator(
        "component", "capture_trace_id",
        "outer_message", "outer_type", "innermost_message",
        "details", "result_code",
        mode="before",
    )
    @classmethod
    def _empty_to_none(cls, v: str | None) -> str | None:
        if v is None or (isinstance(v, str) and v.strip() == ""):
            return None
        return v
```

### Pattern 3: `Details` truncation at MCP boundary

**What:** The CONTEXT.md recommendation is "project raw, let MCP layer truncate
if >2000 chars." Align with the existing `trace_lifecycle` truncation pattern
in `mcp/server.py` (lines 186-200) which keeps last N records and emits a
`note` field explaining the truncation.

**Why:** The query layer stays pure (no lossy transforms). The MCP tool is the
presentation layer where output-size concerns belong.

**Example:**
```python
# Source: new pattern, parallels existing trace_lifecycle truncation
DETAILS_MAX_CHARS = 2000

def _truncate_details(details: str | None) -> tuple[str | None, bool]:
    """Return (possibly-truncated details, was_truncated)."""
    if details is None or len(details) <= DETAILS_MAX_CHARS:
        return details, False
    return details[:DETAILS_MAX_CHARS] + "...[truncated]", True

# In trace_lifecycle / recent_errors tool:
for event in events:
    if event.get("details"):
        event["details"], event["details_truncated"] = _truncate_details(event["details"])
```

### Anti-Patterns to Avoid

- **Projecting a non-existent column:** `ResultCode` does NOT exist on
  `AppExceptions` (verified against official schema 2026-04-14). A bare
  `ResultCode,` in the projection will make the KQL query fail at parse time
  with `Failed to resolve column reference 'ResultCode'`. Either extract from
  `OuterMessage` via regex or drop the field.
- **Dropping the `coalesce` chain:** Removing `Message` from the `coalesce`
  breaks AppTraces rows where `logger.error("message")` populates `Message` but
  not `OuterMessage`. Keep `Message` as the second element after `OuterMessage`.
- **Changing validator signature mid-sequence:** Adding a new field name to the
  `@field_validator(...)` decorator requires rewriting the decorator in one
  edit (the auto-format hook will not strip it, but decorator changes are
  atomic). See `MEMORY.md` note about auto-format hook stripping unused
  imports immediately — same risk applies if the validator references a new
  field before the field declaration is added.
- **Hand-rolling the exception-chain walk:** Azure Monitor already flattens the
  outer/inner chain into `OuterMessage`/`InnermostMessage`. Don't parse
  `Details` JSON to reconstruct it — just read the two fields.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Exception chain flattening | Custom walker over `__cause__` / `__context__` | `OuterMessage`, `InnermostMessage`, `OuterType`, `InnermostType` columns | Azure Monitor's exception schema already does this (documented in official schema reference) |
| Stack-frame parsing | Regex over `str(traceback)` | `Details` dynamic column (array of `{message, type, parsedStack}`) | Azure serializes Python tracebacks into structured JSON on ingest |
| HTTP status extraction | Introspect `exc.response.status_code` inside a logging wrapper | Read the status from `OuterMessage` substring or skip and let Claude Code infer | Requires producer-side changes (out of scope) |
| String truncation utility | Custom `truncate_with_ellipsis()` | Python `s[:N] + "..."` | One-liner; no library needed |
| KQL template rendering | Jinja2 / custom interpolator | `str.format()` with `{name}` placeholders | Existing pattern in `kql_templates.py` |

**Key insight:** AppExceptions already stores the unwound exception chain as
separate columns. This phase is about *surfacing* fields that already exist —
the data is in the workspace, it's the query that's lossy.

## Common Pitfalls

### Pitfall 1: `Details` is `dynamic`, not `string`

**What goes wrong:** Projecting `Details,` bare returns a KQL dynamic value.
When the Azure SDK serializes it to a Python dict, the parser in `queries.py`
calls `str(row.get("Details"))` which produces `"<dynamic object>"` or the
repr of a list, not the JSON you want.

**Why it happens:** Confusing string columns (`OuterMessage`) with dynamic
columns (`Details`, `Properties`, `Measurements`).

**How to avoid:** Always wrap dynamic columns in `tostring()` in the projection:
`Details = tostring(Details)`. This serializes the dynamic value to a JSON
string at the KQL layer, so the Python parser receives a plain `str`.

**Warning signs:** Pydantic validation error `Input should be a valid string`
on the `details` field, or the Azure SDK returning an unexpected type from
`LogsQueryClient.query_workspace()`.

### Pitfall 2: `ResultCode` doesn't exist on AppExceptions

**What goes wrong:** The CONTEXT.md-locked field list includes `ResultCode`,
but the official schema has no such column. A projection like
`ResultCode,` will fail with a KQL parse error, crashing the query entirely
and returning empty results for the whole tool call (not just missing the
one field).

**Why it happens:** CONTEXT.md was written from inference, not from the
official schema. Portal-UX docs sometimes suggest "status code" is on the
row, but that refers to `AppRequests.ResultCode`, a different table.

**How to avoid:** Either (a) drop `result_code` from the locked field set
(planner should raise this as a deviation from CONTEXT.md), or (b) synthesize
it via `extract(@"'(\d{3})'", 1, OuterMessage)` to grab the 3-digit code
between single quotes in the conventional `HttpResponseError` message format
(`"Operation returned an invalid status '429'"`). Recommended: (a) drop, add
a follow-up phase for typed status extraction if Claude Code's text parsing
proves insufficient in UAT.

**Warning signs:** `FailedQueryExecutionException` during tool calls,
`execute_kql` returning `tables=[]` unexpectedly, KQL error messages
mentioning "column reference".

### Pitfall 3: `coalesce` ordering changes `Message` semantics for AppTraces

**What goes wrong:** If `OuterMessage` is listed before `Message` in the
coalesce and an AppTraces row somehow had a non-null `OuterMessage`
(shouldn't happen, but let's be defensive), the behavior flips from what
existing consumers expect.

**Why it happens:** `AppTraces` has a `Message` column but no `OuterMessage`
column. In the `union` result, `OuterMessage` is always null on AppTraces
rows, so the coalesce correctly falls through to `Message`. But if a future
schema change or a custom `logger.info(..., extra={"OuterMessage": ...})`
populates the column, behavior changes.

**How to avoid:** Add a regression test: mock `execute_kql` to return a row
with `ItemType="Log"`, `Message="log line"`, `OuterMessage=None` — assert
that the final `FailureRecord.message == "log line"`. This pins the
AppTraces path.

**Warning signs:** One or more severity-pinning tests still pass but
trace_lifecycle output against live data shows different `message` strings
for routine `logger.error` lines than before.

### Pitfall 4: Auto-format hook strips new fields before they're used

**What goes wrong:** Per `MEMORY.md` — ruff's auto-format PostToolUse hook
removes unused imports/symbols immediately after each Edit. If a plan adds
`outer_message: str | None = None` to `FailureRecord` in one Edit and the
parser that populates it in a separate Edit, the intermediate state has
field-without-usage, which ruff does not strip (fields aren't imports), BUT
adding a new import to `queries.py` to reference a new type WILL be stripped
if usage hasn't landed yet.

**Why it happens:** Hook runs after every write; it doesn't know about
sequences of edits.

**How to avoid:** Either (a) put the model-field addition and the parser
update in a single multi-Edit `Write` that replaces the whole file, or
(b) land the parser update LAST in a sequence after the model already exists.

**Warning signs:** Subsequent Edit failing because an import was silently
removed; `AttributeError` at test run time on a field that was "just added".

### Pitfall 5: Pydantic `model_dump()` emits None values

**What goes wrong:** MCP tool returns look like
`{"message": "...", "outer_message": null, "outer_type": null, ...}` for every
AppTraces row, inflating the response payload and making the JSON noisier for
Claude Code.

**Why it happens:** `model_dump()` by default includes all fields regardless
of value.

**How to avoid:** Use `model_dump(exclude_none=True)` in `mcp/server.py` tool
returns. This keeps the response compact while preserving full info on
AppExceptions rows that have the new fields populated.

**Warning signs:** Claude Code's investigation response tokens/time inflate
for routine trace_lifecycle calls on successful captures (which are
AppTraces-dominant).

## Open Questions

1. **Should `result_code` remain in the locked field list despite not existing on `AppExceptions`?**
   - What we know: The official Microsoft Learn schema (updated 2026-03-11) lists
     no `ResultCode` column on `AppExceptions`. The CONTEXT.md "locked"
     decision includes it.
   - What's unclear: Is the intent (a) a typed int column (impossible without
     regex extraction), (b) a regex-extracted string synthesized in KQL, or
     (c) a misremembered detail that should be dropped?
   - Recommendation: Drop `result_code` from the model and projection entirely
     for Phase 19.1. The 3-digit status code is already visible inside
     `outer_message` (e.g., `"Operation returned an invalid status '429'"`) so
     Claude Code can read it as prose. If a future phase requires
     programmatic access to the status int, add a KQL
     `extract(@"'(\d{3})'", 1, OuterMessage)` in that phase. This is a
     deviation from the letter of CONTEXT.md — planner should flag it
     explicitly in the plan.

2. **Should `Details` be tostring'd in KQL or projected dynamic and serialized in Python?**
   - What we know: `Details` is `dynamic`. Azure SDK returns dynamics as Python
     objects (list/dict) when the column type is dynamic.
   - What's unclear: Which path Pydantic handles more cleanly — string-typed
     field with KQL `tostring()` upstream, or `Any`/`list[dict]`-typed field
     with no KQL wrapper.
   - Recommendation: `tostring(Details)` in KQL, `details: str | None = None`
     in the model. Simpler, JSON-safe over the wire, aligned with the
     truncation strategy (string slicing is trivial; structured truncation
     requires walking a list).

3. **Should `TraceRecord` get the same new fields as `FailureRecord`?**
   - What we know: `CAPTURE_TRACE` is in the "all three must be updated" locked
     list, so `TraceRecord` needs the fields to represent what's projected.
   - What's unclear: `CAPTURE_TRACE` also unions `AppRequests` and
     `AppDependencies`, which have `Name` but no `OuterMessage`. The
     additive-optional pattern handles that cleanly.
   - Recommendation: Yes, mirror the new fields on `TraceRecord`. The
     `_empty_to_none` validator should cover them.

4. **Does `exclude_none=True` on `model_dump()` break any existing Claude Code prompt or test?**
   - What we know: Current `model_dump()` calls don't pass `exclude_none`, so
     serialized output always includes `None` for absent fields.
   - What's unclear: Whether any downstream consumer (investigation agent prompt,
     skill, or test assertion) depends on `None` being emitted.
   - Recommendation: Plan a Wave-0-style "find all call sites of `model_dump()`
     in `mcp/server.py` + `backend/src/second_brain/observability/`" step
     before adopting `exclude_none=True` as the new default, OR leave existing
     `model_dump()` calls as-is and only add `exclude_none` for new error-detail
     fields via a small dict-filter helper. Either is fine; prefer the
     dict-filter helper for minimum blast radius.
