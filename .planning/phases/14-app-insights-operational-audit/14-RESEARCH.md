**Researched:** 2026-03-22
**Domain:** Azure Application Insights, OpenTelemetry Python, KQL, Azure Monitor Alerts, Mobile Telemetry
**Confidence:** HIGH

## Summary

This phase audits and streamlines the full observability stack for the Second Brain system. The backend already has a solid foundation: `azure-monitor-opentelemetry` v1.8.7 is installed, `configure_azure_monitor()` is called at startup, `enable_instrumentation()` tracks agent-framework metrics, and OTel spans with custom attributes exist for captures, tool calls, and admin processing. However, several gaps exist: no per-capture trace ID propagated from mobile through backend, no structured log format convention, inconsistent log levels, no operational KQL queries, no alerts, and the mobile app has zero observability integration.

The phase requires work across five areas: (1) per-capture trace ID propagation from mobile to App Insights, (2) log level audit and structured logging convention, (3) version-controlled KQL query files, (4) Azure Monitor alert rules with action groups, and (5) mobile client-side telemetry. The App Insights React Native plugin exists but explicitly does NOT support Expo. A lightweight alternative approach using the backend as a telemetry proxy is recommended for mobile.

**Primary recommendation:** Use the existing `azure-monitor-opentelemetry` distro (already installed). Add a `trace_id` field generated by the mobile app, propagated as a header and threaded through all backend logging via `extra={}` custom dimensions. Store KQL queries as `.kql` files in the repo. Configure alerts via Azure CLI. For mobile telemetry, send client-side errors to a new backend endpoint that logs them to App Insights (avoiding the Expo-incompatible React Native plugin).

### Pattern 1: Per-Capture Trace ID Propagation
**What:** Mobile generates a UUID per capture, sends as `X-Trace-Id` header. Backend extracts it, sets it as an OTel span attribute AND passes it in `extra={}` on every logger call for that capture's lifecycle.
**When to use:** Every capture path (text, voice, follow-up).
**Example:**
```python
# In capture endpoint handler
capture_trace_id = request.headers.get("X-Trace-Id", str(uuid4()))

# Pass to streaming adapter
async def stream_text_capture(..., capture_trace_id: str = "") -> AsyncGenerator:
    with tracer.start_as_current_span("capture_text") as span:
        span.set_attribute("capture.trace_id", capture_trace_id)
        # ... all logger calls include:
        logger.info("Filed to %s", bucket, extra={"capture_trace_id": capture_trace_id})
```
Source: Azure Monitor OpenTelemetry docs - custom properties via `extra={}` appear as `customDimensions` in `AppTraces`.

### Pattern 2: Structured Logging with Custom Dimensions
**What:** Use Python `logging.info(..., extra={"key": "value"})` to attach structured data. The `azure-monitor-opentelemetry` distro automatically sends `extra` dict entries as `customDimensions` in App Insights `AppTraces` table.
**When to use:** Every log statement that relates to a specific capture, admin processing, or tool invocation.
**Example:**
```python
logger.info(
    "Classification complete",
    extra={
        "capture_trace_id": trace_id,
        "component": "classifier",
        "bucket": "Admin",
        "confidence": 0.92,
    },
)
```
Source: https://learn.microsoft.com/azure/azure-monitor/app/opentelemetry-add-modify#modify-telemetry

### Pattern 3: Logger Scoping with logger_name Parameter
**What:** `configure_azure_monitor(logger_name="second_brain")` scopes log collection to loggers under the `second_brain` namespace. This prevents internal Azure SDK logging from creating noise while ensuring all application logs are captured.
**When to use:** At startup in `main.py`.
**Why:** Currently `configure_azure_monitor()` is called without `logger_name`, which means it attaches to the root logger. This captures ALL Python logging including azure-core, azure-cosmos, and other SDK internal logs. Setting `logger_name="second_brain"` scopes collection to only the app's loggers. The existing codebase already uses `logging.getLogger(__name__)` everywhere (e.g., `second_brain.api.capture`, `second_brain.streaming.adapter`), which all fall under the `second_brain` namespace.
Source: https://learn.microsoft.com/python/api/azure-monitor-opentelemetry/azure.monitor.opentelemetry

### Pattern 4: Admin Agent Trace ID Continuity
**What:** When a capture is classified as Admin and triggers background processing, the `capture_trace_id` must be stored on the inbox document and carried into the `process_admin_capture` coroutine. This ensures the full lifecycle (capture -> classify -> admin process -> errand write) is traceable by a single ID.
**When to use:** In `admin_handoff.py` and the errands endpoint processing trigger.
**Example:**
```python
# Store trace_id on inbox document at classification time (in file_capture tool)
doc_body["captureTraceId"] = capture_trace_id

# In process_admin_capture, read it from the inbox document
trace_id = doc.get("captureTraceId", "unknown")
logger.info("Processing admin capture", extra={"capture_trace_id": trace_id})
```

### Pattern 5: Mobile Telemetry via Backend Proxy
**What:** Instead of the Expo-incompatible React Native App Insights plugin, create a lightweight `POST /api/telemetry` endpoint that accepts client-side error/performance events and logs them to App Insights via the backend's existing OTel pipeline.
**When to use:** For mobile errors, network failures, and performance metrics.
**Why:** The `@microsoft/applicationinsights-react-native` plugin explicitly states "It doesn't work with apps using the Expo framework." A backend proxy avoids this limitation entirely.
**Example:**
```python
@router.post("/api/telemetry", status_code=204)
async def report_client_telemetry(request: Request, body: TelemetryEvent):
    logger.warning(
        "Client error: %s",
        body.message,
        extra={
            "component": "mobile",
            "event_type": body.event_type,
            "capture_trace_id": body.capture_trace_id,
            "device_info": body.device_info,
        },
    )
```

### Anti-Patterns to Avoid
- **Attaching OTel handler to root logger:** Causes recursive logging from azure-core. Use `logger_name="second_brain"` parameter.
- **Using `print()` for debugging:** Already avoided in codebase. Python `logging` module is used consistently.
- **Sampling in hobby-scale app:** Will explicitly decided full fidelity, no sampling. Volume is too low to justify losing data.
- **Debug logs in App Insights without explicit opt-in:** `logger.debug()` calls (warmup, middleware) should NOT be exported to App Insights. Only `info`, `warning`, `error`, `critical` should be exported.

## Common Pitfalls

### Pitfall 1: logger.info Not Appearing in App Insights
**What goes wrong:** Will has already discovered this -- `logger.info` traces don't appear in App Insights queries.
**Why it happens:** The MEMORY.md notes: "App Insights only exports `logger.warning` and above -- `logger.info` traces don't appear in queries." This is likely due to the root logger's level being set to WARNING by default, or the OTel log handler only capturing WARNING+.
**How to avoid:** Set the Python root logger level to INFO explicitly before `configure_azure_monitor()`:
```python
import logging
logging.basicConfig(level=logging.INFO)
configure_azure_monitor(logger_name="second_brain")
```
Or configure the specific logger:
```python
logging.getLogger("second_brain").setLevel(logging.INFO)
```
**Warning signs:** Queries against `AppTraces` only show `SeverityLevel >= 2` (Warning).

### Pitfall 2: Background Task Logs Losing OTel Context
**What goes wrong:** `asyncio.create_task()` creates a new task that may not inherit the parent span context. Logs from `process_admin_capture` may not correlate with the original request.
**Why it happens:** OTel context is thread-local/task-local. When a new task is created, the context may not propagate automatically.
**How to avoid:** Explicitly pass the `capture_trace_id` as a parameter to `process_admin_capture` (already partially done via `inbox_item_id`). The trace ID should be an explicit parameter, not rely on OTel context propagation.
**Warning signs:** Admin Agent logs appearing with no `operation_Id` correlation to the original capture request.

### Pitfall 3: Azure SDK Internal Logs Flooding App Insights
**What goes wrong:** Without `logger_name` scoping, `azure-core`, `azure-cosmos`, and `azure-identity` internal logs all get exported, creating noise.
**Why it happens:** `configure_azure_monitor()` without `logger_name` attaches to the root logger, capturing everything.
**How to avoid:** Use `configure_azure_monitor(logger_name="second_brain")`.
**Warning signs:** AppTraces full of azure.core.pipeline, azure.cosmos, azure.identity internal messages.

### Pitfall 4: Expo + React Native App Insights Plugin Incompatibility
**What goes wrong:** Installing `@microsoft/applicationinsights-react-native` in an Expo project fails or crashes at runtime.
**Why it happens:** The plugin depends on `react-native-device-info` which requires native linking. Expo managed workflow does not support native modules that aren't in the Expo SDK.
**How to avoid:** Do NOT install the React Native App Insights plugin. Use the backend telemetry proxy pattern instead.
**Warning signs:** Build failures, "native module not found" errors.

### Pitfall 5: Alert Fatigue from Transient Errors
**What goes wrong:** Alerts fire on every 5xx response or transient Azure SDK timeout, creating notification spam.
**Why it happens:** Alert rules configured with evaluation period too short or threshold too sensitive.
**How to avoid:** Use sustained evaluation windows (e.g., "greater than 3 failures in a 5-minute window, evaluated every 5 minutes"). Configure auto-resolve when condition clears.
**Warning signs:** Multiple alerts per day for issues that self-resolve.

### Log Level Policy (Proposed)

Based on auditing the current codebase:

| Level | Use For | Current Issues |
|-------|---------|----------------|
| `ERROR` | Failures that break functionality (agent errors, Cosmos write failures, stream errors) | Some errors logged as `warning` (e.g., admin agent processing failures in `admin_handoff.py` line 276) |
| `WARNING` | Degraded but recoverable situations (fallbacks, retries, optional features unavailable) | Appropriately used for Key Vault fallback, Playwright unavailable, etc. |
| `INFO` | Significant lifecycle events (capture filed, admin processed, resource initialized) | Too many INFO logs for routine operations (e.g., "Tasks: 5 items" on every GET) |
| `DEBUG` | Diagnostic detail not needed in production queries | Correctly used for warmup pings, middleware timing |

### KQL Query: What Happened to Capture X?
```kql
// Trace a specific capture by its trace ID through full lifecycle
let trace_id = "PASTE_TRACE_ID_HERE";
union AppTraces, AppDependencies, AppRequests, AppExceptions
| where customDimensions.capture_trace_id == trace_id
    or customDimensions["capture_trace_id"] == trace_id
| project
    TimeGenerated,
    ItemType = case(
        itemType == "trace", "Log",
        itemType == "dependency", "Dependency",
        itemType == "request", "Request",
        itemType == "exception", "Exception",
        itemType
    ),
    Message = coalesce(Message, Name, ExceptionType),
    SeverityLevel,
    Component = tostring(customDimensions.component),
    Details = customDimensions
| order by TimeGenerated asc
```

### KQL Query: What Failed Recently?
```kql
// Recent errors and failures in the last 24 hours
union AppTraces, AppExceptions
| where TimeGenerated > ago(24h)
| where SeverityLevel >= 3  // Error and above
    or itemType == "exception"
| project
    TimeGenerated,
    Type = itemType,
    Message = coalesce(Message, ExceptionType),
    Component = tostring(customDimensions.component),
    TraceId = tostring(customDimensions.capture_trace_id),
    Details = customDimensions
| order by TimeGenerated desc
| take 50
```
