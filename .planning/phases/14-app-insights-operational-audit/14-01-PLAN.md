
    Apply this log level policy consistently:
    - **ERROR**: Unrecoverable failures that break functionality (agent errors, Cosmos write failures, stream errors, processing failures)
    - **WARNING**: Degraded but recoverable situations (fallbacks, retries, optional features unavailable, non-fatal errors)
    - **INFO**: Significant lifecycle events (capture filed, admin processed, resource initialized, agent registered) -- events worth seeing in operational queries
    - **DEBUG**: Routine/noisy operations that fire frequently (GET response counts, warmup pings, middleware timing, per-poll logging)

    Specific fixes identified in RESEARCH.md:

    4. `inbox.py`: "Inbox list: returned %d items" logged as `info` -- demote to `debug` (fires on every screen load)

    Walk through ALL 18 source files with logger calls and audit each one against the policy. Fix any other inconsistencies found. The research identified these 4 but there may be others.

    Do NOT demote startup/initialization INFO logs -- those are genuine lifecycle events that should remain INFO.

    ```python
    capture_trace_id = request.headers.get("X-Trace-Id", str(uuid4()))
    ```

    Pass `capture_trace_id` as a new keyword argument to the corresponding stream function. Update ALL logger calls in capture.py to include `extra={"capture_trace_id": capture_trace_id}`.

    **Step 2: Thread capture_trace_id through adapter functions (adapter.py)**

    Add `capture_trace_id: str = ""` parameter to `stream_text_capture`, `stream_voice_capture`, and `stream_follow_up_capture`.

    Inside each function:
    1. Set it as an OTel span attribute: `span.set_attribute("capture.trace_id", capture_trace_id)`
    2. Include it in ALL `logger.*()` calls via `extra={"capture_trace_id": capture_trace_id}`
    3. Include it in ALL `reasoning_logger.*()` calls via `extra={"capture_trace_id": capture_trace_id}`

    When the adapter creates admin handoff tasks (fire-and-forget via `asyncio.create_task`), pass `capture_trace_id` to the handoff function.

    **Step 3: Store captureTraceId on inbox documents (classification.py)**

    In `ClassifierTools.file_capture`, accept an optional `capture_trace_id` parameter. When building the inbox document body dict, include `"captureTraceId": capture_trace_id`. This field persists the trace ID so background admin processing can retrieve it later.

    To pass the trace ID from the adapter to file_capture (which is an @tool function called by the Foundry agent), use the existing ContextVar pattern from `follow_up_context`:
    - Create a new ContextVar: `capture_trace_id_var: ContextVar[str] = ContextVar("capture_trace_id_var", default="")`
    - Set it before the agent call in the adapter: `token = capture_trace_id_var.set(capture_trace_id)`
    - Read it inside `file_capture`: `trace_id = capture_trace_id_var.get()`
    - Store on document: `doc_body["captureTraceId"] = trace_id`
    - Reset after agent call: `capture_trace_id_var.reset(token)`

    Export `capture_trace_id_var` from classification.py so adapter.py can import and set it.

    **Step 4: Carry captureTraceId through admin processing (admin_handoff.py)**

    Add `capture_trace_id: str = ""` parameter to `process_admin_capture` and `process_admin_captures_batch`.

    In `process_admin_capture`:
    1. Read `captureTraceId` from the inbox document: `trace_id = doc.get("captureTraceId", capture_trace_id or "unknown")`
    2. Set as OTel span attribute: `span.set_attribute("capture.trace_id", trace_id)`
    3. Include in ALL logger calls: `extra={"capture_trace_id": trace_id}`

    In the adapter, update the `asyncio.create_task` calls for admin handoff to pass `capture_trace_id`.

    **Step 5: Add structured logging extra fields**

    Beyond `capture_trace_id`, add a `component` field to logger calls in key files to enable component-level filtering in KQL:
    - capture.py: `extra={"capture_trace_id": ..., "component": "capture"}`
    - adapter.py: `extra={"capture_trace_id": ..., "component": "classifier"}` for classification, `"component": "admin_handoff"` for handoff
    - admin_handoff.py: `extra={"capture_trace_id": ..., "component": "admin_agent"}`
    - classification.py: `extra={"capture_trace_id": ..., "component": "classifier"}`

    Not every logger call in every file needs component -- focus on the capture lifecycle path. Startup/lifespan logs in main.py do NOT need capture_trace_id (there is no capture context at startup).
