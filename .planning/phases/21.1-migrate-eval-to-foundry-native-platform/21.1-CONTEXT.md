# Phase 21.1: Migrate Eval to Foundry Native Platform - Context

**Gathered:** 2026-04-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the custom eval framework (runner, dry-run tools, metrics, API endpoint) built in Phase 21 with Azure AI Foundry's native evaluation platform. The system's guiding principle is to leverage Foundry/Azure built-in capabilities before building custom. Phase 21's research incorrectly concluded Foundry lacked relevant evaluators; this phase corrects that by migrating scoring, run records, tracing, and result storage to Foundry-native evaluation.

After this phase, eval runs are triggered via investigation tools calling the Foundry SDK directly, results are stored in Foundry (not Cosmos EvalResults), and the Foundry portal Tracing/Evaluation pages show real data. Direct Foundry agent-target execution is allowed only after a canary proves that the target agent exposes the same tool behavior as production. If the canary fails, eval execution must use the app-mediated path to invoke agents with local Python tools, then upload generated response artifacts to Foundry for native scoring and result storage.

</domain>

<decisions>
## Implementation Decisions

### Evaluator strategy
- **D-01:** Two separate custom code-based evaluators registered in Foundry: one for classifier bucket accuracy (exact-match), one for admin agent routing accuracy (destination exact-match). They fulfill different functions and performance must be measured separately.
- **D-02:** Also use Foundry's built-in evaluators alongside custom ones: `IntentResolutionEvaluator` for classifier intent quality, `ToolCallAccuracyEvaluator` for tool usage correctness, `TaskAdherenceEvaluator` for admin agent instruction compliance.
- **D-03:** Custom evaluators are code-based (Python `grade()` function), not prompt-based LLM judges. Deterministic exact-match scoring, consistent with Phase 21 CONTEXT.md decision on deterministic metrics.

### Eval execution model
- **D-04:** Prefer Foundry's agent target evaluation (`client.evals.runs.create()` with `azure_ai_target_completions` data source) only after a canary proves the deployed target agent can invoke/capture the same tools used by production (`file_capture`, `add_errand_items`, `add_task_items`, and admin routing context). If the canary fails, use app-mediated response generation with the existing production clients/tools, then run Foundry dataset evaluation over JSONL rows that include query, expected labels, response text, and captured tool-call arguments.
- **D-05:** Investigation tools (`run_classifier_eval`, `run_admin_eval`, `get_eval_results`) call the Foundry SDK directly. No HTTP roundtrip through a backend API endpoint.
- **D-06:** Delete the eval API endpoint (`/api/eval/run`, `/api/eval/status`, `/api/eval/results`) entirely. Investigation tools are the sole trigger.

### Code cleanup
- **D-07:** Delete the custom eval API endpoint and metric-only code after the canary/fallback execution mode is implemented. `eval/runner.py` and dry-run concepts may be replaced by a Foundry integration module, but do not remove the only app-mediated invocation path until Foundry target eval has been proven to capture production-equivalent tool calls or a fallback app-mediated generator exists.
- **D-08:** Keep `eval/__init__.py` as package marker if Foundry SDK integration code lives in `eval/`. Otherwise delete the entire `eval/` package.
- **D-09:** Keep `GoldenDatasetDocument` model in `documents.py` and `seed_golden_dataset.py` script — the golden dataset in Cosmos is still the source of truth for test cases, exported to JSONL for Foundry upload.
- **D-10:** `EvalResultsDocument` model can be deleted — results live in Foundry, not Cosmos EvalResults container.

### Investigation tool response format
- **D-11:** `get_eval_results` output format stays the same (accuracy percentage, per-bucket breakdown, failures listed). Only the data source changes from Cosmos to Foundry's `client.evals.runs.output_items.list()`.
- **D-14:** `get_eval_results` must be restart-safe. Do not rely only on an in-memory run dict. Use stable eval names/run-name prefixes and/or explicit `eval_id` + `run_id` lookup so completed Foundry runs remain discoverable after process restart.

### Foundry resource lifecycle
- **D-15:** Custom evaluator registration must be idempotent. Ensure evaluator versions exist before creating eval definitions; reuse existing versions when names and code hashes match, create a new version when the code changes.
- **D-16:** Golden dataset uploads must use unique, deterministic versions (content hash or timestamp + content hash). Do not upload every run as `version="1"`.
- **D-17:** Stable eval names are required: `second-brain-classifier-eval` and `second-brain-admin-eval`. Stable run name prefixes are required: `classifier-YYYYMMDD-HHMMSS-<dataset_hash>` and `admin-YYYYMMDD-HHMMSS-<dataset_hash>`.
- **D-18:** Admin eval must preserve deterministic routing context. Include the same routing context produced by `build_routing_context()` in the admin eval input unless the canary proves the target agent can call `get_routing_context` during Foundry target evaluation.

### Rollout gating
- **D-19:** Treat Foundry cloud evaluation and custom evaluator behavior as preview-risk features even though the `azure-ai-projects` SDK itself is stable. The old custom eval path may be deleted only after automated tests pass and a live canary eval proves the selected execution mode produces non-empty scoreable outputs.

### Tracing integration
- **D-12:** Connect App Insights (`second-brain-insights`) to the Foundry project so the Tracing page populates with real agent run data. Leverage as much Foundry native capability as possible.
- **D-13:** Enable `AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true` on the Container App so Input/Output columns populate in Foundry Tracing.

### Claude's Discretion
- Foundry SDK client initialization pattern (project client setup, credential handling)
- JSONL export format for golden dataset upload
- How to reference the classifier agent by name+version (may need agent version registration)
- Whether to use `AIAgentConverter` or direct agent target evaluation

</decisions>

<specifics>
## Specific Ideas

- "The purpose of this program was to leverage what was in Azure Services or Foundry AI before creating something new" — this is the core principle driving the migration
- Current Foundry evaluation SDK capabilities exist and are suitable for migration, but Microsoft docs still mark cloud evaluation/custom evaluators as preview. Keep a canary gate before deleting the old eval path.
- The Foundry portal Tracing page shows an "Evaluation metrics" column that's currently empty — after this phase it should show real scores

</specifics>

- Investigation tools return `json.dumps(result)` strings — maintain this for Foundry eval results
- Optional client params with None defaults and early-return error handling
- `asyncio.create_task` for background work with GC prevention via `app.state.background_tasks`

### Integration Points
- `main.py` lifespan: add Foundry project client initialization alongside existing clients
- `main.py` InvestigationTools: pass project client, remove classifier_client/admin_client if no longer needed for direct eval calls
