**Core function pattern** (runner.py lines 82-100):
The existing runner defines top-level async functions that accept injected dependencies (no class). New `foundry.py` should follow the same pattern: module-level async functions, dependencies passed as arguments, return dicts.
```python
async def run_classifier_eval(
    run_id: str,
    cosmos_manager: CosmosManager,
    classifier_client: AzureAIAgentClient,
    runs_dict: dict,
) -> None:
    """Run classifier evaluation against golden dataset."""
    try:
        runs_dict[run_id] = {"status": "running", "progress": "0/0"}
        # ... orchestration logic ...
    except Exception as exc:
        logger.error(
            "Classifier eval failed: %s",
            exc,
            extra={
                "component": "eval",
                "eval_type": "classifier",
                "eval_run_id": run_id,
            },
        )
        runs_dict[run_id] = {"status": "failed", "error": str(exc)}
```

**Logging pattern** (runner.py lines 191-202):
```python
logger.info(
    "Classifier eval complete: accuracy=%.2f, total=%d, correct=%d",
    metrics["accuracy"],
    metrics["total"],
    metrics["correct"],
    extra={
        "component": "eval",
        "eval_type": "classifier",

```

**Error handling pattern** (runner.py lines 213-223):
Top-level try/except wraps entire function body. Sets `runs_dict[run_id]` to `{"status": "failed", "error": str(exc)}`. Per-case exceptions caught inside loop, marked as ERROR but don't abort run.

**Key difference from analog:** The new file will call Foundry SDK (`openai_client.evals.create()`, `openai_client.evals.runs.create()`, polling loop, `openai_client.evals.runs.output_items.list()`) for scoring/run records instead of computing metrics locally. Direct agent invocation moves to Foundry only after a canary proves production-equivalent tool capture. If the canary fails, the new file keeps a thin app-mediated artifact generation path using production clients/tools, then uploads JSONL artifacts for Foundry scoring. Sync SDK calls must be wrapped in `asyncio.to_thread()` per RESEARCH Pitfall 1.

### JSON Return Contract (Investigation Tools)
**Source:** `backend/src/second_brain/tools/investigation.py` (all tools)
**Apply to:** All modified investigation tool methods
Every tool returns `json.dumps(result_dict, default=str)` -- never raises, always returns a JSON string. Error responses use `{"error": "message"}` key.

### Async Background Task Pattern
**Source:** `backend/src/second_brain/tools/investigation.py` lines 632-639
**Apply to:** `run_classifier_eval` and `run_admin_eval` in investigation tools
```python
asyncio.create_task(
    _run_classifier_eval(
        run_id=run_id,
        cosmos_manager=self._cosmos_manager,
        classifier_client=self._classifier_client,
        runs_dict=_eval_runs,
    )
)
```
Note: For the new Foundry SDK flow, wrapping sync calls in `asyncio.to_thread()` inside the background task is critical (RESEARCH Pitfall 1). Do not treat the in-memory task/run map as durable state; Foundry run listing/retrieval is the source of truth.

### Eval Execution Mode Canary
**Source:** `backend/src/second_brain/streaming/adapter.py` passes Classifier tools at call time; `backend/src/second_brain/processing/admin_handoff.py` prepends routing context and passes Admin tools at call time.
**Apply to:** `eval/foundry.py`, `tools/investigation.py`
Direct Foundry target evaluation must prove that `sample.output_items` includes production-equivalent tool calls before the app-mediated invocation path is removed. If canary output lacks `file_capture`, `add_errand_items`, or `add_task_items`, use app-mediated artifact generation and Foundry dataset scoring.

### Dataset Versioning
**Apply to:** `eval/foundry.py`, `seed_golden_dataset.py`
Dataset uploads use a normalized JSONL content hash plus timestamp/version string. Never reuse fixed `version="1"` for repeated uploads.

### Logging with Component Extra
**Source:** `backend/src/second_brain/tools/investigation.py` line 131, runner.py line 196
**Apply to:** All new eval functions in `foundry.py`
