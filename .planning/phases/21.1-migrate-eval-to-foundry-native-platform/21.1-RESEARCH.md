# Phase 21.1: Migrate Eval to Foundry Native Platform - Research

**Researched:** 2026-04-24
**Domain:** Azure AI Foundry Evaluation SDK, Custom Evaluators, Agent Target Evaluation
**Confidence:** MEDIUM-HIGH

## Summary

This phase replaces the custom eval scoring/result-storage framework built in Phase 21 with Azure AI Foundry's native evaluation platform. The Foundry SDK (`azure-ai-projects >= 2.0.0`) provides a comprehensive evaluation API accessed via `AIProjectClient.get_openai_client()`, which exposes `client.evals.create()`, `client.evals.runs.create()`, and `client.evals.runs.output_items.list()`. Custom code-based evaluators are registered via `project_client.beta.evaluators.create_version()` with a Python `grade(sample, item) -> float` function. Built-in agent evaluators (Intent Resolution, Tool Call Accuracy, Task Adherence) are available as `builtin.*` evaluators in the testing criteria.

The migration has one critical validation gate: the current production agents are created as Foundry shells, while Python tools are supplied at request time through `ChatOptions`. Direct Foundry agent-target evaluation invokes an agent by name/version. Before deleting the old eval path, a canary must prove that target evaluation can invoke and capture the same production tool behavior. If not, the migration should still use Foundry for evaluator catalog, scoring, run records, and portal results, but generate response artifacts through the app-mediated production clients/tools and run Foundry dataset evaluation over those artifacts.

**Primary recommendation:** Use `AIProjectClient` + `get_openai_client()` for all eval operations. Register two custom code-based evaluators (classifier bucket accuracy, admin routing accuracy) idempotently in the Foundry evaluator catalog. Run a live canary using `azure_ai_target_completions` with `azure_ai_agent`; use direct agent-target evaluation only if output items contain the expected production tool calls. Otherwise, use app-mediated response generation and Foundry dataset evaluation. Poll for completion, retrieve results via `output_items.list()`, and format them to match the existing investigation tool output contract.

### Pattern 2: Custom Code-Based Evaluator Registration

**What:** Register custom evaluators in the Foundry evaluator catalog.
**When to use:** One-time setup (or check-and-create-if-missing at startup).

```python
# Source: https://learn.microsoft.com/en-us/azure/foundry/concepts/evaluation-evaluators/custom-evaluators
from azure.ai.projects.models import EvaluatorCategory, EvaluatorDefinitionType

CLASSIFIER_GRADE_FN = '''
def grade(sample: dict, item: dict) -> float:
    """Exact-match scoring for classifier bucket accuracy."""
    expected = item.get("expected_bucket", "")
    # For agent target evaluation, response is in sample.output_text
    predicted = item.get("sample", {}).get("output_text", "")
    # Parse the bucket from the agent's response text
    # (exact logic depends on how agent formats classification)
    return 1.0 if expected.lower() in predicted.lower() else 0.0
'''

evaluator = project_client.beta.evaluators.create_version(
    name="classifier_bucket_accuracy",
    evaluator_version={
        "name": "classifier_bucket_accuracy",
        "categories": [EvaluatorCategory.QUALITY],
        "display_name": "Classifier Bucket Accuracy",
        "description": "Exact-match scoring for classifier bucket prediction",
        "definition": {
            "type": EvaluatorDefinitionType.CODE,
            "code_text": CLASSIFIER_GRADE_FN,
            "init_parameters": {
                "type": "object",
                "properties": {
                    "deployment_name": {"type": "string"},
                    "pass_threshold": {"type": "number"},
                },
                "required": ["deployment_name", "pass_threshold"],
            },
            "metrics": {
                "result": {
                    "type": "continuous",
                    "desirable_direction": "increase",
                    "min_value": 0.0,
                    "max_value": 1.0,
                }
            },
            "data_schema": {
                "type": "object",
                "required": ["item"],
                "properties": {
                    "item": {
                        "type": "object",
                        "properties": {
                            "expected_bucket": {"type": "string"},
                        },
                    },
                },
            },
        },
    },
)
```

**Critical detail:** Code-based evaluators run in a sandboxed Python environment with NO network access, 2-minute timeout per grading call, and limited packages (numpy, pandas, scikit-learn, rapidfuzz, pydantic, etc.). The `grade()` function must be self-contained.

### Pattern 4: Polling and Result Retrieval

**What:** Poll eval run to completion, then retrieve per-item results.
**When to use:** After creating an eval run, in the investigation tool handler.

```python
# Source: https://learn.microsoft.com/en-us/azure/foundry/how-to/develop/cloud-evaluation
import asyncio

# Poll (sync API -- may need to run in executor for async context)
while True:
    run = openai_client.evals.runs.retrieve(
        run_id=eval_run.id, eval_id=eval_object.id
    )
    if run.status in ("completed", "failed"):
        break
    await asyncio.sleep(5)

# Get per-item results
output_items = list(
    openai_client.evals.runs.output_items.list(
        run_id=run.id, eval_id=eval_object.id
    )
)

# Each output_item has:
# - output_item.datasource_item  (original query + agent response)
# - output_item.results          (list of evaluator scores)
# - output_item.status           ("completed" or "failed")
```

### Anti-Patterns to Avoid

- **Running azure-ai-evaluation locally:** The `azure-ai-evaluation` package (1.16.5) is for local evaluation. D-04 mandates using Foundry's cloud evaluation via `azure_ai_target_completions`. Do NOT install `azure-ai-evaluation`.
- **Keeping eval HTTP endpoints:** D-06 mandates deleting `/api/eval/run`, `/api/eval/status`, `/api/eval/results`. Investigation tools are the sole trigger.
- **Storing results in Cosmos:** D-10 mandates results live in Foundry. Do NOT create EvalResults documents.
- **Using sync OpenAI client in async context:** The `openai_client.evals.*` methods are synchronous. Wrap in `asyncio.get_event_loop().run_in_executor(None, ...)` or use `asyncio.to_thread()` to avoid blocking the event loop.
- **Deleting the old runner before the canary passes:** Do not remove the only production-equivalent app-mediated invocation path until direct Foundry target eval is proven or the app-mediated artifact fallback exists.
- **Uploading every dataset as version `1`:** Use a content hash or timestamp+hash version so Foundry dataset versions remain traceable and rerunnable.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Eval run orchestration | Custom runner with retry logic | `client.evals.runs.create()` + polling | Foundry handles retries, rate limits, parallelism internally |
| Agent invocation during eval | Dry-run tools + `get_response()` per case as the long-term scoring engine | Canary-gated `azure_ai_target_completions`, or app-mediated artifact generation + Foundry dataset scoring | Foundry is preferred for invocation, but app-mediated execution is required if local tools are not exposed to direct target eval |
| Metric computation | Custom `compute_classifier_metrics()` | Custom code-based evaluator in Foundry catalog | Runs in Foundry's sandbox; results stored natively |
| Result storage | Cosmos EvalResults container | Foundry eval results (via `output_items.list()`) | Results visible in Foundry portal Evaluation tab |
| Run status tracking | In-memory `_eval_runs` dict as source of truth | Stable eval names/run prefixes + `client.evals.runs.retrieve()` | Foundry, not memory, tracks run lifecycle and remains queryable after restart |

**Key insight:** The custom metrics, results container, and API endpoints are replaced by Foundry SDK calls. Agent invocation can move fully to Foundry only after the canary proves production-equivalent tool capture; otherwise the backend keeps a thin artifact-generation layer while Foundry owns scoring and run records.

## Common Pitfalls

### Pitfall 1: Sync OpenAI Client in Async FastAPI Context
**What goes wrong:** The `openai_client.evals.*` methods appear to be synchronous. Calling them directly in an async investigation tool blocks the event loop.
**Why it happens:** `AIProjectClient.get_openai_client()` returns a sync `OpenAI` client, not `AsyncOpenAI`.
**How to avoid:** Wrap all eval SDK calls in `asyncio.to_thread()` or run them in a background task (the existing pattern uses `asyncio.create_task()` for eval runs). The polling loop is naturally suited to a background task.
**Warning signs:** Request timeouts, frozen SSE streams during eval execution.

### Pitfall 2: Custom Evaluator grade() Cannot Access Network
**What goes wrong:** Code-based evaluators run in a sandboxed environment with no network access. Any attempt to call external APIs (Cosmos, HTTP) will fail silently and return 0.0.
**Why it happens:** Foundry's sandbox restricts code-based evaluators for security.
**How to avoid:** The `grade()` function must be pure computation. All data must come from the `item` dict (dataset fields) and `sample` dict (agent response). For classifier accuracy, compare `item.expected_bucket` against the predicted bucket extracted from `sample.output_text` or `sample.output_items`.
**Warning signs:** All evaluator scores returning 0.0 with error status.

### Pitfall 3: Agent Name Must Match Exactly
**What goes wrong:** The `target.name` in `azure_ai_agent` must match the agent's registered name in Foundry exactly (case-sensitive).
**Why it happens:** Foundry looks up the agent by name, not by ID.
**How to avoid:** The classifier agent is named `"Classifier"` (from `ensure_classifier_agent()`, line 60). The admin agent is named `"AdminAgent"` (from `ensure_admin_agent()`, line 62). Use these exact names in the eval target config.
**Warning signs:** 404 errors from Foundry during eval run creation.

### Pitfall 4: Extracting Predicted Bucket from Agent Response
**What goes wrong:** The custom evaluator's `grade()` function needs to extract the predicted bucket from the agent's response, but the agent calls tools (file_capture) rather than returning the bucket in plain text.
**Why it happens:** The Classifier agent responds by calling the `file_capture` tool with a `bucket` parameter. The response text (`sample.output_text`) may be empty or just a confirmation message. The actual prediction is in the tool call arguments within `sample.output_items`.
**How to avoid:** In the `grade()` function, parse `item.get("sample", {}).get("output_items")` to find tool calls and extract the `bucket` argument. The `output_items` field contains the full structured response including tool calls per the Foundry docs.
**Warning signs:** Custom evaluator always returning 0.0 because it looks at `output_text` instead of tool call arguments.

### Pitfall 4b: Direct Target Eval Does Not See Local Python Tools
**What goes wrong:** Foundry target evaluation runs the deployed agent by name, but the app currently supplies Python tools at call time through `ChatOptions`. The target run may return plain text or no tool calls, making exact-match evaluators score everything as 0.0.
**Why it happens:** The deployed agent shell has name/model instructions, while production tool functions are bound by the app during `get_response()`.
**How to avoid:** Add a canary that runs one classifier and one admin example through direct target evaluation and asserts output items include the expected tool calls. If the canary fails, switch to app-mediated artifact generation and Foundry dataset scoring.
**Warning signs:** `sample.output_items` is empty, contains no `file_capture`/`add_*` calls, or built-in tool-call evaluators report missing tools.

### Pitfall 5: init_parameters Require deployment_name Even for Code-Based Evaluators
**What goes wrong:** Creating a code-based evaluator without `deployment_name` in `init_parameters` fails with a validation error.
**Why it happens:** The Foundry API schema requires `deployment_name` for all evaluator types, even code-based ones that don't use an LLM. This is a platform requirement, not a logical one.
**How to avoid:** Always include `"deployment_name"` and `"pass_threshold"` in `init_parameters` for code-based evaluators. Pass any valid model deployment name (e.g., `"gpt-4o"`).
**Warning signs:** 400 errors during evaluator creation or eval run creation.

### Pitfall 6: Dataset Schema Must Match Evaluator data_mapping
**What goes wrong:** Eval runs fail with schema errors when JSONL field names don't match the `data_mapping` references.
**Why it happens:** The `{{item.field}}` syntax in `data_mapping` must reference fields that exist in the JSONL dataset. Field names are case-sensitive.
**How to avoid:** Ensure JSONL fields exactly match what `data_mapping` references. For classifier eval: `query` and `expected_bucket`. For admin eval: `query` and `expected_destination`.
**Warning signs:** Schema validation errors during eval run creation.

### Pitfall 6b: Fixed Dataset Version Causes Conflicts or Stale Runs
**What goes wrong:** Uploading every JSONL as the same dataset name/version can fail or silently reuse unexpected data.
**Why it happens:** Foundry datasets are versioned project resources.
**How to avoid:** Compute a dataset content hash and upload as `YYYYMMDDHHMMSS-<hash>` or reuse an existing matching version. Include the dataset hash in the eval run name.
**Warning signs:** Upload conflicts, eval results that do not match the current golden dataset, or ambiguous run history.

### Pitfall 7: Async Credential for Sync AIProjectClient
**What goes wrong:** `AIProjectClient` may not accept `AsyncDefaultAzureCredential` directly since it wraps a sync OpenAI client.
**Why it happens:** The project uses `AsyncDefaultAzureCredential` everywhere for async context. `AIProjectClient` may need the sync `DefaultAzureCredential`.
**How to avoid:** Create a separate sync `DefaultAzureCredential` for `AIProjectClient`, or test whether `AsyncDefaultAzureCredential` works. If it doesn't, create the sync credential in the background task that runs the eval.
**Warning signs:** Auth errors when calling `project_client.get_openai_client()`.

### Custom Evaluator grade() Function: Classifier Bucket Accuracy

```python
# Source: Synthesized from custom-evaluators docs + project tool call format
# This runs inside Foundry's sandbox -- NO network, NO imports beyond allowed list
import json

def grade(sample: dict, item: dict) -> float:
    """Exact-match scoring for classifier bucket accuracy.
    
    Extracts the predicted bucket from the agent's tool call arguments
    (file_capture tool, 'bucket' parameter) and compares to expected.
    """
    expected = item.get("expected_bucket", "").strip().lower()
    if not expected:
        return 0.0
    
    # Agent target evaluation puts full response in item.sample.output_items
    output_items = item.get("sample", {}).get("output_items", [])
    if isinstance(output_items, str):
        try:
            output_items = json.loads(output_items)
        except (json.JSONDecodeError, TypeError):
            return 0.0
    
    # Search for file_capture tool call in the response
    predicted = ""
    if isinstance(output_items, list):
        for msg in output_items:
            content = msg.get("content", [])
            if isinstance(content, list):
                for part in content:
                    if part.get("type") == "tool_call" and part.get("name") == "file_capture":
                        args = part.get("arguments", {})
                        if isinstance(args, str):
                            try:
                                args = json.loads(args)
                            except (json.JSONDecodeError, TypeError):
                                continue
                        predicted = args.get("bucket", "").strip().lower()
                        break
    
    return 1.0 if predicted == expected else 0.0
```

## Open Questions

1. **Sync vs Async AIProjectClient**
   - What we know: `AIProjectClient` is a sync client. The project uses async everywhere.
   - What's unclear: Whether `AIProjectClient` accepts `AsyncDefaultAzureCredential`, and whether `get_openai_client()` returns sync or async OpenAI client.
   - Recommendation: Create `AIProjectClient` with sync `DefaultAzureCredential` inside the background task that runs evals. This isolates the sync context from the async event loop.

2. **Custom Evaluator grade() Access to output_items**
   - What we know: Docs say `item.get("sample", {}).get("output_text")` for text, and `{{sample.output_items}}` for structured response. The classifier agent responds via tool calls, not text.
   - What's unclear: Exact structure of `output_items` when accessed inside `grade()` function -- is it a JSON string or a parsed dict? Docs say "This access pattern is subject to change in a future API update."
   - Recommendation: Implement grade() defensively with both string and dict parsing. Test with a small dataset first. If output_items is not accessible in grade(), fall back to pattern-matching on output_text.

3. **Agent Version for Eval Target**
   - What we know: The target config accepts optional `version` field. Omitting it uses the latest version.
   - What's unclear: Whether agents created via `create_agent()` have version numbers. The project creates agents without version management.
   - Recommendation: Omit `version` in the target config to use the latest agent version. This matches the project's pattern of managing agents by ID, not version.

4. **Dataset Caching Strategy**
   - What we know: Each eval run needs a `file_id` from an uploaded dataset. Uploading the same JSONL each time is wasteful.
   - What's unclear: Whether `datasets.upload_file()` deduplicates or creates new versions.
   - Recommendation: Upload once per unique dataset content hash with a version string (`YYYYMMDDHHMMSS-<hash>`). Cache or look up by dataset name/version. Never reuse fixed version `1`.

5. **Direct Target Eval vs App-Mediated Fallback**
   - What we know: Foundry agent target evaluation supports `azure_ai_agent` by name/version and exposes `{{sample.output_items}}` for structured output. Current production code supplies Python tools at request time through `ChatOptions`.
   - What's unclear: Whether the deployed Foundry agent target sees those same local Python tools when Foundry service invokes the agent directly.
   - Recommendation: Add a blocking canary before deletion. If output items include expected tool calls, use direct target eval. If not, generate response/tool-call artifacts through production app clients/tools and run Foundry dataset evaluation over those artifacts.
