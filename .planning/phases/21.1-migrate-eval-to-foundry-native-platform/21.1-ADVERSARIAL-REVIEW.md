# Phase 21.1 Adversarial Review

**Reviewer:** Codex (mounted repo review)
**Date:** 2026-04-24
**Original verdict:** DO NOT EXECUTE AS WRITTEN — address blockers first
**Current status:** Findings folded into CONTEXT, RESEARCH, PATTERNS, 21.1-01-PLAN, and 21.1-02-PLAN on 2026-04-24. The executable plans now include canary-gated direct target eval, app-mediated fallback, deterministic admin routing context, idempotent evaluator registration, restart-safe run discovery, hash-versioned datasets, and preview-risk rollout gating.

---

## Blocker: Foundry target eval may not have the app's Python tools

Plan 01 assumes `sample.output_items` will contain `file_capture`, `add_errand_items`, or `add_task_items` calls (21.1-01-PLAN.md line 181, line 205). But the current agents are created as shells with only model and name (`classifier.py` line 57, `admin.py` line 59), and production passes tools at call time (`adapter.py` line 181, `admin_handoff.py` line 234). Foundry agent target eval invokes an agent by name/version via `azure_ai_agent`, with input messages and target config — not the app's local `ChatOptions` tools.

**Resolution in plans:** Added `run_foundry_target_canary()` and made direct target execution conditional. If the canary fails, the plan uses app-mediated response/tool-call artifact generation with Foundry dataset scoring.

---

## Admin eval removes deterministic routing context

Plan 01 says `routing_context` is no longer needed because the AdminAgent has its own routing context via tools (21.1-01-PLAN.md line 380). Current production explicitly prepends destinations/rules before calling the agent (`admin_handoff.py` line 220).

**Resolution in plans:** Admin eval must prepend deterministic `build_routing_context()` output unless the canary proves direct target eval can call `get_routing_context`.

---

## Evaluator registration is defined but not wired

Original Plan 01 created `register_evaluators()` (old 21.1-01-PLAN.md line 186), but `run_classifier_eval`/`run_admin_eval` did not call it, and `main.py` setup only initialized `AIProjectClient` (old line 403).

**Resolution in plans:** Replaced `register_evaluators()` with `ensure_evaluators_registered()` and required code-hash-based reuse/new-version behavior before eval creation.

---

## Result retrieval after restart is underspecified

Results move out of Cosmos, but Plan 01 replaces durable run state with `_foundry_eval_runs` in memory (21.1-01-PLAN.md line 341). After restart, `get_eval_results(eval_type=...)` needs a deterministic way to discover eval objects/runs by type.

**Resolution in plans:** Added stable eval names (`second-brain-classifier-eval`, `second-brain-admin-eval`), run prefixes, and Foundry listing as source of truth. `_foundry_eval_runs` is now cache-only.

---

## Dataset upload uses fixed version 1

`export_and_upload_dataset()` uploads `name=f"{eval_type}-golden-dataset"`, `version="1"` every time (21.1-01-PLAN.md line 196). That risks version conflicts or stale data.

**Resolution in plans:** Dataset uploads now require timestamp/content-hash versions and must not use fixed `version="1"`.

---

## Production-readiness is overstated

The original context said Foundry eval was GA/production-ready (old 21.1-CONTEXT.md line 53), but current Microsoft docs mark cloud evaluations and custom evaluators as preview/no SLA. The SDK package is stable, but some evaluation features remain preview.

**Resolution in plans:** CONTEXT now marks Foundry cloud/custom evaluation as preview-risk and gates deletion on tests plus a live canary or working app-mediated fallback.
