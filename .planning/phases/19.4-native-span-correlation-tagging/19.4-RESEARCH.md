# Phase 19.4: Native Span Correlation Tagging - Research

**Researched:** 2026-04-19
**Domain:** OpenTelemetry span attributes, Azure Monitor custom dimensions, Cosmos diagnostic log correlation
**Confidence:** HIGH

## Summary

Phase 19.4 must tag all native telemetry rows with `capture_trace_id` so the web segment page's native renderers show real data instead of "runs (0)." The architecture has four emit sites: (1) backend_api AppRequests, (2) Foundry agent SDK spans, (3) Cosmos diagnostic logs via `activityId_g`, and (4) investigation custom spans.

The load-bearing question -- whether OTel baggage propagation flows through the Foundry SDK -- has been partially answered by research. Microsoft's official docs (updated 2026-04-14) document `enable_baggage_propagation` as a parameter of `AIProjectInstrumentor().instrument()` from `azure.ai.projects.telemetry`. However, the project uses `enable_instrumentation()` from `agent_framework.observability`, which does NOT support baggage parameters. Furthermore, OTel baggage does NOT automatically become span attributes -- a `BaggageSpanProcessor` or custom `SpanProcessor` is needed to copy baggage entries to span attributes. This means even if baggage propagation is enabled, a second step (span processor) is required to make `capture_trace_id` appear in App Insights `Properties`.

The recommended approach -- and the spike memo's primary recommendation -- should be a custom `SpanProcessor` that reads `capture_trace_id` from the Python `ContextVar` (which already exists: `capture_trace_id_var`) and injects it as a span attribute on every span. This is simpler than baggage propagation and covers all four emit sites with one piece of infrastructure.

**Primary recommendation:** Use a custom `SpanProcessor` (registered via `configure_azure_monitor(span_processors=[...])`) that reads `capture_trace_id_var` and calls `span.set_attribute("capture.trace_id", value)` on every span start. This single mechanism covers Sites 1, 2, and 4. Site 3 (Cosmos) uses the existing `trace_headers()` helper.

### Pattern 1: Custom SpanProcessor for ContextVar injection
**What:** A `SpanProcessor` subclass that reads a Python `ContextVar` on span start and sets it as a span attribute.
**When to use:** When you need a cross-cutting concern applied to ALL spans (app-created and auto-instrumented).
**Example:**
```python
# Source: Microsoft Learn - Add and Modify OpenTelemetry
# https://learn.microsoft.com/en-us/azure/azure-monitor/app/opentelemetry-add-modify
from opentelemetry.sdk.trace import SpanProcessor
from opentelemetry.trace import Span

from second_brain.tools.classification import capture_trace_id_var


class CaptureTraceSpanProcessor(SpanProcessor):
    """Inject capture_trace_id from ContextVar onto every span.

    The ContextVar is set by the capture adapter before invoking
    the Foundry agent. All spans created within that context
    (including auto-instrumented Azure SDK spans) get the attribute.
    """

    def on_start(self, span: Span, parent_context=None) -> None:
        trace_id = capture_trace_id_var.get("")
        if trace_id:
            span.set_attribute("capture.trace_id", trace_id)

    def on_end(self, span) -> None:
        pass  # No-op
```

**Registration:**
```python
# In main.py, BEFORE enable_instrumentation()
from second_brain.observability.span_processor import CaptureTraceSpanProcessor

configure_azure_monitor(
    logger_name="second_brain",
    span_processors=[CaptureTraceSpanProcessor()],
)
```

### Pattern 2: Cosmos activityId_g mapping via trace_headers()
**What:** Every capture-correlated Cosmos SDK call passes `capture_trace_id` as `x-ms-client-request-id`.
**When to use:** When you need to correlate Cosmos diagnostic logs back to a capture.
**Example:**
```python
# Source: existing spine/cosmos_request_id.py (proven in production)
from second_brain.spine.cosmos_request_id import trace_headers

# Every capture-correlated Cosmos call:
await container.create_item(body=doc, **trace_headers(capture_trace_id))
```

### Anti-Patterns to Avoid
- **Per-site explicit span.set_attribute() for auto-instrumented spans:** You cannot access the auto-instrumented AppRequests span from within FastAPI middleware. The SpanProcessor is the correct injection point.
- **Mixing AIProjectInstrumentor with enable_instrumentation():** Both instrument OTel. Using both could cause double-counting of spans or conflicts. Stick with the existing `enable_instrumentation()` path and use `configure_azure_monitor(span_processors=[...])` for custom processing.
- **Storing full prompt/output in span attributes:** Span attributes are exported to App Insights with 90-day retention and per-GB cost. Keep attributes small (IDs, outcomes, durations). Full content belongs in Foundry (Phase 19.5 concern).
- **Baggage propagation to external services:** Microsoft docs warn that baggage is sent to Azure OpenAI and may be logged. If `capture_trace_id` contains identifiable information, baggage propagation exposes it to the Foundry service.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Injecting attributes on auto-instrumented spans | FastAPI middleware that tries to find "the current span" | `SpanProcessor.on_start()` registered via `configure_azure_monitor(span_processors=[...])` | SpanProcessor is called for EVERY span, including auto-instrumented ones; middleware can only access app-level context |
| Cosmos correlation | Custom HTTP interceptor for Cosmos SDK calls | `trace_headers()` + `x-ms-client-request-id` | The SDK natively supports `initial_headers`; diagnostic logs automatically record `activityId_g` |
| Propagating context across async boundaries | Manual token passing through every function signature | `ContextVar` (already `capture_trace_id_var`) | ContextVar flows through asyncio task boundaries automatically |

**Key insight:** The `SpanProcessor` approach is strictly superior to per-site injection because it covers ALL spans -- app-created, middleware-created, and Azure SDK auto-instrumented -- with zero per-site code changes. The spike memo's primary job is confirming this approach works empirically (deploy, capture, query KQL, verify `Properties.capture_trace_id` appears).

## Common Pitfalls

### Pitfall 1: ContextVar not set for auto-instrumented AppRequests span
**What goes wrong:** The `capture_trace_id_var` ContextVar is set by `adapter.py` INSIDE the streaming generator, but the auto-instrumented `AppRequests` span for the `/api/capture` endpoint is created BEFORE the generator runs (at request entry time).
**Why it happens:** FastAPI auto-instrumentation creates the request span when the request arrives. The capture handler generates a `capture_trace_id` and sets `request.state.capture_trace_id`, but `capture_trace_id_var` is set later inside the streaming adapter.
**How to avoid:** The `SpanProcessor` must also check `request.state` or the capture handler must set `capture_trace_id_var` at the handler level (before returning the StreamingResponse). Alternatively, read from OTel context/baggage if the handler sets it early enough.
**Warning signs:** `AppRequests` rows in KQL have empty `capture_trace_id` while child spans (`capture_text`) have it.

### Pitfall 2: SpanProcessor on_start receives ReadableSpan in some cases
**What goes wrong:** The Microsoft docs show setting `span._attributes` in `on_end`, but `on_start` receives a writeable `Span` where `span.set_attribute()` works directly.
**Why it happens:** `on_start` gets the mutable `Span` object; `on_end` gets a `ReadableSpan` (immutable). For injecting attributes, `on_start` is correct.
**How to avoid:** Use `on_start` (not `on_end`) for attribute injection. Use `span.set_attribute()` (public API), not `span._attributes` (private).
**Warning signs:** Attributes not appearing on spans.

### Pitfall 3: Cosmos activityId_g filter assumes 1:1 mapping
**What goes wrong:** The KQL filter `where activityId_g == "{capture_trace_id}"` assumes the `x-ms-client-request-id` was set to the `capture_trace_id`. If a Cosmos call site doesn't use `trace_headers()`, its diagnostic logs won't be found.
**Why it happens:** Not all Cosmos call sites are capture-correlated. Inbox reads for the errands screen, affinity rule lookups, etc. are NOT correlated -- and shouldn't be.
**How to avoid:** Audit which Cosmos call sites are capture-correlated (classification writes, admin inbox updates) vs. uncorrelated (errands listing, health checks). Only the correlated sites need `trace_headers()`.
**Warning signs:** Cosmos diagnostic logs showing zero results for a known capture.

### Pitfall 4: Investigation spans have no capture_trace_id
**What goes wrong:** Investigation agent runs are NOT correlated to a capture -- they're user-initiated questions. There is no `capture_trace_id` for investigation.
**Why it happens:** Investigation is a separate workflow. The user asks a question; there's no underlying capture being classified.
**How to avoid:** Investigation spans use `thread_id` for correlation, not `capture_trace_id`. The `CaptureTraceSpanProcessor` will correctly no-op for investigation spans because `capture_trace_id_var` will be empty.
**Warning signs:** Trying to force investigation spans into the capture correlation model.

### Pitfall 5: Auto-format hook strips unused imports
**What goes wrong:** If you add an import in one edit and use it in a subsequent edit, the ruff auto-format hook removes the import between edits.
**Why it happens:** Ruff removes unused imports on every file save.
**How to avoid:** Write the import and its usage in a single Write operation. This is documented in MEMORY.md (Phase 17.1 lesson).
**Warning signs:** `NameError` or `ImportError` after what seemed like a correct edit sequence.

## Open Questions

1. **Does `capture_trace_id_var` have a value at AppRequests span creation time?**
   - What we know: The ContextVar is set inside `adapter.py` (inside the generator). The AppRequests span is created by auto-instrumentation when the request enters FastAPI, BEFORE the generator runs.
   - What's unclear: Whether the SpanProcessor's `on_start` for the AppRequests span fires before or after the capture handler sets `request.state.capture_trace_id`.
   - Recommendation: The spike memo must test this empirically. If the ContextVar isn't set in time for AppRequests, the capture handler (`api/capture.py`) must set `capture_trace_id_var` BEFORE returning the `StreamingResponse`. This is a one-line change: `capture_trace_id_var.set(capture_trace_id)` at line 211 (after `request.state.capture_trace_id = capture_trace_id`).

2. **Do AuditAgentMiddleware / ToolTimingMiddleware spans inherit the ContextVar?**
   - What we know: These middleware create spans inside `tracer.start_as_current_span()`. The ContextVar is set in the adapter (parent call context). ContextVars flow through sync calls within the same async task.
   - What's unclear: Whether the Foundry SDK's internal async machinery preserves the ContextVar across its internal awaits. The SDK uses `azure-core`'s HTTP pipeline which may create new async contexts.
   - Recommendation: Spike memo tests this by checking if `AuditAgentMiddleware` spans in `AppDependencies` carry the `capture_trace_id` custom dimension after deploying the SpanProcessor.

3. **activityId_g mapping: spine_events vs correlation-map?**
   - What we know: Today, `trace_headers()` sets `x-ms-client-request-id = capture_trace_id`. The Cosmos diagnostic log's `activityId_g` equals this value. The KQL query already filters by `activityId_g == capture_trace_id`.
   - What's unclear: Whether additional Cosmos call sites need `trace_headers()` and whether we need a separate mapping table or if the direct activityId_g==capture_trace_id equality is sufficient.
   - Recommendation: Audit capture-correlated Cosmos call sites during the spike. If `trace_headers()` is already applied at all capture-correlated sites, no mapping table is needed -- the direct equality filter works.

## Spike Memo Recommendations (for Plan 01)

The spike memo should investigate empirically:

1. **Deploy `CaptureTraceSpanProcessor` to staging/prod.** Register via `configure_azure_monitor(span_processors=[...])`. Set `capture_trace_id_var` early in the capture handler (before StreamingResponse).

2. **Perform one fresh capture.** Record the `capture_trace_id`.

3. **Query KQL for each emit site:**
   - `AppRequests | where tostring(Properties.capture_trace_id) == "{id}"` -- proves Site 1
   - `AppDependencies | where Name endswith "_agent_run" | where tostring(Properties.capture_trace_id) == "{id}"` -- proves Site 2
   - `AzureDiagnostics | where activityId_g == "{id}"` -- proves Site 3
   - `AppDependencies | where Name == "investigate" | where tostring(Properties.capture_trace_id) == "{id}"` -- proves Site 4 (if investigation has a trace_id)

4. **Document which sites work and which don't.** If the SpanProcessor approach covers Sites 1+2+4, Plans 02+04 merge.

5. **Audit Cosmos call sites.** List every `container.create_item/upsert_item/read_item` in capture-correlated code paths. Flag any missing `trace_headers()` calls.

6. **Decide activityId_g mapping strategy.** If the direct `activityId_g == capture_trace_id` pattern works (as evidence suggests), recommend NO new mapping table.
