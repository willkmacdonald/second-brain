## Decisions Made
- **ContextVar timing:** The ASGI auto-instrumentation creates the AppRequests span BEFORE the capture handler executes. The SpanProcessor's on_start sees an empty ContextVar for AppRequests. Fix: use `get_current_span().set_attribute()` in the handler (retroactive tag before span ends). This becomes Plan 02 Task 5.1.
- **activityId_g mapping:** No new correlation-map table needed. `activityId_g` is a Cosmos server-side UUID, not `x-ms-client-request-id`. The SpanProcessor tags Cosmos SDK spans in AppDependencies with `capture.trace_id`, making direct equality filter sufficient. MEMORY.md documentation about `activityId_g` is incorrect and needs correction (Plan 02 Task 5.4).
- **Plan 02 scope:** Merge original Plans 02+04 into one plan. SpanProcessor covers Sites 2 (Foundry) and 4 (Investigation/custom) automatically. Only Site 1 needs explicit per-site code. The 6 tasks from memo section 5 become Plan 02 verbatim.
- **trace_headers() redundancy:** The SpanProcessor makes `trace_headers()` partially redundant for Cosmos correlation (AppDependencies covers it). However, `trace_headers()` remains valuable for AzureDiagnostics server-side metrics (RU charge, duration). 3 capture-correlated call sites still missing it (Plan 02 Task 5.2).
