**Gathered:** 2026-04-19

<domain>
## Phase Boundary

This phase makes native telemetry (App Insights spans, Foundry agent spans, Cosmos diagnostic logs) filterable by `capture_trace_id`. Today the spine ledger works but the web segment page shows "runs (0)" because native renderers filter by `capture_trace_id` and those spans aren't tagged. After this phase, the drill-down from spine → segment → native telemetry shows real data.

Zero UI changes beyond the segment page now showing real native runs (Foundry runs, App Insights requests). Agent decision content (prompt/output/confidence) is Phase 19.5.

</domain>

<decisions>
## Implementation Decisions

### Spike-First Architecture
- Plan 01 is a spike memo that resolves the OTel baggage question empirically before any implementation
- Spike memo gates Plan 02 scope — pattern matches 19.2-01
- If baggage works → Plans 02+04 merge (1 middleware config covers backend_api + Foundry + investigation)
- If baggage doesn't work → explicit `span.set_attribute()` at each emit site (3 separate plans)

### Four Emit Sites (locked)
- Site 1: `backend_api` AppRequests — FastAPI middleware layer
- Site 2: Azure AI Foundry agent spans — runs/steps/tool calls from the Foundry SDK
- Site 3: Cosmos diagnostic logs — `AzureDiagnostics.activityId_g` pivot to `capture_trace_id`
- Site 4: Investigation custom spans — thread-kind spans emitted during investigation runs

### Cosmos Mapping Strategy (locked)
- `trace_headers()` helper in `spine/cosmos_request_id.py` is proven and tested
- Phase 19.4 extends it to remaining capture-correlated Cosmos call sites — no bespoke instrumentation
- Records `(capture_trace_id, activityId_g)` tuples; persistence via spine_events or correlation-map

### Native Capability Check Citations (locked)
- Surface 2 / Baggage propagation → unverified → spike memo required
- Surface 6 / Per-operation client-request-id → proven → extend existing `trace_headers()`
- Surface 1 / Tracing to App Insights → Foundry spans carry `operation_Id` but NOT `capture_trace_id`
- Surface 2 / Cross-process context → httpx calls in recipe tools NOT auto-instrumented → low priority

### Confirmation Checkpoints (locked)
- Checkpoint A (pre-plan 02): memo approved
- Checkpoint B (post-plan 02): live KQL proves `backend_api` spans carry the tag
- Checkpoint C (post-plan 03): live query proves `activityId_g` pivot works
- Checkpoint D (post-plan 05): operator performs fresh capture; drills spine → segment → native; sees real data

### Claude's Discretion
- Exact OTel API calls for baggage vs. span attributes (determined by spike findings)
- Whether `activityId_g` mapping lands in spine_events or a new correlation-map table (memo decides)
- httpx recipe tool instrumentation priority (spec says low priority)
- Test strategy for each emit site
