# Phase 17.3: Address Critical Observability Gaps - Research

**Researched:** 2026-04-11
**Domain:** Mobile crash reporting, React error boundaries, Expo/Sentry integration
**Confidence:** HIGH

## Summary

Phase 17.3 addresses two critical observability blind spots identified in docs/instrumentation.md: (1) the mobile app has zero crash reporting -- if the app crashes from an unhandled JS exception, native module failure, or OOM, there is no signal whatsoever, and (2) there are no React error boundaries -- a rendering error in any component (inbox list, capture screen, etc.) takes down the entire screen with no recovery path and no telemetry.

The standard solution for both problems is `@sentry/react-native`, which provides crash reporting, error boundaries, and performance monitoring in a single SDK. Version 8.7.0 supports Expo SDK 54 (this project's version), React Native 0.81, React 19, and the New Architecture. Sentry's free Developer tier (5,000 errors/month, 1 user) is more than sufficient for this single-user hobby project. The existing `reportError()` / `/api/telemetry` pipeline should be preserved for operational events but Sentry becomes the primary crash and error reporting mechanism.

**Primary recommendation:** Install `@sentry/react-native` 8.x with the Expo config plugin, wrap the root layout with `Sentry.wrap()` and `Sentry.ErrorBoundary`, create a metro.config.js for source maps, and trigger a new EAS development build (native module dependency).

### Pattern 1: Sentry Initialization (sentry.ts)

**What:** Centralized Sentry configuration module imported by root layout.
**When to use:** Once, at app startup in root `_layout.tsx`.
**Example:**
```typescript
// mobile/lib/sentry.ts
// Source: https://docs.sentry.io/platforms/react-native/manual-setup/expo/
import * as Sentry from "@sentry/react-native";
import { isRunningInExpoGo } from "expo";

const navigationIntegration = Sentry.reactNavigationIntegration({
  enableTimeToInitialDisplay: !isRunningInExpoGo(),
});

export function initSentry(): void {
  Sentry.init({
    dsn: process.env.EXPO_PUBLIC_SENTRY_DSN ?? "",
    tracesSampleRate: 1.0,       // 100% for single-user app
    enableNativeFramesTracking: !isRunningInExpoGo(),
    integrations: [navigationIntegration],
    // Don't send PII (email, IP) -- single-user system but good hygiene
    sendDefaultPii: false,
    // Only enable in non-dev (EAS builds)
    enabled: !__DEV__,
  });
}

export { Sentry, navigationIntegration };
```

### Pattern 2: Root Layout Wrapping

**What:** Wrapping the root layout with Sentry.wrap() for navigation tracking and global error boundary.
**When to use:** In `app/_layout.tsx`.
**Example:**
```typescript
// mobile/app/_layout.tsx
// Source: https://docs.sentry.io/platforms/react-native/tracing/instrumentation/expo-router/
import * as Sentry from "@sentry/react-native";
import { Stack, useNavigationContainerRef } from "expo-router";
import { useEffect } from "react";
import { GestureHandlerRootView } from "react-native-gesture-handler";
import { initSentry, navigationIntegration } from "../lib/sentry";
import { ErrorFallback } from "../components/ErrorFallback";

initSentry();

function RootLayout() {
  const ref = useNavigationContainerRef();

  useEffect(() => {
    if (ref?.current) {
      navigationIntegration.registerNavigationContainer(ref);
    }
  }, [ref]);

  return (
    <GestureHandlerRootView style={{ flex: 1 }}>
      <Sentry.ErrorBoundary fallback={ErrorFallback}>
        <Stack screenOptions={{ headerShown: false }}>
          <Stack.Screen name="(tabs)" />
          <Stack.Screen
            name="conversation/[threadId]"
            options={{
              headerShown: true,
              headerTitle: "",
              headerStyle: { backgroundColor: "#0f0f23" },
              headerTintColor: "#ffffff",
            }}
          />
        </Stack>
      </Sentry.ErrorBoundary>
    </GestureHandlerRootView>
  );
}

export default Sentry.wrap(RootLayout);
```

### Pattern 3: Error Boundary Fallback Component

**What:** Reusable fallback UI shown when a rendering error is caught.
**When to use:** Passed to `Sentry.ErrorBoundary` `fallback` prop.
**Example:**
```typescript
// mobile/components/ErrorFallback.tsx
// Source: https://docs.sentry.io/platforms/react-native/integrations/error-boundary/
import { View, Text, Pressable, StyleSheet } from "react-native";

interface ErrorFallbackProps {
  error: Error;
  componentStack: string | null;
  resetError: () => void;
  eventId: string;
}

export function ErrorFallback({ error, resetError }: ErrorFallbackProps) {
  return (
    <View style={styles.container}>
      <Text style={styles.title}>Something went wrong</Text>
      <Text style={styles.message}>{error.message}</Text>
      <Pressable style={styles.button} onPress={resetError}>
        <Text style={styles.buttonText}>Try Again</Text>
      </Pressable>
    </View>
  );
}

const styles = StyleSheet.create({
  container: {
    flex: 1,
    backgroundColor: "#0f0f23",
    justifyContent: "center",
    alignItems: "center",
    padding: 24,
  },
  title: { color: "#ffffff", fontSize: 20, fontWeight: "bold", marginBottom: 12 },
  message: { color: "#999", fontSize: 14, textAlign: "center", marginBottom: 24 },
  button: {
    backgroundColor: "#4a90d9",
    paddingHorizontal: 24,
    paddingVertical: 12,
    borderRadius: 8,
  },
  buttonText: { color: "#ffffff", fontSize: 16, fontWeight: "600" },
});
```

### Anti-Patterns to Avoid
- **Wrapping every component individually with ErrorBoundary:** One root-level boundary is sufficient for a small app. Per-screen boundaries are optional refinement, not a requirement.
- **Setting `enabled: true` in dev builds:** Sentry will spam your dashboard with development errors. Use `enabled: !__DEV__`.
- **Removing the existing `reportError()` pipeline:** It still serves a purpose for operational telemetry (capture flow events with trace IDs). Sentry handles crashes/boundaries; the existing pipeline handles structured operational events.
- **Using `sentry-expo` package:** That is the OLD package for SDK 49 and earlier. For SDK 50+, use `@sentry/react-native` directly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Crash reporting | Custom global error handler + POST to backend | `@sentry/react-native` | Native crashes (OOM, Objective-C exceptions, JNI crashes) cannot be caught by JS. Sentry has native iOS/Android crash handlers. Source map symbolication is non-trivial. |
| Error boundaries | Custom `class ErrorBoundary extends React.Component` | `Sentry.ErrorBoundary` | Automatically reports to Sentry with component stack, provides `resetError`, handles `beforeCapture` for context tagging. Building your own means maintaining the Sentry reporting integration separately. |
| Source map upload | Manual upload script | Sentry Expo config plugin (`@sentry/react-native/expo`) | Plugin hooks into EAS Build automatically. Manual upload requires managing auth tokens, release names, and dist values across builds. |
| Metro config for Sentry | Custom Metro plugin | `getSentryExpoConfig()` | Handles Debug ID injection for source map association. Custom approaches break on Metro updates. |

**Key insight:** Crash reporting requires native code integration (iOS/Android crash signal handlers) that cannot be achieved from JavaScript alone. The existing `reportError()` function is useful for explicit error reporting in try/catch blocks, but it fundamentally cannot catch unhandled exceptions, native crashes, or rendering errors.

## Common Pitfalls

### Pitfall 1: Forgetting the New EAS Build
**What goes wrong:** `@sentry/react-native` includes native modules. Installing it and trying to use the existing dev-client build will crash with "Native module not found" error.
**Why it happens:** Expo managed workflow requires a new native build whenever a package with native code is added.
**How to avoid:** After installing, immediately run `eas build --profile development --platform ios` (or the relevant platform). Update the dev build on the physical device.
**Warning signs:** App crashes on launch with "RNSentry" native module not found error.

### Pitfall 2: Missing SENTRY_AUTH_TOKEN in EAS Build
**What goes wrong:** Source maps are not uploaded during EAS Build, so stack traces in Sentry are minified/unreadable.
**Why it happens:** The Sentry Expo plugin needs `SENTRY_AUTH_TOKEN` set as an EAS secret or environment variable.
**How to avoid:** Set the token via `eas secret:create --name SENTRY_AUTH_TOKEN --value <token>` before building.
**Warning signs:** Sentry errors show minified function names instead of original source locations.

### Pitfall 3: Sentry.init() Must Be Called Before Any Rendering
**What goes wrong:** Early crashes or errors are missed if Sentry initializes too late.
**Why it happens:** If `Sentry.init()` is inside a useEffect or component body, errors during initial render are uncaught.
**How to avoid:** Call `initSentry()` at module scope in `_layout.tsx` (outside the component function), as shown in the architecture pattern above.
**Warning signs:** Some crashes don't appear in Sentry; only errors after the first successful render are captured.

### Pitfall 4: Development Mode Error Double-Counting
**What goes wrong:** In dev mode, React rethrows errors caught by error boundaries to the global handler, causing duplicate Sentry reports.
**Why it happens:** React's development-mode error overlay intercepts errors before the boundary can handle them cleanly.
**How to avoid:** Disable Sentry in dev (`enabled: !__DEV__`). Test error boundary behavior in production/preview EAS builds.
**Warning signs:** Seeing duplicate error reports with different stack traces for the same rendering error.

### Pitfall 5: DSN Exposure in Client Bundles
**What goes wrong:** The Sentry DSN is embedded in the JavaScript bundle, visible to anyone who decompiles the app.
**Why it happens:** Client-side SDKs require the DSN to send events.
**How to avoid:** This is expected and acceptable. DSNs are designed to be public. Sentry has server-side rate limiting and allowed-domain restrictions. Set up rate limits in the Sentry project settings.
**Warning signs:** None -- this is by design, not a security issue.

### Pitfall 6: Using `npm install` Instead of `npx expo install`
**What goes wrong:** Version mismatch between `@sentry/react-native` and Expo SDK 54.
**Why it happens:** `npm install` grabs the latest version without checking Expo compatibility.
**How to avoid:** Always use `npx expo install @sentry/react-native` for Expo projects. This was a lesson from Phase 14 (expo-clipboard incident).
**Warning signs:** Build failures or runtime errors related to native module version mismatches.

## Open Questions

1. **Sentry organization/project slug**
   - What we know: Will needs a Sentry account (free tier). DSN and org slug are needed for config.
   - What's unclear: Whether Will already has a Sentry account.
   - Recommendation: First task in the plan should be Sentry account/project creation. This is a manual step that must happen before any code.

2. **Per-screen vs root-only error boundaries**
   - What we know: A single root `Sentry.ErrorBoundary` catches all rendering errors app-wide.
   - What's unclear: Whether individual screen boundaries (capture, inbox, status) provide enough incremental value for a 3-screen app.
   - Recommendation: Start with root-only. It's simpler and covers the critical gap. Per-screen boundaries are a future refinement if specific screens need isolated recovery behavior.

3. **Coexistence with existing `/api/telemetry` pipeline**
   - What we know: The existing `reportError()` function sends structured events with `captureTraceId` to the backend where they appear in App Insights. Sentry would handle crash/boundary reporting separately.
   - What's unclear: Whether to eventually migrate all `reportError()` calls to Sentry.
   - Recommendation: Keep both. The `/api/telemetry` pipeline serves a different purpose (operational events in App Insights alongside backend traces). Sentry handles what the backend pipeline cannot: unhandled exceptions, native crashes, rendering errors. The two systems are complementary, not redundant.
