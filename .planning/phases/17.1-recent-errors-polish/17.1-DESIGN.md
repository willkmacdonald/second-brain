# Phase 17.1 Design — recent_errors Tool Polish

**Created:** 2026-04-08
**Status:** Approved, ready for planning
**Predecessor:** Phase 17 (Investigation Agent) shipped 2026-04-06
**Source:** Issues surfaced during Phase 17 live verification (Test 2)

---

## Goal

Eliminate seven presentation/correctness issues in the `recent_errors` Investigation Agent tool so its table output is unambiguous, faithful to the underlying data, and reports totals correctly. Apply a small symmetry fix to `trace_lifecycle` for the same null-handling pattern. Add narrowly-scoped unit tests that pin the highest-value invariants.

This is a polish phase, not a redesign. Phase 17's architecture, tool interface names, SSE event types, agent registration, and endpoint logic are all unchanged.

---

## Background

Phase 17 shipped the Investigation Agent on 2026-04-06. Live verification (Test 1, Test 2) against the deployed endpoint at `https://brain.willmacdonald.com/api/investigate` confirmed end-to-end SSE streaming, multi-turn conversation continuity, and clinical-tone output. The Investigation Agent exists in the Foundry portal as `asst_5feSWWTMA8rBSUyQo6aSCsEJ`, with working instructions, and the env var is set on the Container App.

However, Test 2's `recent_errors` output exposed quality issues:

1. **Trace ID column rendered as empty string** for rows where `Properties.capture_trace_id` was missing (e.g., `investigation_agent` self-errors and `JSONDecodeError` rows from the Azure SDK). Empty cells are ambiguous — users could read them as broken data rather than absent data.
2. **Component column rendered as empty string** for rows where `Properties.component` was missing (e.g., exceptions caught by the Azure SDK that didn't carry our extras dict). Same ambiguity.
3. **Duplicate `JSONDecodeError` rows** appeared because Python's exception chaining produced two telemetry entries for one logical failure. (User decided no deduping — out of scope for this phase.)
4. **Trace ID column width** would have line-wrapped if a real 36-char UUID had appeared. The existing portal instructions say to print full UUIDs verbatim, but the table layout doesn't accommodate them gracefully.
5. **Silent truncation bug** — the tool's `count` field is `len(records)` post-cap, so the agent literally cannot say "showing 10 of N" when results are capped. The KQL `take 10` silently drops everything past the limit, and the Python parser has no way to know the true total.
6. **Empty string vs null semantics** — `tostring(Properties.component)` in KQL returns `""` for missing dynamic fields, not `null`. The Python parser passes the empty string straight through to Pydantic. The agent then sees `"component": ""` in the JSON response and faithfully renders it as a blank table cell.
7. **`_SEVERITY_MAP` bug** — the map at `queries.py:201` reads `{"warning": 3, "error": 4}`, but Azure's actual SeverityLevel scale is `2=Warning, 3=Error, 4=Critical`. The current `recent_errors` default of `severity="error"` therefore filters for **Critical only**, not Error. This is a latent bug that explains why Test 2 missed some rows we'd expect to see.

The user's stated bar for this phase was "I don't want any issues" — meaning all seven of the above must be resolved before phase completion.

## Scope

### In scope

- **`RECENT_FAILURES_FILTERED` KQL template** — rewrite to return total count + top N rows in a single multi-statement query
- **`_SEVERITY_MAP` fix** in `queries.py` — correct values to match Azure's scale
- **`query_recent_failures_filtered` parser** — return new `FailureQueryResult` wrapper instead of `list[FailureRecord]`
- **`FailureRecord` model** — add `field_validator` to normalize empty strings to `None` for `component` and `capture_trace_id`; add `capture_trace_id_short` computed field
- **`TraceRecord` model** — same `field_validator` for symmetry (same one-line change, fixes Issue 6 in `trace_lifecycle` too)
- **`FailureQueryResult` new model** — wraps `total_count`, `returned_count`, `truncated`, `records`
- **`recent_errors` tool JSON response** — new shape with `total_count`, `returned_count`, `truncated`, `severity` fields; remove obsolete `note` and `count` fields; add `_RESULT_LIMIT = 10` constant
- **Foundry portal instructions** — modify the existing `InvestigationAgent` instructions in place (replace conflicting rules, add new rendering rules)
- **`docs/foundry/investigation-agent-instructions.md`** — new file in the repo containing the canonical version-controlled source of truth for the portal instructions
- **`backend/tests/test_investigation_queries.py`** — narrowly-scoped unit tests covering severity map, KQL string construction, and truncation branching logic (6 tests, ~80 lines)

### Out of scope

- **Exception chain deduplication** — user explicitly chose no deduping
- **Filtering out `investigation_agent` self-errors** — opinionated design decision deferred
- **Microsecond precision in displayed timestamps** — agent will continue to drop microseconds in display
- **The other 3 tools at the JSON-response level** (`trace_lifecycle` JSON shape, `system_health`, `usage_patterns`) — only `recent_errors` had its output reviewed in Test 2; live audit of the other tools deferred
- **Unit tests for `system_health` and `usage_patterns`** — no observed bugs and thin parser logic; YAGNI
- **SSE adapter (`investigation_adapter.py`) or `/api/investigate` endpoint** — no changes
- **Agent registration (`agents/investigation.py`)** — no changes
- **Any other KQL templates** — only `RECENT_FAILURES_FILTERED` is rewritten

### Non-goals

- Changing Phase 17's agent architecture
- Changing tool interface names or signatures (other than the internal return type of `query_recent_failures_filtered`, which has only one caller)
- Changing SSE event types or the endpoint contract

### Component changes

**`backend/src/second_brain/observability/models.py`** (modified)

```python
from pydantic import BaseModel, computed_field, field_validator


class FailureRecord(BaseModel):
    timestamp: str
    item_type: str
    severity_level: int | None = None
    message: str
    component: str | None = None
    capture_trace_id: str | None = None

    @field_validator("component", "capture_trace_id", mode="before")
    @classmethod
    def _empty_to_none(cls, v: str | None) -> str | None:
        if v is None or (isinstance(v, str) and v.strip() == ""):
            return None
        return v

    @computed_field
    @property
    def capture_trace_id_short(self) -> str | None:
        if self.capture_trace_id is None:
            return None
        return self.capture_trace_id[:8]


class TraceRecord(BaseModel):
    timestamp: str
    item_type: str
    severity_level: int | None = None
    message: str
    component: str | None = None
    capture_trace_id: str | None = None

    @field_validator("component", "capture_trace_id", mode="before")
    @classmethod
    def _empty_to_none(cls, v: str | None) -> str | None:
        if v is None or (isinstance(v, str) and v.strip() == ""):
            return None
        return v


class FailureQueryResult(BaseModel):
    """Result wrapper for recent_failures queries with total count metadata."""
    total_count: int          # true total before the take N cap
    returned_count: int       # actual rows returned (<= total_count)
    truncated: bool           # True if returned_count < total_count
    records: list[FailureRecord]
```

**`backend/src/second_brain/observability/kql_templates.py`** (modified — `RECENT_FAILURES_FILTERED` rewrite)

```kql
let filtered = union withsource=SourceTable
    (AppTraces | where SeverityLevel >= {severity_filter}),
    (AppExceptions | where SeverityLevel >= {severity_filter})
{component_filter}| project
    timestamp = TimeGenerated,
    ItemType = case(
        SourceTable == "AppTraces", "Log",
        SourceTable == "AppExceptions", "Exception",
        SourceTable
    ),
    severityLevel = SeverityLevel,
    Message = coalesce(Message, ExceptionType),
    Component = tostring(Properties.component),
    CaptureTraceId = tostring(Properties.capture_trace_id);
filtered | summarize total_count = count();
filtered | order by timestamp desc | take {limit};
```

The docstring above the template will note: "Returns TWO tables — table 0 is `[{total_count: int}]`, table 1 is the failure rows. The Python parser depends on this order."

**`backend/src/second_brain/observability/queries.py`** (modified)

Bug fix at `_SEVERITY_MAP`:

```python
# Azure SeverityLevel scale: 0=Verbose, 1=Info, 2=Warning, 3=Error, 4=Critical.
# Do NOT change without verifying against Azure docs — historical bug:
# 2026-04-08, this map had warning=3, error=4 which silently filtered
# out all Error-level entries and only showed Critical-level. Test 2 missed
# errors as a result.
_SEVERITY_MAP: dict[str, int] = {
    "warning": 2,
    "error": 3,
}
```

Rewrite of `query_recent_failures_filtered`:

```python
async def query_recent_failures_filtered(
    client: LogsQueryClient,
    workspace_id: str,
    component: str | None = None,
    severity: str = "error",
    limit: int = 10,
    timespan: timedelta = timedelta(hours=24),
) -> FailureQueryResult:
    """Return recent failures with total count metadata.

    The KQL template returns two tables: a single-row count, and the
    top N filtered rows. This lets the agent report 'showing N of M'
    when results are capped.
    """
    severity_level = _SEVERITY_MAP.get(severity, 3)

    component_filter = ""
    if component:
        component_filter = f'| where tostring(Properties.component) == "{component}"\n'

    query = RECENT_FAILURES_FILTERED.format(
        component_filter=component_filter,
        severity_filter=severity_level,
        limit=limit,
    )

    result = await execute_kql(
        client,
        workspace_id,
        query,
        timespan=timespan,
        server_timeout=30,
    )

    # Defensive: KQL must return at least 2 tables (count + rows).
    if not result.tables or len(result.tables) < 2:
        return FailureQueryResult(
            total_count=0, returned_count=0, truncated=False, records=[]
        )

    # Table 0: single row with total_count
    total_row = result.tables[0][0] if result.tables[0] else {}
    total_count = int(total_row.get("total_count", 0) or 0)

    # Table 1: up to {limit} failure rows
    records: list[FailureRecord] = []
    for row in result.tables[1]:
        records.append(
            FailureRecord(
                timestamp=str(row.get("timestamp", "")),
                item_type=str(row.get("ItemType", "")),
                severity_level=row.get("severityLevel"),
                message=str(row.get("Message", "")),
                component=row.get("Component"),
                capture_trace_id=row.get("CaptureTraceId"),
            )
        )

    return FailureQueryResult(
        total_count=total_count,
        returned_count=len(records),
        truncated=len(records) < total_count,
        records=records,
    )
```

The existing `query_capture_trace` function is unchanged at the call-site level — the new `TraceRecord` validator handles the empty-string normalization automatically without parser changes.

**`backend/src/second_brain/tools/investigation.py`** (modified — `recent_errors` tool only)

```python
_RESULT_LIMIT = 10  # Mirrored in portal instructions; keep in sync.


@tool(approval_mode="never_require")
async def recent_errors(
    self,
    time_range: Annotated[str, Field(description="...")] = "24h",
    component: Annotated[str | None, Field(description="...")] = None,
) -> str:
    """Query recent errors and failures from App Insights.

    Returns Error- and Critical-level log entries with optional component
    filtering. Results are capped at 10 most recent entries; the response
    includes total_count so the agent can report 'showing N of M' when
    truncated.
    """
    log_extra: dict = {"component": "investigation_agent"}
    logger.info(
        "recent_errors called: time_range=%s component=%s",
        time_range,
        component,
        extra=log_extra,
    )

    try:
        time_range = _validate_time_range(time_range, "24h")
        _kql_duration, td = TIME_RANGE_MAP[time_range]

        result = await query_recent_failures_filtered(
            self._logs_client,
            self._workspace_id,
            component=component,
            severity="error",
            limit=_RESULT_LIMIT,
            timespan=td,
        )

        return json.dumps(
            {
                "total_count": result.total_count,
                "returned_count": result.returned_count,
                "truncated": result.truncated,
                "time_range": time_range,
                "component_filter": component,
                "severity": "error_or_critical",
                "errors": [r.model_dump() for r in result.records],
            },
            default=str,
        )

    except Exception as exc:
        logger.error(
            "recent_errors error: %s", exc, exc_info=True, extra=log_extra
        )
        return json.dumps({"error": f"Failed to query recent errors: {exc}"})
```

**`backend/tests/test_investigation_queries.py`** (new — 6 tests, narrowly scoped)

Covers exactly three failure modes:

1. **`_SEVERITY_MAP` value pinning** — regression test against the historical bug
2. **KQL string construction** — verify `severity="error"` produces `SeverityLevel >= 3` in the query, and `severity="warning"` produces `SeverityLevel >= 2`
3. **Truncation branching logic** — `truncated=False` when total equals returned, `truncated=True` when total exceeds returned, empty result returns sensible defaults

No tests for Pydantic field validators or computed fields (those are testing the framework, not our logic). No tests for `_validate_time_range` (the existing 5-line helper is well-covered by static reading).

Test framework: `pytest` + `pytest-asyncio` + `unittest.mock.AsyncMock`. Mocks `execute_kql` so nothing hits Azure.

## Risks & mitigations

| # | Risk | Mitigation |
|---|------|------------|
| 1 | Multi-statement KQL may not return tables in the expected order. | Defensive `if len(result.tables) < 2: return empty result` guard prevents crashes. Unit tests assert on expected order — if Log Analytics ever changes behavior, tests fail loudly before deploy. |
| 2 | `field_validator` is a parse-time check only. If Azure SDK ever returns a non-string for `Component`/`CaptureTraceId`, the `isinstance(v, str)` check passes it through. | Pydantic type validation downstream (`component: str \| None`) will reject the value with a clear error visible in logs. Acceptable — silent corruption is the only unacceptable failure mode. |
| 3 | Foundry portal instructions can drift from the JSON schema the tool emits. | New `docs/foundry/investigation-agent-instructions.md` file is the canonical source in git. Header comment instructs all editors to keep both in sync. Phase verification gate explicitly compares portal contents to file. |
| 4 | Severity filter now applied to AppExceptions for the first time — any handled exceptions logged at Warning (level 2) will disappear from results. | Intended behavior per user decision. `logger.exception()` defaults to Error severity. If future use surfaces a need for Warning-level visibility, the existing `severity="warning"` parameter is already wired up at the query layer. |
| 5 | `capture_trace_id_short` as a computed field changes `model_dump()` output for every caller of `FailureRecord`. | Only caller is `recent_errors` in `tools/investigation.py`. No external API consumes `FailureRecord` shape. Verified by file search. |
| 6 | The captured baseline of the existing portal instructions might be out of date (someone edits the portal between capture and paste). | Capture happens manually as the first step of execution (user pastes current portal text into the doc file). Within a single plan execution, this is atomic enough to be safe. |

## Definition of Done

The phase is complete when ALL FOUR gates pass:

1. **Code shipped** — All four file modifications and the new test file are merged to `main`, CI/CD has built the image, and Azure Container Apps shows the new revision active. `curl https://brain.willmacdonald.com/health` returns 200.

2. **Doc file exists** — `docs/foundry/investigation-agent-instructions.md` is in the repo and contains the captured baseline plus the new rendering rules. The header comment directing future editors to keep portal+file in sync is present.

3. **Portal matches doc** — You have manually pasted the contents of `docs/foundry/investigation-agent-instructions.md` into the Foundry portal at `ai.azure.com → second-brain-foundry-resource → InvestigationAgent → Instructions` and clicked Save. (No automated check; this is a manual verification step performed by the user.)

4. **Test 2 re-verified** — Send a `POST /api/investigate` request to `https://brain.willmacdonald.com/api/investigate` with body `{"question": "show me errors from the last 24h"}` and the existing API key auth header. Read the SSE stream until the `done` event. The accumulated `text` events must form an answer that:
   - Has zero blank cells in the table (Component and Trace ID columns either show real values or `N/A`)
   - Includes a "Found N errors in the last 24h" total count line above the table
   - Includes a "Showing X most recent:" suffix when the underlying tool response had `truncated=true`
   - Includes a "Full trace IDs:" footer line below the table when any row has a non-null trace ID
   - Renders trace IDs in the table column as `XXXXXXXX…` (8 chars + ellipsis) when present, never as the full UUID
   - Does NOT include any rows logged at SeverityLevel 2 (Warning) or below

   The verification is performed manually by the user, with the user reporting back whether the output meets all criteria. There is no automated assertion.

If any gate fails, the phase is not done.
