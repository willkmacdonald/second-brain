# Phase 19.2: Transaction-First Spine - Context

**Gathered:** 2026-04-18
**Status:** Ready for planning
**Source:** Will's own phase proposal (verbatim), treated as locked decisions

<domain>
## Phase Boundary

The spine currently delivers heartbeat + coarse health cleanly, but is not usable for diagnosing a broken user flow. The segment detail pages jump straight to native infrastructure telemetry (App Insights requests/exceptions, Cosmos diagnostics, Sentry events) which mostly shows GETs, Reads, Queries, and health noise.

This phase changes the primary drill-down model so the first thing an operator sees when drilling into a segment is **transactional evidence** (what real operations touched this segment, what happened, what failed), not infrastructure shapes.

**In scope:**
- Investigation spike to diagnose why 8 of 9 segments emit zero workload events today
- Backend read paths for transaction events by segment and by correlation id
- Segment page redesign: ledger-first, native telemetry moves below
- Transaction page: full cross-segment path for one correlation id, with explicit gap callouts ("backend_api seen, classifier missing")
- Preserve all existing native diagnostics panels — they become the *next* drill-down after the transaction path, not the first one

**Out of scope:**
- New evaluator rules, new status tiers, or changes to the status board itself
- Rewriting the spine storage schema (events already land in `spine_events`)
- Any mobile-UI work beyond what's needed for mobile_capture/mobile_ui emit parity

</domain>

<decisions>
## Implementation Decisions (LOCKED)

### Operator Workflow (the 4 drill-downs, in order)

1. **Status board** answers: *Where is the platform breaking down?*
2. **Segment page** answers: *What transactions touched this segment, and what happened or failed to happen?*
3. **Transaction page** answers: *Show me the full path across segments.*
4. **Native diagnostics** answers: *Give me root-cause evidence (App Insights requests/exceptions, Cosmos diagnostics, Sentry events, heartbeat history).*

The product must support drill-downs in this order. Native telemetry must not surface before step 2.

### Investigation Spike (non-negotiable first step)

Before any emitter fixes or UI work, an investigation spike must:
- Reproduce one real capture end-to-end against the deployed system
- Verify which segments are actually writing workload events into `spine_events`
- Confirm where correlation is missing or incomplete
- Produce a short memo distinguishing three categories:
  - **broken emitters** — code exists but events don't land
  - **segments that are native/pull by design** — no workload events expected, ever
  - **native telemetry that lacks correlation tags** — events exist somewhere but can't be joined by correlation_id

The memo drives the subsequent emitter-fix plan's scope. Do not plan emitter fixes speculatively before the spike concludes.

### Transaction Ledger Read Model

New API read paths:
- Recent transaction events by segment (not aggregated, the raw ledger)
- Full event stream for one correlation id

Each event must include: stage (segment_id), timestamp, duration, outcome, correlation id, and a small payload summary.

### Segment Page Redesign

- Keep current health summary at the top (unchanged)
- Add a **ledger-first section** immediately below: recent transactions that touched this segment
- Where possible, show **missing expected stages** — if a capture touched backend_api but never reached classifier, that gap must be visible on the classifier segment page or the transaction page
- Move current native telemetry panels (App Insights, Cosmos, Sentry) BELOW the ledger section as diagnostics

### Transaction Page

- One URL per correlation id, showing the full cross-segment path
- Gaps must be explicit: "backend_api seen, classifier missing" (not implicit absence)
- Each segment row links to the native diagnostics for that segment during that time window

### Native Diagnostics

- Preserve heartbeat, infra errors, mobile errors, App Insights, Cosmos, Sentry detail
- Re-position: they are the drill-down AFTER the transaction path, not the first drill-down

### Why This Phase Ships As One Unit

- Fixing emitters without a ledger leaves the new data invisible to operators
- Building a ledger without fixing emitters leaves the ledger permanently empty
- Both must ship together for the system to become operationally useful

### Known architecture limits (NOT bugs, don't confuse with emitter fixes)

- `mobile_capture` only forwards `crud_failure` events into spine today. Normal mobile capture traces are not expected in `spine_events`. If the phase decides mobile captures SHOULD produce transaction rows, that's a new push path, not a bug fix.
- `backend_api`, `external_services`, `container_app` all share the same App Insights adapter. The segment pages for those three show generic HTTP shapes by design. The ledger section must be additive — it does not replace the native panel for those segments.

### Existing read API surface (reuse where possible)

- `backend/src/second_brain/spine/storage.py:140` — `get_recent_events()` already reads by segment+time
- `backend/src/second_brain/spine/api.py` — existing FastAPI router where new endpoints belong
- `GET /api/spine/status` — existing status board endpoint
- `GET /api/spine/segment/{id}` — existing segment detail endpoint (currently native-telemetry-first; this phase inverts it)
- `GET /api/spine/correlation/{kind}/{id}` — existing correlation endpoint; extend or add a sibling for the raw event stream
