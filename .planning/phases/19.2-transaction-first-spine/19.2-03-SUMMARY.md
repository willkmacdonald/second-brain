## Decisions Made

- **LEDGER_EXPECTED_CHAINS starts from EXPECTED_CHAINS unchanged.** SPIKE-MEMO §4 already pre-approved `mobile_capture` as transactional (Option B post-capture emit), which matches the raw audit registry's `required=True` flag for `mobile_capture` in the capture chain. No override was needed. If a future memo revision downgrades `mobile_capture` to native-only, the override would apply in `LEDGER_EXPECTED_CHAINS` — `SEGMENT_LEDGER_METADATA["mobile_capture"]` would need to flip to `native_only` in tandem. Documented in the module docstring.
- **`mobile_ui` added to SEGMENT_LEDGER_METADATA as native_only.** The plan only called out cosmos and container_app explicitly, but memo §4b pins `mobile_ui` as native-only for the capture chain (crud-failure-only emit design). Added it so the UI renders the correct empty-state text instead of "no rows found."
- **Query(...) not Field(...) on route parameters.** Plan called this out explicitly; used `Query(..., ge=60, le=86400)` for window_seconds and `Query(..., ge=1, le=200)` for limit. Verified by an explicit 422 test.
- **Full Write used for api.py** instead of stepwise Edits for imports + route handlers. MEMORY.md Phase 17.1 lesson: ruff auto-format hook strips unused imports between Edits. A single Write lands imports + usage atomically.
- **Storage tests use MagicMock(return_value=async_iter()) or MagicMock(side_effect=fake_query).** Matches existing test_spine_storage conventions. For tests that need to inspect the SQL WHERE clause (transaction/noise boundary test), the `side_effect=fake_query` pattern captures `query` and `parameters` for assertion without breaking the iterator protocol.
- **Correlation enrichment keyed by (segment_id, timestamp).** Two queries run unbounded within their respective cutoffs; the Python-side join uses timestamp equality. This matches the existing correlation upsert's `timestamp: inner.timestamp.isoformat()` contract so equality is exact for the same event. Operational risk: if Cosmos ever produces a timestamp with different microsecond precision between the raw event write and the correlation upsert read, the enrichment would fall through — but both go through the same `inner.timestamp.isoformat()` call in `record_event`, so precision is identical.

## TypeScript types to add in Plan 04 (`web/lib/types.ts`)

Copy verbatim from the Python Pydantic shapes and translate:

```typescript
// Phase 19.2 — Transaction ledger types

export type SegmentLedgerMode = "transactional" | "native_only";

export type WorkloadOutcome = "success" | "failure" | "degraded";

export interface TransactionLedgerRow {
  segment_id: string;
  timestamp: string;           // ISO datetime
  operation: string;
  outcome: WorkloadOutcome;
  duration_ms: number;
  correlation_kind: CorrelationKind | null;   // nullable
  correlation_id: string | null;
  error_class: string | null;
}

export interface SegmentLedgerResponse {
  segment_id: string;
  mode: SegmentLedgerMode;
  empty_state_reason: string | null;
  rows: TransactionLedgerRow[];
  envelope: ResponseEnvelope;
}

export interface TransactionEvent {
  segment_id: string;
  timestamp: string;
  status: SegmentStatus;
  operation: string | null;          // null when raw event lookup falls through
  outcome: WorkloadOutcome | null;
  duration_ms: number | null;
  error_class: string | null;
  headline: string;
}

export interface TransactionPathResponse {
  correlation_kind: CorrelationKind;
  correlation_id: string;
  events: TransactionEvent[];
  missing_required: string[];         // sorted; segments in the chain that did NOT appear
  present_optional: string[];         // sorted; optional segments that DID appear
  unexpected: string[];               // sorted; segments that appeared but are NOT in the chain
  envelope: ResponseEnvelope;
}
```

**UI rendering notes for Plan 04:**
- `SegmentLedgerResponse.mode === "native_only"` + empty `rows` → render `empty_state_reason` verbatim in the ledger section above the native telemetry panel.
- `SegmentLedgerResponse.mode === "transactional"` + empty `rows` → render generic "No recent transactions in the last {window_seconds}s" message (no `empty_state_reason` available for transactional segments).
- `TransactionPathResponse.missing_required` is the PRIMARY gap signal — render each as "expected but did not appear" badges, deep-linking to the segment page for the missing segment if it has one.
- `unexpected` is informational — render as "segment appeared but is NOT in the capture chain" (helpful for detecting new emit sites).
