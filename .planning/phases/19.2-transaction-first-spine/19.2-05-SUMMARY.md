## Decisions Made

- **Three callouts ordered by severity.** `missing_required` (red, urgent — "something the chain requires did not fire") renders FIRST because it's the headline signal. `unexpected` (yellow, warning — "something fired that isn't supposed to be in this chain") renders SECOND because it's informational but actionable. `present_optional` (muted grey single line) renders THIRD because it's passive confirmation. If all three arrays are empty, nothing renders — the plan was explicit: "do NOT render a placeholder — show nothing."
- **OUTCOME_COLOR kept local to this page, not extracted to a shared module.** `WorkloadOutcome` ("success" | "failure" | "degraded") is a different union from `SegmentStatus` ("green" | "yellow" | "red" | "stale"). Sharing a color module would require a mapping adapter that adds indirection without clarity. Both pages use identical hex values for the overlapping colors, which keeps the visual language consistent.
- **`!event.operation && event.headline` fallthrough retained.** Plan 03's TransactionPathResponse has `operation: string | null` specifically because the two-query enrichment (spine_correlation for which-segments, spine_events for operation/duration) uses a `(segment_id, timestamp)` join that can miss if timestamps drift at Cosmos microsecond precision. When `operation` is null, the `headline` from the correlation query is still available and renders as a fallback line — preserving the pre-19.2 "something" in the worst case.
- **Title renamed "Correlation Timeline" → "Transaction Path".** The plan's `<objective>` and CONTEXT.md both use "transaction" language; "Correlation Timeline" was the pre-19.2 name that reflected only the spine_correlation read. The new name matches the enriched response type (`TransactionPathResponse`) and the semantic pivot from "when did segments tick" to "what did the chain DO for this one trace."
- **Single Write used for the whole-file rewrite.** MEMORY.md Phase 17.1 lesson: ruff auto-format strips unused imports between edits. Dropping `CorrelationResponse` and adding `TransactionPathResponse` had to land together with the `spine.transactionPath` call site in one atomic write. Stepwise Edits would have risked an intermediate state where `TransactionPathResponse` was imported-but-unused (auto-format strips it) OR `spine.transactionPath` was called-but-untyped (tsc fails).
- **Graceful-degradation try/catch preserved.** Same pattern Plan 04 used on the `LedgerSection` fetch in `web/app/segment/[id]/page.tsx`. During the integrated release rollout window, the backend route and the web build may not go live at exactly the same moment; the page must render a purposeful empty-state message ("No data available for this correlation") rather than a blank 500.
