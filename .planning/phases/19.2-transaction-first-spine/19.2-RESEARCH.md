## User Constraints (from CONTEXT.md)

### Locked Decisions

**Operator Workflow (the 4 drill-downs, in order):**

1. **Status board** answers: *Where is the platform breaking down?*
2. **Segment page** answers: *What transactions touched this segment, and what happened or failed to happen?*
3. **Transaction page** answers: *Show me the full path across segments.*
4. **Native diagnostics** answers: *Give me root-cause evidence (App Insights requests/exceptions, Cosmos diagnostics, Sentry events, heartbeat history).*

The product MUST support drill-downs in this order. Native telemetry must NOT surface before step 2.

**Investigation Spike (non-negotiable first step):**

Before any emitter fixes or UI work, an investigation spike must:
- Reproduce one real capture end-to-end against the deployed system
- Verify which segments are actually writing workload events into `spine_events`
- Confirm where correlation is missing or incomplete
- Produce a short memo distinguishing three categories:
  - **broken emitters** — code exists but events don't land
  - **segments that are native/pull by design** — no workload events expected, ever
  - **native telemetry that lacks correlation tags** — events exist somewhere but can't be joined by correlation_id

The memo drives the subsequent emitter-fix plan's scope. Do not plan emitter fixes speculatively before the spike concludes.

**Transaction Ledger Read Model:**
- New API read paths: recent transaction events by segment (raw ledger, not aggregated), and full event stream for one correlation id
- Each event must include: stage (segment_id), timestamp, duration, outcome, correlation id, and a small payload summary

**Segment Page Redesign:**
- Keep current health summary at the top (unchanged)
- Add a **ledger-first section** immediately below: recent transactions that touched this segment
- Where possible, show **missing expected stages** — if a capture touched backend_api but never reached classifier, that gap must be visible on the classifier segment page or the transaction page
- Move current native telemetry panels (App Insights, Cosmos, Sentry) BELOW the ledger section as diagnostics

**Transaction Page:**
- One URL per correlation id, showing the full cross-segment path
- Gaps must be explicit: "backend_api seen, classifier missing" (not implicit absence)
- Each segment row links to the native diagnostics for that segment during that time window

**Native Diagnostics:**
- Preserve heartbeat, infra errors, mobile errors, App Insights, Cosmos, Sentry detail
- Re-position: they are the drill-down AFTER the transaction path, not the first drill-down

**Why This Phase Ships As One Unit:**
- Fixing emitters without a ledger leaves the new data invisible to operators
- Building a ledger without fixing emitters leaves the ledger permanently empty
- Both must ship together for the system to become operationally useful

## Summary

This is a two-headed phase: **(A)** diagnose and fix the emitter gap that leaves 8 of 9 segments with zero workload events in `spine_events`, and **(B)** invert the segment-detail page so transactional evidence from the spine ledger renders BEFORE the existing native-telemetry panels, with a companion transaction page showing the full cross-segment path for one correlation id.

The spine data model is already correct — `IngestEvent` (liveness/readiness/workload discriminated union), `SpineRepository.record_event` writing to both `spine_events` and `spine_correlation` on workload-with-correlation, `get_recent_events(segment_id, window_seconds)` for raw per-segment reads, and `get_correlation_events(kind, id)` for cross-segment timeline reads — all exist and are tested. The correlation container already upserts a row per (correlation_id, segment_id, event) tuple, which gives us the full ledger for free. The phase does NOT need to change the storage schema or the evaluator.

The UI gap is clearer: `web/app/segment/[id]/page.tsx` dispatches immediately to native-telemetry renderers based on `data.schema`, with zero spine-ledger surface. `web/app/correlation/[kind]/[id]/page.tsx` renders a simple CorrelationResponse timeline, but the response shape (`CorrelationEvent { segment_id, timestamp, status, headline }`) lacks the duration/outcome/payload summary CONTEXT.md requires, and the page has no concept of "expected but missing" segments — a capability that already exists server-side in `spine/audit/chains.py` (`EXPECTED_CHAINS`) and `spine/audit/walker.py` (`missing_required`).

**Primary recommendation:** Plan breakdown should be: (1) investigation spike memo (non-code, read-only against deployed), (2) emitter fixes driven by the memo, (3) new ledger read-API endpoints + updated CorrelationEvent payload, (4) web segment-page ledger section + renderers, (5) web transaction-page redesign consuming the expected-chain info. Reuse `SpineRepository.get_recent_events` and `get_correlation_events`; reuse `EXPECTED_CHAINS` for the "missing stages" callout. Do not hand-roll a new ledger container or a new chain registry.

### Pattern 1: Spine read method + FastAPI route
**What:** Add a new read method on `SpineRepository`, expose via `build_spine_router` in `api.py`.
**When to use:** Any new ledger read path.
**Example (verified against existing code):**
```python
# backend/src/second_brain/spine/storage.py — append alongside get_recent_events
async def get_recent_workload_events(
    self,
    segment_id: str,
    window_seconds: int,
    limit: int = 50,
) -> list[dict[str, Any]]:
    """Return workload-type events for a segment, newest-first."""
    cutoff = (datetime.now(UTC) - timedelta(seconds=window_seconds)).isoformat()
    results: list[dict[str, Any]] = []
    async for item in self._events.query_items(
        query=(
            "SELECT * FROM c"
            " WHERE c.segment_id = @sid"
            " AND c.timestamp >= @cutoff"
            " AND c.event_type = 'workload'"
            " ORDER BY c.timestamp DESC"
        ),
        parameters=[
            {"name": "@sid", "value": segment_id},
            {"name": "@cutoff", "value": cutoff},
        ],
    ):
        results.append(item)
        if len(results) >= limit:
            break
    return results
```

### Pattern 2: Expected-chain awareness on correlation read
**What:** Join the correlation records against `EXPECTED_CHAINS[kind]` and surface `missing_required` / `present_optional` / `unexpected` sets.
**When to use:** The transaction page and any "missing stages" callout.
**Example (pattern already exists in walker.py; copy the set math):**
```python
from second_brain.spine.audit.chains import EXPECTED_CHAINS

records = await repo.get_correlation_events(kind, correlation_id)
segments_seen = {r["segment_id"] for r in records}

chain = EXPECTED_CHAINS[kind]
required = {s.segment_id for s in chain if s.required}
optional = {s.segment_id for s in chain if not s.required}
chain_ids = required | optional

missing_required = sorted(required - segments_seen)
present_optional = sorted(optional & segments_seen)
unexpected = sorted(segments_seen - chain_ids)
```

### Anti-Patterns to Avoid

- **Changing the evaluator or its inputs.** The evaluator reads workload events to compute red/yellow/green; changing emitters can affect status rollups. Make sure new emit sites produce events with correct outcome classification (don't emit `failure` where `degraded` is appropriate, and vice versa). CONTEXT.md explicitly rules out evaluator changes but does not rule out "more emit sites feeding the evaluator."
- **Touching native-telemetry adapters.** `backend_api`, `external_services`, `container_app`, `cosmos`, `sentry`, `foundry_agent`, `mobile_telemetry` adapters are stable. The ledger-first UX is ADDITIVE — the native renderer still runs below.
- **Breaking the discriminator.** `IngestEvent` is a `RootModel[Annotated[... | ..., Field(discriminator="event_type")]]`. New fields on `WorkloadPayload` must be Optional with a default or the discriminator breaks parsing of existing events in Cosmos.
- **Adding new correlation kinds.** `CorrelationKind = Literal["capture", "thread", "request", "crud"]` is referenced across the audit chain registry, middleware, mobile telemetry, and 4+ tests. Don't introduce new kinds in this phase — the ledger works with what's there.
- **Rolling a new adapter pattern for the ledger.** The ledger lives ABOVE the adapter layer. Adapters are for native-shape detail; the ledger is raw spine data. Keep them separate.
- **Querying Cosmos without a cutoff.** All `query_items` in `storage.py` use a time cutoff. Unbounded queries across `spine_events` (25k+ rows/day for backend_api alone) will RU-throttle.

## Common Pitfalls

### Pitfall 1: Confusing "no emit" with "no correlation"
**What goes wrong:** The audit tool's first run (documented in `project_followup_audit_first_findings.md`, 2026-04-18) flagged backend_api with an `instrumentation_warning` meaning spine recorded events but native App Insights spans weren't tagged with `capture_trace_id`. The spike MUST distinguish:
- **Emit missing entirely:** `spine_events` has zero workload rows for the segment on a real capture
- **Emit present but uncorrelated:** `spine_events` has rows but with `correlation_kind = None` (so they never hit `spine_correlation`)
- **Emit present and correlated, but native telemetry lost the tag:** `spine_correlation` has the segment but App Insights / Cosmos / Sentry records don't carry the trace id

All three look the same from the status board. Only the spike distinguishes them.

**Why it happens:** Three different code paths (agent_emitter, middleware, recipe tool) and three different native-tagging paths (App Insights OTel context, Cosmos request-id headers, Sentry tags).

**How to avoid:** The spike memo MUST bucket findings into the three categories CONTEXT.md specifies. Don't let the planner write emitter-fix tasks until the memo arrives.

**Warning signs:** A "broken emitter" verdict for a segment that has a clean `emit_agent_workload` call at its completion site — that's not a broken emitter, that's a correlation-plumbing gap and the fix is different (usually `capture_trace_id` not being threaded from the originating request).

### Pitfall 2: Recipe tool emits workload without correlation
**What goes wrong:** `backend/src/second_brain/tools/recipe.py:175-185` emits `_WorkloadEvent(segment_id="external_services", ...)` but the `WorkloadPayload` does not carry `correlation_kind` or `correlation_id`. That event lands in `spine_events` but NEVER in `spine_correlation`. From the status board's per-segment view it reads as "external_services is alive and has workload." From the transaction page's per-correlation view it reads as "external_services never ran for this capture." Both are technically true. This is a pattern the spike will likely confirm for more than one segment.
**Why it happens:** `emit_agent_workload` threads `capture_trace_id` through to `correlation_kind="capture"`. Direct `_WorkloadEvent` construction (as recipe.py does) skips that helper and loses the tag unless the caller sets it explicitly.
**How to avoid:** Prefer `emit_agent_workload` over direct `_WorkloadEvent` construction. If direct construction is needed, set `correlation_kind` and `correlation_id` on the `WorkloadPayload` explicitly.
**Warning signs:** A segment appears in `spine_events` segment summary but never in a `spine_correlation` walk for any capture.

### Pitfall 3: Mobile segments are Sentry-sourced, not spine-sourced (by design)
**What goes wrong:** Planner writes a task "add normal mobile_capture workload emission so captures appear in the ledger" — but mobile_capture today is deliberately crud-failure-only (`telemetry.py:86-110`). Adding a normal-capture emit path is a design decision, not a bug fix. CONTEXT.md says "deferred: mobile app UI changes unless mobile emit parity requires otherwise" — implying the phase SHOULD consider whether a push path is justified, but treats this as one of the spike's decisions, not a foregone conclusion.
**Why it happens:** Audit walker's `_APPINSIGHTS_SEGMENTS` set documents the architectural split: `mobile_capture` and `mobile_ui` are Sentry-only. The same logic applies to the ledger.
**How to avoid:** The spike memo must decide: do normal mobile captures belong in the ledger? If yes, the phase includes a new push path from the mobile app. If no, the ledger UI must explicitly render "mobile segments have no workload events by design — see Sentry for native telemetry" rather than implying emit is broken.
**Warning signs:** Planning tasks that say "emit normal mobile capture events" without the spike having concluded it's warranted.

### Pitfall 4: `CorrelationEvent` shape is under-specified for the ledger
**What goes wrong:** `CorrelationResponse.events` today carries `CorrelationEvent(segment_id, timestamp, status, headline)` — no duration, no outcome (headline is free-text), no operation name, no payload summary. CONTEXT.md requires "stage, timestamp, duration, outcome, correlation id, and a small payload summary." The existing `spine_correlation` container rows DO carry `status` (derived from outcome) and a `headline` synthesized from `operation + outcome + error_class` — but not the raw duration_ms. Duration lives in the raw `spine_events` row.
**Why it happens:** The correlation container was designed for status rollups, not transaction ledgers. The ledger needs to JOIN correlation rows against raw events (or extend the correlation upsert to carry the extra fields).
**How to avoid:** Choose one approach intentionally:
- **Option A (no schema change):** Read `spine_correlation` for the "which segments" list, then read `spine_events` filtered to `correlation_id` to get duration/operation. Two queries per transaction page. Safer.
- **Option B (additive schema change):** Extend the correlation upsert in `SpineRepository.record_event` to also write `duration_ms` and `operation` onto the correlation row. Forward-only; old rows don't have these fields; UI must handle missing-field case gracefully.

Option A is the lower-risk choice. Document the choice in the plan.
**Warning signs:** A plan that promises "rich transaction ledger" without specifying which container supplies duration_ms.

### Pitfall 5: CorrelationResponse ordering assumption
**What goes wrong:** `get_correlation_events` sorts by `timestamp` ASC. The transaction page renders "happened first" to "happened last" — the current sort is correct. BUT `get_recent_workload_events` for the segment ledger should be DESC (newest first). Mixing directions silently will render lists upside-down.
**Why it happens:** Two different use cases: cross-segment timeline = chronological; per-segment recent list = reverse-chronological.
**How to avoid:** Explicit `ORDER BY c.timestamp DESC` in the ledger query (see Pattern 1 above), and a test that asserts the order.
**Warning signs:** Tests missing assertions about ordering, UI showing oldest-first by default.

### Pitfall 6: ruff auto-format strips unused imports mid-task
**What goes wrong:** MEMORY.md documents that when adding new imports (e.g., extending a Pydantic model with `RootModel` or `Field`), the auto-format hook strips them before the next edit can use them. This bit Phase 17.1 execution.
**Why it happens:** PostToolUse auto-format runs `ruff --fix` which removes unused imports.
**How to avoid:** Add import + usage in a single `Write` (full-file replacement) OR make the import addition the LAST edit after usage is already in place.
**Warning signs:** Any plan task that adds "import X" as step 1 and "use X" as step 2 with separate Edit calls.

```python
        async for event in inner:
            yield event
    except Exception as exc:
        outcome = "failure"
        error_class = type(exc).__name__
        raise
    finally:
        duration_ms = int((time.perf_counter() - start) * 1000)
        await emit_agent_workload(
            repo=repo,
            segment_id=segment_id,
            operation=operation,
            outcome=outcome,
            duration_ms=duration_ms,
            capture_trace_id=capture_trace_id,  # <-- threads correlation
            run_id=run_id,
            thread_id=thread_id,
            error_class=error_class,
        )
```

## Open Questions

1. **Correlation row enrichment vs. two-query ledger read**
   - What we know: `spine_correlation` has `segment_id/timestamp/status/headline`; `spine_events` has the raw duration/operation. CONTEXT.md requires both surfaced.
   - What's unclear: Whether to extend `SpineRepository.record_event` to write `duration_ms` + `operation` into the correlation upsert (forward-only) OR to join in the read path (no schema change).
   - Recommendation: Default to the two-query read path for plan 01. Only add schema enrichment if query latency on the transaction page exceeds ~500ms consistently.

2. **Should normal mobile captures become a push path?**
   - What we know: Today `mobile_capture` spine coverage is `crud_failure`-only by design.
   - What's unclear: Whether the ledger UX demands that every capture shows a `mobile_capture` row (CONTEXT.md's "mobile_capture seen, classifier missing" example implies yes).
   - Recommendation: The spike MUST answer. Don't prejudge.

3. **Pagination cursor or simple limit for ledger reads?**
   - What we know: ResponseEnvelope already has an unused `cursor` field.
   - What's unclear: Whether this phase needs real pagination.
   - Recommendation: Simple `limit=50` parameter for v1. Cursor is infrastructure for a later phase.

4. **"Missing expected stages" callout placement**
   - What we know: CONTEXT says "where possible, show missing expected stages" on the segment page.
   - What's unclear: Whether the segment page per se shows missing stages, or only the transaction page does. Logic is easier on the transaction page (one trace, clear chain). On the segment page, "missing" is harder to define (missing from WHICH trace?).
   - Recommendation: Transaction page shows explicit gaps. Segment page shows a "recent transactions with incomplete chains" badge instead.

5. **Cache strategy for the correlation-id query path**
   - What we know: Next App Router uses `force-dynamic` today; no cache.
   - What's unclear: Whether a memory cache on the server side is useful for the same correlation_id fetched repeatedly.
   - Recommendation: Defer. Single-user app, traffic is tiny.
