## Decisions Made

- **Ledger fetch is non-fatal.** Wrapped in try/catch with a `console.warn`. Rationale: during the integrated release, Plan 04's web build may deploy before Plan 03's backend build is fully live, or vice versa. The segment page must not crash during that window — it falls through to the native-telemetry renderer, which is the pre-19.2 behavior. Matches user preference "best, most robust solution" over "simplest."
- **Two empty states live inside `LedgerSection`, not the page.** The page just passes `ledger.mode` + `ledger.empty_state_reason` through. This keeps the branch logic in one file and means adding a third empty-state type later (e.g. `batch_only`) is a single-file change.
- **`correlation_id` truncated to 12 chars with an ellipsis.** Capture trace ids are 16–20 chars (UUIDv4-ish via `secrets.token_hex`). Showing the first 12 keeps rows visually scannable; the full id is revealed on click via the `href` attribute. Short ids (< 13 chars) render in full.
- **Outcome column color map matches correlation page `STATUS_COLOR` semantics** — green/orange/red for success/degraded/failure. Kept local to LedgerSection rather than sharing a module, because the status values are distinct (`WorkloadOutcome` vs `SegmentStatus`); sharing would require a union-type adapter.
- **Error column renders as `#b33b3b` when present, `#555` when `null`** — not as an empty cell. Reinforces that "no error" is a positive signal, not missing data. Consistent with the em-dash used in the Correlation column when the row has no correlation id.
- **Single `Write` on `page.tsx` for Task 4.** Followed the plan's `<tdd_reminder>` guidance and MEMORY.md Phase 17.1 lesson: import addition + first usage done in one file write to prevent auto-format from stripping `SegmentLedgerResponse` or `LedgerSection` between edits.

### Additions beyond the plan's explicit list

**1. [Rule 2 - Missing critical] "Segment: {segmentId}" footer line inside `<LedgerSection />`**

- **Found during:** Task 3 (drafting LedgerSection)
- **Issue:** The plan listed six columns for the non-empty table but said nothing about echoing the segment id in the section. Without it, a ledger row rendered on, say, `/segment/admin` with a `classifier` row (which can happen during a cross-segment capture) would visually misattribute — the operator would read the row as an `admin` event. The `segment_id` column in the table solves this when there are rows; the footer line provides a "you are here" anchor in the empty state too.
- **Fix:** Rendered `Segment: <code>{segmentId}</code>` in a muted `#555` footer line inside the component, outside the rows table.
- **Files modified:** `web/components/LedgerSection.tsx`
- **Verification:** `tsc --noEmit` clean; visual inspection pending at checkpoint.
