NO Inbox-doc persistence for Investigation (different from Classifier which DOES persist — see 24-16). GA streaming pattern with Option A — explicit message history (per probe 1 + P0-1 OUTCOME):
```python
from agent_framework import Agent, Message
import uuid

# Build messages from history + new user turn.
# history is a list of dicts: [{"role": "user"|"assistant", "content": str}, ...]
msg_list = [Message(role=t["role"], contents=[t["content"]]) for t in (history or [])]
msg_list.append(Message(role="user", contents=[question]))

stream = agent.run(msg_list, stream=True)

async for update in stream:
    # update.role == 'assistant' always
    # update.text — accumulated user-visible text (empty during tool execution)
    # update.contents — list of Content objects (tool calls, tool results, text)
    # update.finish_reason — always None on streaming deltas
    text_delta = getattr(update, "text", "")
    if text_delta:
        yield encode_sse({"type": "text", "content": text_delta})
    for content in getattr(update, "contents", None) or []:
        # Existing content.type matching for function_call / function_result
        ...

# After stream completes — emit a fresh UUID; mobile expects thread_id but doesn't introspect it.
yield encode_sse({"type": "done", "thread_id": str(uuid.uuid4())})
```

Why this design (P0-1 OUTCOME, see `session_rehydration_fresh_process.json`):
- Original RC contract used `options["conversation_id"] = thread_id` for server-side thread continuity.
- GA SDK 1.3.0 removed the equivalent; `AgentSession.session_id` is a CLIENT-GENERATED UUID, not a server pointer.
- Cross-process probe attempted `AgentSession(session_id=prior_uuid)` and asked the agent to recall the prior turn — failed (`recalled_pineapple=false`).
- Locked design: stateless agent invocation. The caller owns the message history and passes it explicitly on every turn.
- For Investigation specifically: mobile already holds visible chat history (chat bubbles on screen). Mobile passes that history with each new turn; the backend is a thin pass-through. No Inbox-doc persistence — Investigation is a transient agent session, not a captured artifact.

Existing SSE wire format (KEEP unchanged):
- `{"type": "text", "content": ...}` for text deltas
- `{"type": "tool_call", "tool": ..., "args": ...}` for function calls
- `{"type": "tool_result", "tool": ..., "result": ...}` for function results
- `{"type": "error", "message": ...}` for errors
- `{"type": "done", "thread_id": <fresh-uuid>}` for completion — fresh UUID, no server meaning

Custom span deletion (F-15):
The current code has `with tracer.start_as_current_span("investigate") as span:` wrapping the entire stream processing. Delete this `with` block. The framework's `invoke_agent` span (auto-emitted) is now the canonical agent span; capture-trace middleware tags it at source. The `investigate.question_length` and `investigate.thread_id` attributes move onto AppRequests via the api endpoint (mirror `api/capture.py:228` pattern) — the endpoint sets the attributes before calling into the adapter generator.

api/investigate.py request body extension:
```python
# Pydantic request model — mobile sends visible chat history with each follow-up.
class InvestigateRequest(BaseModel):
    question: str
    thread_id: str | None = None  # backward compat — accepted but no longer used server-side
    history: list[dict[Literal["role", "content"], str]] | None = None
```
Exact Pydantic shape can be adapted by the executor to fit existing request-model conventions; what matters is that `history` is extracted and threaded through to `stream_investigation`.
