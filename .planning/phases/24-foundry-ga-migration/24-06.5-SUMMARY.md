## Deliverables

### 1. Probe harness extension (`backend/scripts/foundry_probe.py`)

Three new async entry points:

- **`session_rehydration_fresh_process_phase_a(state_file)`** — sub-probe. Creates `AgentSession()`, sends turn 1 (`"Remember the magic word PINEAPPLE for later."`), persists `session.session_id`, `service_session_id`, `magic_word`, `phase_a_run_id` to the state file, exits.
- **`session_rehydration_fresh_process_phase_b(state_file)`** — sub-probe. Loads `session_id` from state file, reconstructs `AgentSession(session_id=loaded_session_id)` (no Python-level state shared with phase A), sends turn 2 (`"What was the magic word I told you to remember?"`), persists `turn_two_text` and `phase_b_service_session_id`.
- **`session_rehydration_fresh_process()`** — driver. Spawns phase A then phase B as **separate** subprocesses via `subprocess.run([sys.executable, "-m", "scripts.foundry_probe", ...])`. Reads the final state file, computes `recalled_pineapple = magic_word.upper() in turn_two_text.upper()`, writes the fixture.

Argparse extended with `--state-file` (required by phase sub-probes). `_REQUIRES_STATE_FILE` frozenset validates correct usage. `all` mode skips the sub-probes (they only run via the driver). Probe-span tagging convention preserved (`probe.run_id` + `probe.name`; no `capture.trace_id`, so production KQL queries naturally exclude probe traffic).

## Critical finding — operator checkpoint required

**Cross-process `AgentSession` rehydration via `session_id` alone does NOT work on the GA Foundry SDK (1.3.0) against the deployed RC endpoint.**

### Evidence

1. **Two independent live runs**, ~50s apart, returned `recalled_pineapple=false`:
   - Run 1: turn 2 → `"You didn't share a magic word with me. Let me know if you have one you'd like me to remember!"`
   - Run 2: turn 2 → `"I don't have a record of a magic word from you yet. Could you remind me what it is?"`
   - Both responses semantically deny that any prior turn happened.

2. **`service_session_id` differs between phase A and phase B** in both runs (e.g. run 2: `resp_022e8e6...` for phase A vs `resp_07284da...` for phase B). The Foundry server assigned a NEW response/session ID to phase B even though phase B passed the persisted `session_id` UUID. This strongly suggests the SDK uses `session_id` only as a client-side correlator for an in-memory message accumulator — the server does not look up prior history from the client-generated UUID.

3. **`phase_a_exit_code=0` and `phase_b_exit_code=0`** in both runs. The probe ran cleanly. This is not auth, network, or environmental failure.

### What this invalidates

| Plan | Affected lines / claim |
|------|-----------------------|
| 24-07 | line 55 — "GA persists `session_id` and reconstructs `AgentSession(session_id=stored_id)` for follow-ups" |
| 24-16 | line 46 — capture handler reconstructs session from `sessionId` field on Inbox doc |
| 24-17 | line 39 — InboxDocument `sessionId` field replaces `foundryThreadId` and is the rehydration key |

Each of these plans assumes that storing only `AgentSession.session_id` on Cosmos is sufficient to recover conversation continuity in a NEW container instance / process. The fixture proves it isn't.

### Fallback options (per plan's decision branch)

The plan pre-specifies three operator-choosable paths:

**Option A — Persist full message history on Inbox doc**
Inbox doc grows a `conversationHistory: list[{role, content}]` field. On each follow-up, the capture handler reconstructs the agent's message accumulator from this list before calling `agent.run()`. The `session_id` (client UUID) is regenerated each turn (or dropped entirely). Pros: works with the actual SDK behavior; deterministic. Cons: Cosmos doc grows unbounded with conversation length; serialization edge cases for content blocks/tool calls.

**Option B — Serialize `AgentSession.state` + message accumulator**
Probe the SDK to find the internal message accumulator (likely `session.state` or `session._messages`). Serialize both on persist; deserialize and reattach on rehydrate. Pros: closer to RC's "thread is opaque" design. Cons: relies on SDK internals (`agent_framework._types.AgentSession.state` etc.) — fragile across SDK versions; the GA SDK's experimental warnings suggest internal shape may shift.

**Option C — Redesign follow-up flow within a single request handler**
Eliminate cross-process state. All follow-up turns live within the same FastAPI request (e.g. long-lived SSE stream). Pros: avoids the rehydration problem entirely. Cons: changes UX (mobile must hold the connection); breaks the current pattern where follow-ups arrive on independent HTTP requests; defeats fan-out across container instances.

### Recommendation

Based on the probe data, **Option A (full message history)** is the lowest-risk path forward:
- Works against current SDK behavior with zero internal-shape dependencies.
- Storage cost is bounded by typical capture follow-up depth (≤5 turns) — well within Cosmos doc size limits.
- Forward-compatible with future SDK changes (SDK could later support real cross-process rehydration without breaking the option-A code path).

Option B should be deferred until the GA SDK leaves experimental markers behind. Option C is a substantial UX redesign and is out of scope for Phase 24.
