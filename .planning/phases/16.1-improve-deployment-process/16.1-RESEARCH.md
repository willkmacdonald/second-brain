# Phase 16.1: Improve Deployment Process - Research

**Researched:** 2026-04-06
**Domain:** GitHub Actions CI/CD, Azure Container Apps revision management, uv dependency safety
**Confidence:** HIGH

## Summary

This phase hardens the existing `deploy-backend.yml` GitHub Actions workflow with four capabilities: dependency safety via `uv lock --check`, commit-correlated revision naming via `--revision-suffix`, post-deploy health verification with polling, old revision cleanup, and a structured job summary. All changes are confined to the single workflow YAML file -- no application code changes required.

The current workflow is minimal: checkout, Azure login, ACR login, Docker build+push, `az containerapp update`. It has no verification that the deploy succeeded, no lockfile consistency check, no revision naming, and no summary output. The health endpoint already exists at `/health` returning `{"status":"ok",...}` and is publicly accessible (no auth required).

**Primary recommendation:** Add `uv lock --check` as a pre-build gate, use `--revision-suffix` with a `sha-` prefix on `az containerapp update`, poll the health endpoint after deploy, deactivate stale revisions, and write a `$GITHUB_STEP_SUMMARY` block at the end.

## Architecture Patterns

### Current Workflow Structure
```
.github/workflows/deploy-backend.yml
```

Single job (`build-and-deploy`), single runner (`ubuntu-latest`). This should remain a single job since all steps are sequential and share the same Azure login context.

### Recommended Step Order
```
1. Checkout
2. Install uv (pinned 0.5.4)
3. uv lock --check (within backend/ directory)
4. Azure login (OIDC)
5. ACR login
6. Build and push image (with --revision-suffix-safe tag)
7. Deploy to Container Apps (with --revision-suffix)
8. Post-deploy verification (health poll loop)
9. Revision cleanup (deactivate old revisions)
10. Job summary (write $GITHUB_STEP_SUMMARY)
```

### Pattern 1: Health Poll Loop
**What:** Shell loop polling the health endpoint with backoff
**When to use:** After `az containerapp update` returns (Azure CLI returns before new revision is fully serving)

```bash
# Source: GitHub Actions best practice for health verification
HEALTH_URL="https://brain.willmacdonald.com/health"
MAX_ATTEMPTS=15  # 15 * 12s = 180s = 3 minutes
POLL_INTERVAL=12

for i in $(seq 1 $MAX_ATTEMPTS); do
  RESPONSE=$(curl -s -o /dev/null -w "%{http_code}" "$HEALTH_URL" 2>/dev/null || echo "000")
  if [ "$RESPONSE" = "200" ]; then
    BODY=$(curl -s "$HEALTH_URL")
    STATUS=$(echo "$BODY" | jq -r '.status')
    if [ "$STATUS" = "ok" ]; then
      echo "Health check passed on attempt $i"
      break
    fi
  fi
  if [ "$i" -eq "$MAX_ATTEMPTS" ]; then
    echo "::error::Health check failed after $MAX_ATTEMPTS attempts"
    exit 1
  fi
  echo "Attempt $i/$MAX_ATTEMPTS: status=$RESPONSE, waiting ${POLL_INTERVAL}s..."
  sleep $POLL_INTERVAL
done
```

### Pattern 2: Revision Suffix with Alpha Prefix
**What:** Prefix the 7-char git SHA with `sha-` to satisfy Azure naming rules
**When to use:** Always -- git SHAs can start with digits

```bash
# Revision suffix must start with alphabetic character (a-z)
# Git SHAs like "3a7b9c2" are fine, but "9f3a2b1" would fail
REVISION_SUFFIX="sha-$(echo $GITHUB_SHA | cut -c1-7)"

az containerapp update \
  --name $CONTAINER_APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --image "$IMAGE_TAG" \
  --revision-suffix "$REVISION_SUFFIX"
```

The full revision name becomes: `second-brain-api--sha-{7chars}`

### Pattern 3: Revision Cleanup
**What:** Deactivate all active revisions except the latest after successful verification
**When to use:** After health check passes

```bash
LATEST_REVISION="second-brain-api--${REVISION_SUFFIX}"

# List all active revisions
ACTIVE_REVISIONS=$(az containerapp revision list \
  --name $CONTAINER_APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --query "[?properties.active==\`true\`].name" \
  -o tsv)

for rev in $ACTIVE_REVISIONS; do
  if [ "$rev" != "$LATEST_REVISION" ]; then
    echo "Deactivating old revision: $rev"
    az containerapp revision deactivate \
      --revision "$rev" \
      --resource-group $RESOURCE_GROUP
  fi
done
```

### Pattern 4: Image SHA Verification
**What:** Confirm the deployed revision is running the expected image
**When to use:** After deploy, before declaring success

```bash
DEPLOYED_IMAGE=$(az containerapp revision show \
  --name $CONTAINER_APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --revision "$LATEST_REVISION" \
  --query "properties.template.containers[0].image" \
  -o tsv)

EXPECTED_IMAGE="${ACR_NAME}.azurecr.io/${IMAGE_NAME}:${GITHUB_SHA}"

if [ "$DEPLOYED_IMAGE" != "$EXPECTED_IMAGE" ]; then
  echo "::error::Image mismatch! Expected: $EXPECTED_IMAGE, Got: $DEPLOYED_IMAGE"
  exit 1
fi
```

### Pattern 5: Traffic Weight Verification
**What:** Confirm the latest revision has 100% traffic
**When to use:** After health check passes

```bash
TRAFFIC_WEIGHT=$(az containerapp revision show \
  --name $CONTAINER_APP_NAME \
  --resource-group $RESOURCE_GROUP \
  --revision "$LATEST_REVISION" \
  --query "properties.trafficWeight" \
  -o tsv)

# In single-revision mode, the latest revision should have 100% traffic
# Azure may show transient states during activation -- tolerate this
if [ "$TRAFFIC_WEIGHT" != "100" ]; then
  echo "::warning::Latest revision traffic weight is $TRAFFIC_WEIGHT%, expected 100%"
fi
```

### Pattern 6: Azure Portal URL Construction
**What:** Build a clickable URL to the Container App revisions page
**When to use:** In the job summary

```bash
# Azure Portal deep link format for Container App revisions
# Requires subscription ID for the full URL
SUBSCRIPTION_ID=$(az account show --query id -o tsv)
PORTAL_URL="https://portal.azure.com/#@willmacdonald.com/resource/subscriptions/${SUBSCRIPTION_ID}/resourceGroups/${RESOURCE_GROUP}/providers/Microsoft.App/containerApps/${CONTAINER_APP_NAME}/revisionManagement"
```

### Anti-Patterns to Avoid
- **Polling too aggressively:** Don't poll every 1-2 seconds -- Azure Container Apps takes time to provision revisions. 10-15s is the sweet spot.
- **Using `--no-wait` then checking status:** The `az containerapp update` command already waits for the update to be accepted. The health poll verifies the actual serving status.
- **Hardcoding subscription ID:** Use `az account show --query id` to get it dynamically from the logged-in session.
- **Using `az containerapp update` without `--revision-suffix`:** Without a suffix, Azure generates random names making git correlation impossible.

## Common Pitfalls

### Pitfall 1: Revision Suffix Starting with Digit
**What goes wrong:** `az containerapp update --revision-suffix "3a7b9c2"` fails because revision suffixes must start with an alphabetic character (a-z).
**Why it happens:** Git SHAs are hex strings -- they can start with 0-9 or a-f. A SHA like `9f3a2b1` starts with a digit.
**How to avoid:** Always prefix with `sha-` to guarantee alphabetic start: `sha-3a7b9c2`.
**Warning signs:** Intermittent deploy failures when the git SHA happens to start with a digit.
**Source:** [Azure CLI extension issue #6449](https://github.com/Azure/azure-cli-extensions/issues/6449), [Azure docs revision naming rules](https://learn.microsoft.com/azure/container-apps/revisions#customize-revisions)

### Pitfall 2: uv Version Mismatch Between CI and Dockerfile
**What goes wrong:** `uv lock --check` in CI passes, but `uv sync --frozen` in Dockerfile fails (or vice versa) because different uv versions produce different lockfile formats.
**Why it happens:** The lockfile format `revision` field (currently `revision = 3` in this project) is tied to uv versions. A newer uv may write `revision = 4`.
**How to avoid:** Pin uv to exactly 0.5.4 in CI using `astral-sh/setup-uv@v5` with `version: "0.5.4"`. This matches `COPY --from=ghcr.io/astral-sh/uv:0.5.4` in the Dockerfile.
**Warning signs:** Lockfile regeneration in CI that wasn't intended.

### Pitfall 3: Health Endpoint Returning 200 Before App Is Ready
**What goes wrong:** The old revision responds to the health check while the new revision is still starting.
**Why it happens:** In single revision mode, Azure Container Apps uses zero-downtime deployment -- the old revision keeps serving until the new one is ready. During this transition, `/health` returns the OLD revision's response.
**How to avoid:** After health check passes, verify the image SHA on the latest revision matches the expected SHA. This confirms the new revision is active, not the old one.
**Warning signs:** Health check passes immediately (within first poll) even though deploy just started.

### Pitfall 4: Revision Suffix Already Exists
**What goes wrong:** Re-running a failed deploy for the same commit fails because `--revision-suffix sha-abc1234` already created a revision with that name.
**Why it happens:** Revision names are unique within a Container App. Re-deploying the same SHA tries to create a duplicate.
**How to avoid:** This is expected behavior -- if the revision already exists, you may need to deactivate/delete it first or skip the suffix. For this project (single-developer, rare reruns), this is acceptable. If it becomes a problem, append a short run number: `sha-abc1234-42`.
**Warning signs:** `az containerapp update` returning a conflict error on re-runs.

### Pitfall 5: uv lock --check Needs to Run in backend/ Directory
**What goes wrong:** Running `uv lock --check` from the repo root fails because `pyproject.toml` and `uv.lock` are in `backend/`.
**Why it happens:** `uv lock` operates on the current directory's project.
**How to avoid:** Use `working-directory: backend` on the step, or pass `--directory backend` (if supported by the uv version).
**Warning signs:** "No `pyproject.toml` found" error in CI.

### Pitfall 6: Deactivating the Only Active Revision
**What goes wrong:** If the revision list filtering is wrong, you could deactivate the revision that's currently serving traffic.
**Why it happens:** Race condition or incorrect JMESPath query.
**How to avoid:** Always check `rev != LATEST_REVISION` before deactivating. Never deactivate all revisions -- always keep the latest.
**Warning signs:** 503 errors from the app after deploy "succeeds".

```yaml
    version: "0.5.4"

  run: uv lock --check
```

### Health Check Polling Step (Full)
```yaml
- name: Verify deployment health
  env:
    HEALTH_URL: https://brain.willmacdonald.com/health
    MAX_ATTEMPTS: 15
    POLL_INTERVAL: 12
  run: |
    for i in $(seq 1 $MAX_ATTEMPTS); do
      HTTP_CODE=$(curl -s -o /tmp/health.json -w "%{http_code}" "$HEALTH_URL" 2>/dev/null || echo "000")
      if [ "$HTTP_CODE" = "200" ]; then
        STATUS=$(jq -r '.status' /tmp/health.json)
        if [ "$STATUS" = "ok" ]; then
          echo "Health check passed on attempt $i"
          cat /tmp/health.json | jq .
          exit 0
        fi
        echo "Attempt $i/$MAX_ATTEMPTS: HTTP 200 but status=$STATUS"
      else
        echo "Attempt $i/$MAX_ATTEMPTS: HTTP $HTTP_CODE"
      fi
      if [ "$i" -eq "$MAX_ATTEMPTS" ]; then
        echo "::error::Health check failed after $MAX_ATTEMPTS attempts (${MAX_ATTEMPTS}x${POLL_INTERVAL}s)"
        exit 1
      fi
      sleep $POLL_INTERVAL
    done
```

### Revision Deactivation with Logging
```yaml
- name: Cleanup old revisions
  run: |
    ACTIVE_REVISIONS=$(az containerapp revision list \
      --name ${{ env.CONTAINER_APP_NAME }} \
      --resource-group ${{ env.RESOURCE_GROUP }} \
      --query "[?properties.active==\`true\`].name" \
      -o tsv)

    ACTIVE_COUNT=0
    for rev in $ACTIVE_REVISIONS; do
      ACTIVE_COUNT=$((ACTIVE_COUNT + 1))
    done

    echo "Found $ACTIVE_COUNT active revision(s)"

    for rev in $ACTIVE_REVISIONS; do
      if [ "$rev" != "$LATEST_REVISION" ]; then
        echo "Deactivating: $rev"
        az containerapp revision deactivate \
          --revision "$rev" \
          --resource-group ${{ env.RESOURCE_GROUP }}
      fi
    done
```

## Open Questions

1. **Revision suffix collision on re-deploy of same commit**
   - What we know: If a deploy fails and is re-run for the same commit, `--revision-suffix sha-abc1234` will conflict with the existing revision.
   - What's unclear: Whether `az containerapp update` handles this gracefully or errors out.
   - Recommendation: Accept the risk for now (single developer, rare reruns). If it becomes a problem, append `$GITHUB_RUN_NUMBER` to the suffix.

2. **Container App revision mode**
   - What we know: The app appears to be in single revision mode (default). In single mode, Azure auto-deactivates old revisions during zero-downtime deployment.
   - What's unclear: If the app is in single revision mode, explicit deactivation of old revisions may be redundant (Azure handles it). However, it's harmless and provides explicit cleanup.
   - Recommendation: Include the cleanup step regardless -- it's a safety net and provides logging.

3. **Azure Portal URL tenant domain**
   - What we know: The portal URL includes a tenant reference. The user's Azure account is `will@willmacdonald.com`.
   - What's unclear: Whether the tenant slug in the portal URL is exactly `willmacdonald.com` or requires the full tenant ID.
   - Recommendation: Construct the URL with `#view/Microsoft_App/ContainerAppBlade/~/revisionManagement/id/` pattern using the resource ID, which is tenant-agnostic.
