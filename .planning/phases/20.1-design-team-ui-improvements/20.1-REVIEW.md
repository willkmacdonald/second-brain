# Phase 20.1: Code Review Report

**Reviewed:** 2026-04-22T15:30:00Z
**Depth:** standard
**Files Reviewed:** 19
**Status:** issues_found

## Summary

Phase 20.1 is a visual reskin that introduces a centralized design token system (`theme.ts`), custom font loading, and consistent styling across all mobile screens. The implementation is solid overall -- the theme system is well-structured, font loading gates rendering correctly via SplashScreen, and all components consistently reference theme tokens instead of inline values.

No critical issues were found. Three warnings and five informational items are documented below. The warnings relate to hardcoded color values that bypass the theme system, a dead-code hook, and a `CapsLabel` type signature that restricts style composition.

## Warnings

### WR-01: Hardcoded colors in _layout.tsx bypass theme system

**File:** `mobile/app/_layout.tsx:67-86`
**Issue:** Three `Stack.Screen` options blocks hardcode `backgroundColor: "#0a0a12"` and `headerTintColor: "#f0f0f5"` instead of referencing `theme.colors.bg` and `theme.colors.text`. While these happen to match the current token values, they will silently drift if the theme tokens are ever updated. The `theme` import is not present in this file at all.
**Fix:**
```tsx
import { theme } from "../constants/theme";

// Then in each Stack.Screen options:
headerStyle: { backgroundColor: theme.colors.bg },
headerTintColor: theme.colors.text,
```

### WR-02: useTheme hook is defined but never imported anywhere

**File:** `mobile/hooks/useTheme.ts:1-5`
**Issue:** The `useTheme()` hook was created as part of the design token foundation (D-05) but zero components import or call it. Every component imports `theme` directly from `../constants/theme` instead. This is dead code that could mislead future contributors into thinking it provides additional functionality (e.g., reactive theme switching) when it does not.
**Fix:** Either remove the file entirely, or add a comment documenting it as a forward-looking hook for future dark/light theme switching. If removed, also update any planning docs that reference it.

### WR-03: CapsLabel style prop type prevents StyleProp composition

**File:** `mobile/components/CapsLabel.tsx:8`
**Issue:** The `style` prop is typed as `TextStyle` (a plain object) rather than `StyleProp<TextStyle>`. This means callers cannot pass arrays of styles or `undefined` without a type error. Currently all callers pass plain objects so this works at runtime, but it violates the standard React Native pattern and will cause a type error if anyone tries `style={[styles.label, condition && styles.extra]}`.
**Fix:**
```tsx
import { Text, StyleSheet, StyleProp, TextStyle } from "react-native";

interface CapsLabelProps {
  children: React.ReactNode;
  color?: string;
  size?: number;
  style?: StyleProp<TextStyle>;
}
```

## Info

### IN-01: Hardcoded rgba values scattered across components

**File:** `mobile/app/(tabs)/_layout.tsx:12`, `mobile/app/(tabs)/inbox.tsx:667,728,733`, `mobile/app/(tabs)/index.tsx:1377,1382`
**Issue:** Several `rgba()` color values are hardcoded inline rather than defined in the theme. Examples: tab bar background `rgba(10,10,18,0.85)`, modal overlay `rgba(0,0,0,0.7)`, toast backgrounds `rgba(125,220,160,0.2)` and `rgba(255,107,107,0.2)`. These are derived from theme colors but computed manually. If the base colors change, these will drift.
**Fix:** Consider adding these as named tokens in `theme.ts` (e.g., `colors.tabBarBg`, `colors.overlayBg`, `colors.toastSuccessBg`, `colors.toastErrorBg`) or computing them from base tokens at definition time.

### IN-02: console.warn and console.error in capture screen

**File:** `mobile/app/(tabs)/index.tsx:147,219,631`
**Issue:** Three `console.warn`/`console.error` calls remain in the capture screen. These are for speech recognition errors and voice capture failures. They serve a debugging purpose but are debug artifacts that could be noisy in production. The code already uses Sentry for structured error reporting alongside some of these.
**Fix:** These are borderline -- they aid development debugging. Consider gating behind `__DEV__` or removing entirely since Sentry already captures the errors.

### IN-03: fontWeight specified alongside fontFamily with weight-specific font

**File:** `mobile/components/ErrorFallback.tsx:55`, `mobile/components/ApiKeyGate.tsx:100,130`, `mobile/app/(tabs)/index.tsx:1358,1540,1608`, and others
**Issue:** Multiple styles set both `fontFamily: theme.fonts.bodySemiBold` (which is `InstrumentSans_600SemiBold`) and `fontWeight: "600"`. When a weight-specific font file is loaded via `expo-font`, the `fontWeight` property is redundant. On some Android devices, specifying both can cause double-application of weight, making text appear bolder than intended.
**Fix:** Remove the explicit `fontWeight` when using weight-specific `fontFamily` values (bodySemiBold, bodyBold, bodyMedium). The font file already encodes the weight. Similarly, `fontWeight: "400"` paired with `theme.fonts.display` is redundant.

### IN-04: Settings "Signed in as" row displays hardcoded username

**File:** `mobile/app/settings.tsx:89`
**Issue:** The "Signed in as" row displays the hardcoded string `"will"`. This is not a bug per se (single-user app), but it reads as a development placeholder rather than deriving from any user context.
**Fix:** No action needed if this is intentional for the single-user context. If deriving from auth context is preferred, wire it up. Otherwise, consider a comment explaining it is intentional.

### IN-05: InstrumentSerif_400Regular imported but only Italic variant used in theme

**File:** `mobile/app/_layout.tsx:8`
**Issue:** The font `InstrumentSerif_400Regular` (non-italic) is imported and loaded in the font map, but the theme only references `InstrumentSerif_400Regular_Italic` via `theme.fonts.display`. No component uses the non-italic serif variant. This adds unnecessary bundle weight (the font file is loaded at startup even though it is never rendered).
**Fix:** Remove the non-italic import and its entry from the `useFonts` map unless there is a planned use case:
```tsx
// Remove this import:
// InstrumentSerif_400Regular,
// And remove from useFonts map:
// InstrumentSerif_400Regular,
```
