# Phase 11.1: Classifier Multi-Bucket Splitting Verification Report

**Phase Goal (from ROADMAP.md):** When a user captures mixed-content text containing multiple distinct intents targeting different buckets, the Classifier splits it into separate inbox items per bucket. Single-intent captures remain unaffected.

**Gap closure additions (from phase directive):** Fix batched tool call handling so multi-bucket captures produce correct results, and revise Classifier instructions to stop over-analyzing conversational phrasing.

## Re-verification Context

The previous VERIFICATION.md (status: passed, 2026-03-01) was written before UAT. UAT on 2026-03-02 discovered two major gaps:

1. **Gap A (adapter batched tool calls):** The SDK delivers all `function_call` contents in one update and all `function_result` contents in another. The single-variable tracking (`current_fc_name`/`current_fc_args`) lost the second result because `current_fc_name` was reset to `None` after the first result was processed. Multi-split captures showed single-bucket toast.

2. **Gap B (Classifier over-analysis):** The instruction "Before classifying, check if the capture contains MULTIPLE distinct intents" caused the Classifier to treat conversational phrasing like "I need to buy dog food" as ambiguous, calling `file_capture` with `status="misunderstood"` instead of classifying directly.

Plan 03 closed Gap A (code fix). Plan 04 closed Gap B (Foundry portal instruction update + live UAT confirmation).
