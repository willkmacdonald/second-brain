  <name>Task 1: Create MCP server package with pyproject.toml and server.py</name>
  <files>mcp/pyproject.toml, mcp/.python-version, mcp/server.py</files>
  <action>
    Create a new `mcp/` directory at the repo root (sibling to `backend/` and `mobile/`).

    **mcp/.python-version**: Single line containing `3.12` (match backend).

    **mcp/pyproject.toml**: Project named `second-brain-mcp`, Python >=3.12. Dependencies:
    - `mcp[cli]` (official MCP SDK with CLI extras)
    - Editable dependency on `../backend` (via `[tool.uv.sources]` with `{ path = "../backend", editable = true }`)

    Use this structure for the editable backend dependency:
    ```toml
    [project]
    name = "second-brain-mcp"
    version = "0.1.0"
    requires-python = ">=3.12"
    dependencies = [
        "mcp[cli]",
        "second-brain-backend",
    ]

    [tool.uv.sources]
    second-brain-backend = { path = "../backend", editable = true }
    ```

    Do NOT add azure-identity or azure-monitor-query -- they come transitively from the backend editable dep.

    **mcp/server.py**: Single-file FastMCP server (~250-300 lines). Structure:

    1. **Logging setup**: `logging.basicConfig(stream=sys.stderr, level=logging.INFO)` -- CRITICAL: all output to stderr, never stdout (stdio transport uses stdout for JSON-RPC).

    2. **AppContext dataclass**: holds `logs_client: LogsQueryClient`, `workspace_id: str`, `credential: DefaultAzureCredential`.

    3. **Lifespan**: Async context manager that:
       - Reads `AZURE_LOG_ANALYTICS_WORKSPACE_ID` from env (empty string if not set, with warning to stderr)
       - Creates `DefaultAzureCredential()` and `LogsQueryClient(credential=credential)` from `azure.identity.aio` and `azure.monitor.query.aio` (async variants)
       - Yields `AppContext` instance
       - In `finally`: closes both `logs_client` and `credential`

    4. **FastMCP instance**: `mcp = FastMCP("second-brain-telemetry", lifespan=lifespan)`

    5. **Helper functions**:
       - `_get_app(ctx: Context[ServerSession, AppContext]) -> AppContext`: extracts `ctx.request_context.lifespan_context`
       - `_check_config(app: AppContext) -> dict | None`: returns error dict if `workspace_id` is empty, else None

    6. **TIME_RANGE_MAP**: Copy from investigation.py (same dict mapping "1h"/"6h"/"24h"/"3d"/"7d" to (kql_duration, timedelta) tuples).

    7. **RESULT_LIMIT = 20**: Higher than the agent's 10 since Claude Code has more screen space.

    8. **ALLOWED_GROUP_BY**: `frozenset({"day", "hour", "bucket", "destination"})`.

    9. **6 tool functions**, all decorated with `@mcp.tool()`:

       **IMPORTANT: Context typing** — Every tool function must use the fully-generic type `ctx: Context[ServerSession, AppContext] = None` (importing `ServerSession` from `mcp.server.session`). Plain `Context` without type parameters will cause `ctx.request_context.lifespan_context` to return `None`. See RESEARCH.md Pitfall 6.

       **IMPORTANT: Tool parity note** — The deployed Investigation Agent (investigation.py) has 4 tools: trace_lifecycle, recent_errors, system_health, usage_patterns. The `admin_audit` tool is NEW for MCP — it wraps `query_admin_audit()` from queries.py which exists but was never wired into the Investigation Agent's class. The MCP server adds it directly.

       **trace_lifecycle(trace_id: str | None = None, ctx: Context[ServerSession, AppContext] = None) -> dict**
       - Docstring: "Trace a specific capture through its full lifecycle. Returns the ordered sequence of log entries showing classification, filing, and admin processing. Pass null to look up the most recent capture."
       - If no trace_id, call `query_latest_capture_trace_id()`
       - Call `query_capture_trace()`, return `[r.model_dump() for r in records]`
       - Truncation: use RESULT_LIMIT but ONLY if len(records) > RESULT_LIMIT. For a single trace, most results fit within 20 rows. If truncating, keep the LAST N records (not first N) since the terminal event (success/failure) is at the end. Return `{"events": records_slice, "total_events": len(records), "truncated": True/False, "note": "Kept last N events to preserve terminal outcome"}`.

       **recent_errors(time_range: str = "24h", component: str | None = None, ctx: Context[ServerSession, AppContext] = None) -> dict**
       - Docstring: "Query recent errors and failures from App Insights. Returns Error-level and Critical-level log entries with optional component filtering."
       - Call `query_recent_failures_filtered()` with `limit=RESULT_LIMIT`
       - Return dict with total_count, returned_count, truncated, time_range, component_filter, errors
       - `FailureQueryResult` already handles truncation metadata (total_count, truncated flag)

       **system_health(time_range: str = "24h", ctx: Context[ServerSession, AppContext] = None) -> dict**
       - Docstring: "Check system health metrics including capture counts, success rate, latency percentiles (P95/P99), and trend comparison against the previous period."
       - Call `query_enhanced_system_health()` with `timespan=td * 2` (double for trend comparison)
       - Return `summary.model_dump()` — single-row result, no truncation needed

       **usage_patterns(time_range: str = "7d", group_by: str = "day", ctx: Context[ServerSession, AppContext] = None) -> dict**
       - Docstring: "Analyze capture usage patterns grouped by time period, bucket, or destination."
       - Validate group_by against ALLOWED_GROUP_BY (default "day")
       - Call `query_usage_patterns()`
       - Truncate pattern list at RESULT_LIMIT if needed: return `{"time_range": ..., "group_by": ..., "patterns": patterns[:RESULT_LIMIT], "total_patterns": len(patterns), "truncated": len(patterns) > RESULT_LIMIT}`

       **admin_audit(ctx: Context[ServerSession, AppContext] = None) -> dict**
       - Docstring: "Query Admin Agent activity logs from the last 24 hours. Shows processing events, successes, and failures."
       - Call `query_admin_audit()`
       - Truncate at RESULT_LIMIT: return `{"events": [r.model_dump() for r in records[:RESULT_LIMIT]], "total_events": len(records), "truncated": len(records) > RESULT_LIMIT}`

       **run_kql(query: str, time_range: str = "24h", ctx: Context[ServerSession, AppContext] = None) -> dict**
       - Docstring: "Execute a raw KQL query against the App Insights Log Analytics workspace. Use for ad-hoc queries not covered by the other tools. Log Analytics is read-only."
       - Call `execute_kql()` with the raw query string
       - Return dict with tables, is_partial, partial_error fields (QueryResult already carries partial metadata)

    **Partial-result propagation:** The underlying `execute_kql()` returns `QueryResult` with `is_partial` and `partial_error` fields. For `run_kql`, these are surfaced directly. For the other 5 tools, the higher-level query functions (query_enhanced_system_health, query_recent_failures_filtered, etc.) consume `execute_kql` internally and don't expose partial metadata in their return types. This is acceptable because: (a) partial results from Log Analytics are rare and indicate server-side timeouts, (b) the structured query functions already handle partial data gracefully, and (c) adding partial propagation to every model would require changing the backend's shared models. If a partial result occurs on a structured tool, the data returned is still the best available. The `run_kql` tool surfaces partial metadata for ad-hoc queries where the user needs to know.

    Every tool function follows this pattern:
    ```python
    app = _get_app(ctx)
    if err := _check_config(app):
        return err
    try:
        # ... query logic ...
        return { ... structured result ... }
    except Exception as exc:
        return {"error": True, "message": str(exc), "type": type(exc).__name__}
    ```
