# Phase 12.1: Admin Agent Deletes Processed Inbox Items - Context

**Gathered:** 2026-03-02
**Status:** Ready for planning

<domain>
## Phase Boundary

After the Admin Agent successfully processes an inbox item (e.g., extracts shopping list items), it deletes the inbox item from Cosmos DB instead of flagging it as processed. This keeps the inbox clean — users see results on the Status/Shopping Lists screen, not stale processed items in the inbox.

</domain>

<decisions>
## Implementation Decisions

### Delete timing
- Delete each inbox item immediately after its own processing succeeds — inline in the same function (`process_admin_capture`)
- Only delete on success — failed items stay in inbox (prevents data loss)
- Delete replaces the "set processed" upsert entirely — remove that code path

### Failure handling
- If the delete call fails (but processing succeeded): log the error to App Insights and move on. Item stays as a harmless orphan.
- If the item was already deleted (user swipe-deleted during processing): catch `CosmosResourceNotFoundError` and treat as success — silent no-op, no log noise
- No retry logic on delete failures

### Audit trail
- Admin bucket document (created during processing) is sufficient as audit trail
- No additional logging of deleted inbox items needed
- No back-reference from Admin bucket doc to original inbox item ID

### Data model
- Keep `adminProcessingStatus` field in `InboxDocument` Pydantic model for backward compatibility with existing documents
- Stop writing the field — items get deleted instead of flagged
- Existing documents with the field still work (Cosmos is schema-less)

### Mobile UI
- No mobile changes — Admin items just disappear from inbox on next fetch
- Leave existing 'processed' badge code in place (harmless dead code, will never render)
- Inbox screen already handles items disappearing between renders via `useFocusEffect` refetch

### Testing
- Replace existing "sets processed status" test with a test verifying `delete_item` is called on success
- Add test for NotFound handling (treat as no-op)
- Add test for delete failure handling (log and continue)

### Claude's Discretion
- Exact error logging format and level for delete failures
- Whether to remove the upsert import if it becomes unused

</decisions>
