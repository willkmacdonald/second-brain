# Phase 12.1: Admin Agent Deletes Processed Inbox Items - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Change the Admin Agent processing trigger from automatic (fires immediately after classification) to user-initiated (fires when user opens the Status screen). After successful processing, delete the inbox item instead of flagging it as "processed". This gives the user a review step in Inbox before processing happens.

The user's workflow: capture → lands in Inbox → user reviews in Inbox, fixes if needed → opens Status → processing triggers → shopping list items appear on Status → inbox item is deleted.

</domain>

<decisions>
## Implementation Decisions

### Processing trigger
- Remove the automatic `asyncio.create_task` call in adapter.py that fires the Admin Agent immediately after classification
- Adapter classifies item as Admin, writes it to Inbox with a `readyForProcessing: true` flag, and stops — no Admin Agent call
- Processing triggers when the user opens the Status screen — the GET /api/shopping-lists endpoint checks for unprocessed Admin inbox items and kicks off processing as a side effect
- All unprocessed Admin items are processed at once (no need for the user to mark individual items as ready)
- Once triggered, processing continues in the background even if the user navigates away from Status
- The Admin Agent processes whatever the current rawText is in the inbox document — if the user edited the item in Inbox, the corrected text is what gets processed
- No visual difference for unprocessed Admin items in the Inbox screen — the bucket label is sufficient

### Status screen UX during processing
- Show a general indicator like "Processing 3 new captures..." banner at the top while the Admin Agent works
- Existing shopping list items display normally below the indicator
- Auto-refresh when processing completes — new shopping list items appear without navigating away
- Failed items also retry: items with `adminProcessingStatus: failed` are re-processed alongside new unprocessed items

### Delete timing
- Inbox item is deleted automatically after the Admin Agent successfully processes it (shopping list items written)
- The item disappears from Inbox on next tab focus (useFocusEffect refetch), not in real-time
- If the user swipe-deletes an Admin item from Inbox before opening Status, the capture is lost — user's choice, no processing happens
- If the user swipe-deletes during processing (race condition), the Admin Agent finishes writing shopping list items — the delete of an already-gone inbox item is a silent no-op

### Failure handling
- If the Admin Agent fails (timeout, agent error), the inbox item stays in Inbox with `adminProcessingStatus: failed`
- Failed items show a visual indicator in Inbox (e.g., red badge or error text) so the user knows something went wrong
- Failed items auto-retry next time the user opens Status
- If processing succeeds but the inbox item delete fails: log a warning and move on — shopping list items are the durable output, the orphaned inbox item is harmless

### Claude's Discretion
- Exact processing indicator design on Status screen
- How the auto-refresh works after processing completes (polling, callback, etc.)
- Exact visual indicator for failed items in Inbox
- Whether `readyForProcessing` is the right flag name or something else fits better

</decisions>

<specifics>
## Specific Ideas

No specific requirements — open to standard approaches. The key insight is that the user reviews captures in Inbox before processing, so processing must not be automatic.

</specifics>
