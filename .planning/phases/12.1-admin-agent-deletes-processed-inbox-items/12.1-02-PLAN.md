**Fix 1 -- admin_handoff.py (lines 93-99):** In the generic `except Exception as del_exc` handler for delete_item failure on the success path, after logging the warning, update the inbox document's adminProcessingStatus to 'failed' so the retry query picks it up. This ensures items never get stuck in permanent 'pending' state when delete fails.

Change the except block (lines 93-99) to:

```python
            except Exception as del_exc:
                # Non-fatal: shopping list items are the durable output
                # But mark as 'failed' so retry query picks it up
                logger.warning(
                    "Failed to delete processed inbox item %s: %s",
                    inbox_item_id,
                    del_exc,
                )
                try:
                    doc = await inbox_container.read_item(
                        item=inbox_item_id, partition_key="will"
                    )
                    doc["adminProcessingStatus"] = "failed"
                    await inbox_container.upsert_item(body=doc)
                except Exception:
                    logger.warning(
                        "Also failed to mark inbox item %s as 'failed' "
                        "after delete failure",
                        inbox_item_id,
                    )
```

This mirrors the pattern already used on the agent error path (lines 118-130). The inner try/except prevents double-fault from propagating.

**Fix 2 -- shopping_lists.py (lines 119-125):** Expand the retry query to also include items with adminProcessingStatus='pending' that have been stuck for more than 5 minutes. This is a safety net for any edge case where an item gets stuck in 'pending' (background task crash, container restart, etc.).

Change the query to:

```python
            query = (
                "SELECT c.id, c.rawText FROM c "
                "WHERE c.userId = @userId "
                "AND c.classificationMeta.bucket = 'Admin' "
                "AND (NOT IS_DEFINED(c.adminProcessingStatus) "
                "     OR c.adminProcessingStatus = 'failed' "
                "     OR c.adminProcessingStatus = 'pending')"
            )
```

Note: Including ALL 'pending' items (not just stale ones) is acceptable because:
- If the item is currently being processed by an active background task, the Admin Agent is idempotent (adding duplicate shopping items is the less harmful outcome vs. items being stuck forever)
- The 'pending' state is only set momentarily before get_response -- if an item is still 'pending' by the time the user opens Status again, it is effectively stale
- Cosmos has no server-side timestamp comparison in partition-key queries, and adding a client-side age filter would require querying a timestamp field we don't currently store

This is the pragmatic fix. The primary fix (Fix 1) prevents most 'pending' stuck states from occurring in the first place.
