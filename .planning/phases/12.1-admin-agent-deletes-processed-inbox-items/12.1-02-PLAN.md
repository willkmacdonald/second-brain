**shopping_lists.py (lines 119-125):** Expand the retry query to also include items with adminProcessingStatus='pending'. This is the safety net for items stuck in 'pending' state from any cause (delete failure after successful processing, background task crash, container restart, etc.).

Change the query to:

```python
            query = (
                "SELECT c.id, c.rawText FROM c "
                "WHERE c.userId = @userId "
                "AND c.classificationMeta.bucket = 'Admin' "
                "AND (NOT IS_DEFINED(c.adminProcessingStatus) "
                "     OR c.adminProcessingStatus = 'failed' "
                "     OR c.adminProcessingStatus = 'pending')"
            )
```

Note: Including ALL 'pending' items (not just stale ones) is acceptable because:
- If the item is currently being processed by an active background task, the Admin Agent is idempotent (adding duplicate shopping items is the less harmful outcome vs. items being stuck forever)
- The 'pending' state is only set momentarily before get_response -- if an item is still 'pending' by the time the user opens Status again, it is effectively stale
- Cosmos has no server-side timestamp comparison in partition-key queries, and adding a client-side age filter would require querying a timestamp field we don't currently store

**DO NOT modify admin_handoff.py.** Per user decision (CONTEXT.md, "Delete timing"): "If processing succeeds but the inbox item delete fails: log a warning and move on." The current delete failure handler already does exactly this -- it logs a warning and does not update any status. This is correct behavior. The orphaned inbox item with adminProcessingStatus='pending' will be picked up by the expanded retry query on next Status open.
