# Phase 12.2: Rename Admin infrastructure from shopping lists to errands - Context

**Gathered:** 2026-03-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Rename the Admin Agent's data model, API endpoints, tools, and UI from "shopping list" terminology to a generic "errands" system. This is a rename/refactoring phase — no new capabilities are added. The Admin function handles any errand (appointments, reminders, purchases), not just shopping. Existing destination values (jewel, cvs, pet_store, other) remain unchanged.

</domain>

<decisions>
## Implementation Decisions

### Naming terminology
- Top-level concept: "Errands" (replaces "shopping list")
- Item model: `ErrandItem` (replaces `ShoppingListItem`) with fields: id, name, destination
- Field rename: `store` → `destination` (where the errand gets done — physical place or online)
- Known destination values stay as-is: jewel, cvs, pet_store, other
- No new destinations added in this phase

### API path design
- New base path: `/api/errands` (replaces `/api/shopping-lists`)
- GET `/api/errands` — list all errand items grouped by destination
- DELETE `/api/errands/{id}` — delete a single errand item
- Backend file rename: `shopping_lists.py` → `errands.py`
- Admin Agent tool rename: `add_shopping_list_items()` → `add_errand_items()`
- Tool's LLM-facing description updated to reference "errand items" and "destination" instead of "shopping list" and "store"

### Database migration
- Create new Cosmos DB container `Errands` with partition key `/destination`
- One-time CI/CD migration script: copy items from `ShoppingLists` to `Errands`, mapping `store` → `destination`
- Verify item count matches after migration
- Delete old `ShoppingLists` container after successful verification

### UI copy & labels
- Screen title: "Errands" (replaces shopping list header)
- Component file rename: `ShoppingListRow.tsx` → `ErrandRow.tsx`
- Component name: `ErrandRow` (replaces `ShoppingListRow`)
- Section type string: `"errand"` (replaces `"shopping"`)
- Empty state text: "No errands yet"
- Section headers remain destination display names (Jewel, CVS, Pet Store, Other)

### Claude's Discretion
- Response model naming (ErrandResponse, DestinationSection, etc.)
- Test file naming and organization
- Migration script error handling details
- STORE_DISPLAY_NAMES constant rename (→ DESTINATION_DISPLAY_NAMES or similar)

</decisions>

<specifics>
## Specific Ideas

- Will wants the ability to create "affinity" to destinations (e.g., doctor, Jewel, CVS) with knowledge about what types of errands go where — e.g., "laundry detergent goes to CVS", "prescription cat food is online"
- Destinations can be physical places OR online — pet_store could be online for litter/prescription food
- Multiple grocery stores depending on what's being bought
- Knowledge/affinity approach preferred over hardcoded Agent instructions for destination routing

</specifics>

<deferred>
## Deferred Ideas

- **Affinity/knowledge system** — Teaching the Agent where items go (destination routing intelligence). Would allow the Agent to know "laundry detergent → CVS" or "cat litter → online pet store". Separate phase.
- **Dynamic destinations** — Removing the hardcoded destination list in favor of user-defined destinations. Depends on affinity system.
- **Online vs physical distinction** — Marking destinations as online or physical. Tied to affinity/knowledge system.

</deferred>
