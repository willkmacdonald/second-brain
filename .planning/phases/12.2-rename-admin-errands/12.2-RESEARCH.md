# Phase 12.2: Rename Admin Infrastructure from Shopping Lists to Errands - Research

**Researched:** 2026-03-15
**Domain:** Backend/mobile refactoring + Cosmos DB container migration
**Confidence:** HIGH

## Summary

This is a pure rename/refactoring phase with a Cosmos DB container migration. Every reference to "shopping list" terminology must change to "errands" terminology across backend models, API endpoints, admin tools, tests, and mobile UI. The scope is well-bounded: there are exactly 8 source files, 3 test files, and 1 infrastructure constant file that need changes, plus a new Cosmos DB container needs to be created and data migrated.

The riskiest part is the Cosmos DB container migration. The existing `ShoppingLists` container uses `/store` as partition key. The new `Errands` container will use `/destination` as partition key. Since the partition key path is immutable on a Cosmos DB container, this requires creating a new container, copying data with field mapping (`store` -> `destination`), verifying counts, and deleting the old container.

**Primary recommendation:** Execute as a coordinated rename across all layers (model -> tools -> API -> UI), with the Cosmos DB migration as a separate CI/CD script that runs before the code changes deploy.

### Pattern: Cosmos DB Container Migration

The existing `ShoppingLists` container has partition key `/store`. The new `Errands` container needs partition key `/destination`. Cosmos DB does not allow changing partition keys on existing containers, so this requires:

1. **Create** the `Errands` container in Cosmos DB with partition key `/destination`
2. **Copy** all documents from `ShoppingLists` to `Errands`, mapping `store` -> `destination`
3. **Verify** item counts match between source and target
4. **Delete** the `ShoppingLists` container after verification

The migration script needs Azure credentials (DefaultAzureCredential) and the COSMOS_ENDPOINT env var.

**Critical ordering:** The migration script MUST run before the new code deploys. If new code deploys first and references the `Errands` container that doesn't exist yet, the app will fail.

### Pattern: API Path Change and Mobile Impact

The mobile app fetches from `${API_BASE_URL}/api/shopping-lists`. When the backend path changes to `/api/errands`, the mobile app must be updated in the same deployment. Since the mobile app is an Expo app served via Metro (not deployed to the backend), the mobile change is independent -- but both must be updated before either is used.

Since Will doesn't do local testing and deploys via CI/CD, the backend rename and mobile rename should be in the same commit/PR so they deploy together.

### Anti-Patterns to Avoid

- **Partial rename:** Leaving some references as "shopping" while others say "errands" creates confusion. Use search to verify zero remaining references after the rename.
- **Changing destination values:** The values `jewel`, `cvs`, `pet_store`, `other` must NOT change -- only the field name changes from `store` to `destination`.
- **Changing Admin Agent Foundry instructions during this phase:** The Agent instructions in the Foundry portal use "store" terminology. These should be updated to say "destination" but this is a portal change, not a code change. The tool's `Field(description=...)` is what the agent sees for tool parameters.
- **Breaking the query parameter name:** The DELETE endpoint currently uses `?store=` as a query param. This must change to `?destination=` consistently.

**Key insight:** The migration script should use the same `CosmosManager` pattern already in the codebase, extended to handle both old and new container names temporarily.

## Common Pitfalls

### Pitfall 1: Migration Script Runs After Code Deploy
**What goes wrong:** New code references `Errands` container, but it doesn't exist yet. App crashes on startup or first request.
**Why it happens:** CI/CD pipeline deploys code before running migration.
**How to avoid:** Run migration script as a pre-deployment step in CI/CD, or run it manually before merging the code changes.
**Warning signs:** 503 errors on `/api/errands`, Cosmos `ValueError` for unknown container name.

### Pitfall 2: Cosmos DB Container Creation Partition Key
**What goes wrong:** New `Errands` container created with wrong partition key (e.g., `/userId` default).
**Why it happens:** `cosmos.py` has a global `PARTITION_KEY = "/userId"` constant. The `ShoppingLists` container was created with `/store` outside of the code. The new `Errands` container needs `/destination`.
**How to avoid:** Explicitly specify `/destination` as partition key in container creation. Do NOT use the default `PARTITION_KEY` constant.
**Warning signs:** Cross-partition queries, items not found by destination.

### Pitfall 3: Stale `__pycache__` After File Rename
**What goes wrong:** Python imports the old `shopping_lists.pyc` from `__pycache__` instead of the new `errands.py`.
**Why it happens:** Docker build doesn't always clean `__pycache__`. Development environments retain cached bytecode.
**How to avoid:** The Docker build runs fresh each time in CI/CD, so this is only a local development risk. Not a concern for this project since Will doesn't run the backend locally.
**Warning signs:** Import errors mentioning `shopping_lists` after rename.

### Pitfall 4: Admin Agent Tool Name in Foundry
**What goes wrong:** The Admin Agent in Azure AI Foundry portal still references `add_shopping_list_items` tool name. When the tool is renamed to `add_errand_items`, the agent might not recognize the tool.
**Why it happens:** The `@tool` decorator generates the tool schema from the method name. The Foundry agent's instructions may reference the old tool name.
**How to avoid:** The agent calls tools by the schema name generated from the Python method name. As long as the tool is registered correctly at startup, the agent will see `add_errand_items`. The Foundry portal instructions should be updated to reference the new name, but the agent adapts to available tools.
**Warning signs:** Agent responds without calling any tool (the "no tool call" failure path already handled in `admin_handoff.py`).

### Pitfall 5: Mobile Cache Serving Stale Code
**What goes wrong:** Metro serves cached bundle with old `ShoppingListRow` import.
**Why it happens:** Metro bundler caches resolved imports.
**How to avoid:** Run `npx expo start --clear` after the rename (already documented in project memory).
**Warning signs:** Import errors in the mobile app after rename.

## Code Examples

### Migration Script Pattern

The migration script should follow the existing `CosmosManager` pattern. Key operations:

```python
# Source: Existing codebase pattern (backend/src/second_brain/db/cosmos.py)
from azure.cosmos.aio import CosmosClient
from azure.identity.aio import DefaultAzureCredential

async def migrate():
    credential = DefaultAzureCredential()
    client = CosmosClient(url=endpoint, credential=credential)
    database = client.get_database_client("second-brain")

    # Create new container with /destination partition key
    await database.create_container_if_not_exists(
        id="Errands",
        partition_key=PartitionKey(path="/destination"),
    )

    old_container = database.get_container_client("ShoppingLists")
    new_container = database.get_container_client("Errands")

    # Copy with field mapping
    count = 0
    for store in ["jewel", "cvs", "pet_store", "other"]:
        async for item in old_container.query_items(
            query="SELECT * FROM c",
            partition_key=store,
        ):
            new_doc = {
                "id": item["id"],
                "name": item["name"],
                "destination": item["store"],  # Field rename
            }
            await new_container.create_item(body=new_doc)
            count += 1

    # Verify counts match
    # ... then delete old container
```

## Open Questions

1. **Cosmos DB container creation method**
   - What we know: The `Errands` container needs to be created with partition key `/destination`. The existing code does not create containers -- it assumes they exist.
   - What's unclear: Whether to create the container via the migration script (using the SDK) or via Azure CLI before running the script.
   - Recommendation: Use the SDK's `create_container_if_not_exists` in the migration script itself. This is idempotent and self-contained.

2. **Admin Agent Foundry portal instructions**
   - What we know: The Admin Agent's instructions in the Foundry portal reference "store" terminology. The tool parameter descriptions come from the Python code's `Field(description=...)`.
   - What's unclear: Whether the Foundry portal instructions need updating in this phase or a separate step.
   - Recommendation: Update the Foundry portal instructions to reference "destination" instead of "store" as a manual step after the code deploys. The tool parameter schema (which the agent sees) is updated automatically by the code changes. The portal instructions are supplementary context.

3. **Migration timing relative to deployment**
   - What we know: CI/CD deploys on push to main. Migration must happen before the new code starts referencing `Errands` container.
   - What's unclear: Best way to orchestrate this -- add migration step to CI/CD, or run manually?
   - Recommendation: Run migration script manually before merging the rename PR. Will has admin access. This is a one-time operation and avoids CI/CD complexity. Alternatively, the `CosmosManager.initialize()` could be temporarily modified to handle both container names during a transition period, but this adds unnecessary complexity for a one-time migration.
