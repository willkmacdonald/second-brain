# Phase 12.5: On-Device Voice Transcription (SpeechAnalyzer) - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace cloud-based Azure OpenAI gpt-4o-transcribe with iOS on-device transcription via Apple's SpeechAnalyzer (iOS 26). Voice captures transcribe locally on the phone, then send the resulting text through the existing text capture endpoint (/api/captures). The backend voice-capture endpoint (upload → Whisper) remains fully operational as a fallback. Backend changes are minimal — this is primarily a mobile-side change.

</domain>

<decisions>
## Implementation Decisions

### Transcription UX Flow
- Real-time streaming text appears as the user speaks — words show on screen during recording (like iMessage dictation)
- Live transcription text replaces the elapsed timer — the streaming text IS the recording feedback
- Pulsing red ring around record button stays during recording (same as today)
- No editing step — transcription auto-submits to classification immediately when user taps stop
- Manual stop only — no auto-stop on silence
- After stop, transcribed text stays visible on screen while classification step dots (Orchestrator → Classifier) appear below
- Empty/poor transcription submits anyway — let the classifier handle it (may trigger HITL)
- Perception step removed from step dots — on-device transcription means only Orchestrator → Classifier steps shown

### Fallback Behavior
- Silent automatic fallback — if SpeechAnalyzer is unavailable (older iOS), app silently uses existing cloud upload → Whisper flow without notifying the user
- Audio must still be recorded even during on-device transcription, so there's a file to upload if on-device fails mid-recording
- If on-device transcription fails mid-recording, fall back to cloud for that capture (upload the recorded audio)
- Cloud fallback UX reverts to the old flow exactly: timer + pulsing dot during recording, then upload, then Perception → Orchestrator → Classifier step dots
- Backend voice-capture endpoint kept fully operational — no code removal, no deprecation

### Backend Pipeline Changes
- On-device transcribed voice uses the existing /api/captures text endpoint — from the backend's perspective, it's just text
- No SSE indication that the capture originated from voice — it's just text now
- Keep sendVoiceCapture function as-is for cloud fallback path; on-device path uses sendTextCapture (existing)
- Two separate client code paths: on-device → sendTextCapture, cloud fallback → sendVoiceCapture
- Include X-Capture-Source: voice header on on-device voice captures for App Insights observability (doesn't affect classification)
- Orchestrator instructions updated to reflect both paths: voice captures may arrive as pre-transcribed text (on-device) or via Perception Agent (cloud fallback)
- Azure Blob Storage left alone — still used for cloud fallback
- Multi-bucket splitting works the same — voice-to-text is just text
- Voice follow-ups during HITL also use on-device transcription (user can reply by voice, transcribed on-device, sent as text follow-up)

### iOS Version Gating
- Will is on iOS 26.3.1 — SpeechAnalyzer is immediately usable
- Keep current minimum iOS deployment target — runtime check for SpeechAnalyzer availability
- If SpeechAnalyzer not available at runtime → silent fallback to cloud transcription
- iPhone only — no iPad considerations
- Researcher should evaluate expo-speech-transcriber and alternatives for best Expo-compatible SpeechAnalyzer wrapper

### Claude's Discretion
- Exact layout of live transcription text area (font size, positioning, scroll behavior for long captures)
- How interim vs final transcription results are visually distinguished (if at all)
- Error handling details for SpeechAnalyzer permission requests

</decisions>

## Specific Ideas

- Live transcription should feel like dictation in iMessage — words appearing in real-time as you speak
- The streaming text replaces the timer as the primary "recording is happening" feedback
- Cloud fallback should be invisible to the user — they just don't get live text, but everything else works

</specifics>
