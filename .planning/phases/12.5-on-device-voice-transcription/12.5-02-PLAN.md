  <action>
    Rewrite the voice recording logic in CaptureScreen to use on-device speech recognition as the primary path, with silent fallback to cloud. This modifies the existing voice mode behavior significantly.

    **New state variables to add:**
    ```typescript
    const [transcriptText, setTranscriptText] = useState(""); // Real-time transcription
    const [audioFileUri, setAudioFileUri] = useState<string | null>(null); // Persisted audio for fallback
    const [useOnDevice, setUseOnDevice] = useState(false); // Capability flag
    const [speechPermissionGranted, setSpeechPermissionGranted] = useState(false);
    ```

    **New imports to add:**
    ```typescript
    import { useSpeechRecognitionEvent } from "expo-speech-recognition";
    import {
      checkOnDeviceSupport,
      requestSpeechPermissions,
      startOnDeviceRecognition,
      stopRecognition,
      abortRecognition,
    } from "../../lib/speech";
    ```

    **Permission request change (useEffect for voice mode):**
    Replace the existing mic-only permission request with a combined permission request:
    - If on-device is supported (`checkOnDeviceSupport()`), request speech recognition permissions via `requestSpeechPermissions()` (which requests both mic + speech recognition).
    - If speech recognition permission granted: set `useOnDevice = true` and `speechPermissionGranted = true`.
    - If speech recognition permission denied but mic was granted: fall back to cloud path. Set `speechPermissionGranted = false`, `useOnDevice = false`, but set `permissionGranted = true` (mic-only, for cloud fallback).
    - If on-device is NOT supported: request mic permission only (existing `AudioModule.requestRecordingPermissionsAsync()`) for cloud fallback path.
    - Always set audio mode when mic permission is granted: `AudioModule.setAudioModeAsync({ allowsRecording: true, playsInSilentMode: true })`.

    **Speech recognition event hooks (add at component top level, alongside other hooks):**

    1. `useSpeechRecognitionEvent("result", ...)` — update `transcriptText` with `event.results[0]?.transcript ?? ""`.

    2. `useSpeechRecognitionEvent("audiostart", ...)` — store `event.uri` in `audioFileUri` (the persisted audio file for fallback).

    3. `useSpeechRecognitionEvent("error", ...)` — on error during recording:
       - Log warning: `console.warn("Speech recognition error:", event.error, event.message)`
       - If `audioFileUri` exists: fall back to cloud upload via `sendVoiceCapture({ audioUri: audioFileUri, ... })` with the SAME callbacks pattern as the existing voice capture (uploading -> classifying stages, all HITL callbacks).
       - Clear recording state: `setIsRecording(false)`.

    4. `useSpeechRecognitionEvent("end", ...)` — when recognition ends (after stop()):
       - If `transcriptText.trim()` is non-empty AND `isRecording` was just stopped (use a ref `wasRecordingRef` to track this — set to true before stop, check here):
         - Submit via `sendCapture({ message: transcriptText.trim(), apiKey: API_KEY, callbacks: { ... same callbacks as existing text capture ... }, captureSource: "voice" })`.
         - Set `processingStage = "classifying"` and `processing = true`.
         - Show step dots: Orchestrator -> Classifier (NOT Perception — on-device path skips Perception).
         - Keep transcribed text visible on screen while classification runs (per CONTEXT.md decision).
       - If `transcriptText.trim()` is empty: submit anyway — let the classifier handle it (may trigger HITL). Per CONTEXT.md: "Empty/poor transcription submits anyway."
       - Clear the `wasRecordingRef`.

    **Rewrite handleRecordToggle:**

    The existing `handleRecordToggle` uses expo-audio exclusively. Rewrite to branch on `useOnDevice`:

    **If on-device (`useOnDevice === true`):**
    - **Start:** Call `startOnDeviceRecognition()` from speech.ts. Set `isRecording = true`, `transcriptText = ""`, `audioFileUri = null`. Haptic feedback. Do NOT start expo-audio recording (expo-speech-recognition handles audio recording via `recordingOptions.persist`). This avoids the AVAudioEngine conflict pitfall.
    - **Stop:** Set `wasRecordingRef.current = true`. Call `stopRecognition()` from speech.ts. Set `isRecording = false`. Haptic feedback. The "end" event handler (above) will handle submission. Do NOT check duration here — the "end" event handles everything.

    **If cloud fallback (`useOnDevice === false`):**
    - Keep the EXACT existing behavior (expo-audio record -> stop -> check duration -> sendVoiceCapture). No changes to this path.

    **Voice mode UI changes (JSX):**

    In the voice mode area (`mode === "voice" && !processingStage && !agentQuestion`):

    - **When recording AND useOnDevice:** Show `transcriptText` instead of the timer. Use a ScrollView for long captures. Style: similar to the timer position, but left-aligned text with good readability. Font size ~18, color #ccc, max height ~200 with scroll. If transcriptText is empty (waiting for first words), show a subtle "Listening..." placeholder.
    - **When recording AND NOT useOnDevice (cloud fallback):** Show the timer (existing behavior, unchanged).
    - **The pulsing red ring around the record button stays in both paths** (per CONTEXT.md locked decision).
    - **After stop (transcribed text visible during classification):** When `processingStage` is active AND `transcriptText` is non-empty, show the transcribed text above the processing indicator. This gives the user the "transcribed text stays visible on screen while classification step dots appear below" behavior from CONTEXT.md.

    **Processing stage text changes:**
    - On-device path: only show "Classifying..." (skip "Uploading..." and "Transcribing..." — there's no upload or cloud transcription).
    - Cloud fallback path: show "Uploading..." then "Classifying..." (existing behavior, unchanged).

    **Voice follow-up changes:**

    Update `handleFollowUpRecordToggle` and `handleVoiceFollowUp` to use on-device transcription for follow-ups:

    - **If useOnDevice:** Start on-device recognition on record start. On stop, use the "end" event to get transcribed text, then submit via `sendFollowUp({ inboxItemId: misunderstoodInboxItemId, followUpText: transcribedText, followUpRound, apiKey: API_KEY, callbacks: { ... existing follow-up callbacks ... } })`.
    - **If NOT useOnDevice:** Keep existing behavior (expo-audio record -> sendFollowUpVoice with audio upload).

    The follow-up voice flow is simpler: transcribe on-device, submit as text follow-up. No need for a separate voice follow-up endpoint.

    **Cleanup on unmount:**
    Add `abortRecognition()` call in the existing cleanup useEffect to stop any active speech recognition session.

    **resetState update:**
    Add `setTranscriptText("")` and `setAudioFileUri(null)` to the resetState callback.

    **CRITICAL anti-patterns to avoid (from RESEARCH.md):**
    - Do NOT run expo-audio recording AND expo-speech-recognition simultaneously. When `useOnDevice` is true, only use expo-speech-recognition (it handles audio recording internally via `recordingOptions.persist`). Only use expo-audio when `useOnDevice` is false (cloud fallback).
    - Do NOT check `isAnalyzerAvailable()` — use `supportsOnDeviceRecognition()` via the speech.ts helper.
    - Do NOT send transcribed text to the voice endpoint. On-device path goes to /api/capture (text endpoint) via sendCapture.
