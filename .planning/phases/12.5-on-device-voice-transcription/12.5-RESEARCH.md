## Summary

This phase replaces cloud-based Azure OpenAI gpt-4o-transcribe with iOS on-device speech recognition for voice captures. The change is **primarily mobile-side** with minimal backend changes (adding X-Capture-Source header support and updating Orchestrator instructions).

**Critical finding:** The CONTEXT.md references "SpeechAnalyzer" as the target API, but the real-time streaming use case (words appearing as the user speaks) is best served by **`SFSpeechRecognizer` with `requiresOnDeviceRecognition: true`**, not `SpeechAnalyzer`. Apple's `SpeechAnalyzer` (iOS 26) is a new modular framework primarily designed for file-based and long-form transcription. For live microphone streaming with interim results, `SFSpeechRecognizer` remains the standard approach (available since iOS 13, on-device since iOS 13). The `expo-speech-recognition` library wraps `SFSpeechRecognizer` and provides exactly the real-time streaming, interim results, and simultaneous audio persistence that this phase requires.

**The recommended library is `expo-speech-recognition` (v3.1.1)** -- the actively maintained successor to `@jamsch/expo-speech-recognition`. It provides: real-time streaming with interim results, on-device-only mode (`requiresOnDeviceRecognition`), simultaneous audio file persistence (`recordingOptions.persist`), and a React hook-based API. It does NOT use Apple's `SpeechAnalyzer` framework -- it uses `SFSpeechRecognizer` which is the correct API for live streaming dictation.

**Primary recommendation:** Use `expo-speech-recognition` v3.x with `requiresOnDeviceRecognition: true`, `interimResults: true`, `continuous: true`, and `recordingOptions.persist: true`. Replace `expo-audio` recording + cloud upload with unified transcribe-and-record in one library. Fallback to existing cloud path when `supportsOnDeviceRecognition()` returns false.

### Why NOT SpeechAnalyzer Directly

Apple's `SpeechAnalyzer` (iOS 26) is a powerful new framework, but it is **not the right choice for this phase**:

1. **No Expo/React Native wrapper exists for real-time SpeechAnalyzer streaming.** The `expo-speech-transcriber` library only uses SpeechAnalyzer for file-based transcription (`transcribeAudioWithAnalyzer`). Its real-time mode uses `SFSpeechRecognizer`.
2. **SFSpeechRecognizer with `requiresOnDeviceRecognition` already provides on-device processing** -- privacy, no network, no API cost. This is the same on-device model.
3. **SpeechAnalyzer's streaming API uses Swift async sequences** that would require complex bridging to React Native's event system. Not worth the effort when `SFSpeechRecognizer` does exactly what's needed.
4. **The user's goal is "words appearing as I speak" (dictation UX)** -- `SFSpeechRecognizer` with `shouldReportPartialResults` / `interimResults` is the standard iOS pattern for this, used by iMessage and every dictation app.

### Pattern 1: Capability-Based Path Selection
**What:** At app startup (or first voice mode entry), check `supportsOnDeviceRecognition()`. Store the result. Use it to decide between on-device transcription path and cloud fallback path.
**When to use:** Every time the user enters voice mode or starts recording.
**Example:**
```typescript
// Source: expo-speech-recognition docs
import {
  ExpoSpeechRecognitionModule,
  useSpeechRecognitionEvent,
  supportsOnDeviceRecognition,
} from "expo-speech-recognition";

// Check once, cache result
const canTranscribeOnDevice = supportsOnDeviceRecognition();

// In recording handler:
if (canTranscribeOnDevice) {
  // On-device path: start speech recognition
  ExpoSpeechRecognitionModule.start({
    lang: "en-US",
    interimResults: true,
    continuous: true,
    requiresOnDeviceRecognition: true,
    addsPunctuation: true,
    recordingOptions: {
      persist: true,           // Save audio file for fallback
      outputEncoding: "m4a",   // Match existing format
    },
  });
} else {
  // Cloud fallback: use existing expo-audio recording -> sendVoiceCapture
  await audioRecorder.prepareToRecordAsync();
  audioRecorder.record();
}
```

### Pattern 2: Real-Time Streaming Display
**What:** Use `useSpeechRecognitionEvent("result", ...)` to receive interim and final transcription results. Display the current text in place of the timer.
**When to use:** During on-device recording.
**Example:**
```typescript
// Source: expo-speech-recognition docs
const [transcriptText, setTranscriptText] = useState("");

useSpeechRecognitionEvent("result", (event) => {
  const text = event.results[0]?.transcript ?? "";
  setTranscriptText(text);
  // event.isFinal indicates final result (after stop)
});
```

### Pattern 3: Stop -> Submit as Text
**What:** When user taps stop, call `ExpoSpeechRecognitionModule.stop()`. On the `"end"` event, take the accumulated text and submit via `sendCapture()` (the existing text capture path) with an added `X-Capture-Source: voice` header.
**When to use:** On every on-device voice capture completion.
**Example:**
```typescript
useSpeechRecognitionEvent("end", () => {
  if (transcriptText.trim()) {
    // Submit as text capture (on-device path)
    sendCapture({
      message: transcriptText.trim(),
      apiKey: API_KEY,
      headers: { "X-Capture-Source": "voice" },
      callbacks: { /* same as existing text callbacks */ },
    });
  }
});
```

### Pattern 4: Mid-Recording Fallback
**What:** If on-device transcription fails during recording (error event), grab the persisted audio file URI and fall back to cloud upload via `sendVoiceCapture()`.
**When to use:** On transcription error during recording.
**Example:**
```typescript
let audioFileUri: string | null = null;

useSpeechRecognitionEvent("audiostart", (event) => {
  audioFileUri = event.uri; // File being written
});

useSpeechRecognitionEvent("error", (event) => {
  if (audioFileUri) {
    // Fall back to cloud: upload the persisted audio file
    sendVoiceCapture({
      audioUri: audioFileUri,
      apiKey: API_KEY,
      callbacks: { /* existing voice callbacks with timer UX */ },
    });
  }
});
```

### Anti-Patterns to Avoid
- **Running expo-audio recording AND expo-speech-recognition simultaneously:** Both compete for `AVAudioEngine` on iOS. Use `expo-speech-recognition`'s built-in `recordingOptions.persist` instead. The library handles dual-output (transcription + file) internally.
- **Checking SpeechAnalyzer availability (`isAnalyzerAvailable`) as the capability gate:** SpeechAnalyzer availability is irrelevant for real-time streaming. Check `supportsOnDeviceRecognition()` from `expo-speech-recognition` instead.
- **Sending the transcribed text to the backend voice endpoint:** On-device transcribed text goes to `/api/capture` (text endpoint), not `/api/capture/voice`. The voice endpoint expects a file upload.
- **Removing expo-audio from the project:** `expo-audio` is still needed for the cloud fallback path on older iOS and for voice follow-ups in the fallback scenario.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Real-time speech recognition | Custom Swift native module with SFSpeechRecognizer bridging | `expo-speech-recognition` | Complex AVAudioEngine + SFSpeechRecognizer lifecycle management, audio session conflicts, permission handling -- all solved by the library |
| Simultaneous recording + transcription | Two separate audio pipelines (expo-audio + speech lib) | `expo-speech-recognition` with `recordingOptions.persist` | AVAudioEngine conflict between expo-audio and speech recognition libraries. expo-speech-recognition handles both internally. |
| Capability detection & fallback | Manual iOS version checking with Platform.Version | `supportsOnDeviceRecognition()` from expo-speech-recognition | Runtime check accounts for actual device capability, not just iOS version |

**Key insight:** The entire native complexity (AVAudioEngine setup, audio session management, SFSpeechRecognizer lifecycle, permission orchestration, simultaneous file persistence) is handled by `expo-speech-recognition`. Hand-rolling any of this invites audio session conflicts and race conditions.

## Common Pitfalls

### Pitfall 1: Audio Session Conflicts
**What goes wrong:** Using `expo-audio` for recording while `expo-speech-recognition` is active causes one or both to fail silently or produce corrupted output.
**Why it happens:** Both libraries configure `AVAudioSession` and install taps on `AVAudioEngine.inputNode`. Only one tap can be active at a time.
**How to avoid:** Never run expo-audio recording and expo-speech-recognition simultaneously. Use `recordingOptions.persist` for the audio file. Only use expo-audio when in the cloud fallback path (where speech recognition is NOT active).
**Warning signs:** Silent recording failures, empty transcriptions, audio routing to earpiece instead of speaker.

### Pitfall 2: Permission Dance
**What goes wrong:** App crashes or silent failures because speech recognition permission wasn't granted separately from microphone permission.
**Why it happens:** iOS requires TWO separate permissions: microphone access (already granted for expo-audio) AND speech recognition. The user may have granted mic but not speech recognition.
**How to avoid:** Request both permissions before first use: `requestMicrophonePermissionsAsync()` + `requestSpeechRecognizerPermissionsAsync()`. If speech recognition is denied, fall back to cloud path silently.
**Warning signs:** `"not-allowed"` error events from expo-speech-recognition despite microphone working fine.

### Pitfall 3: Development Build Required
**What goes wrong:** `expo-speech-recognition` module not found at runtime.
**Why it happens:** The library requires native modules that aren't available in Expo Go. A development build (via EAS Build) must be created after adding the config plugin.
**How to avoid:** Run `eas build --profile development --platform ios` after adding the plugin to app.json. Use the development build for all testing.
**Warning signs:** "Native module not found" errors, blank screens on voice mode.

### Pitfall 4: Continuous Mode and Final Results
**What goes wrong:** Transcription stops after the first sentence or pause.
**Why it happens:** Without `continuous: true`, `SFSpeechRecognizer` ends recognition after detecting a pause.
**How to avoid:** Always set `continuous: true` for the recording session. Call `stop()` explicitly when the user taps the stop button.
**Warning signs:** Transcription stops mid-capture, user has to speak without pausing.

### Pitfall 5: On-Device Model Not Downloaded
**What goes wrong:** `requiresOnDeviceRecognition: true` but recognition fails or returns empty results.
**Why it happens:** The on-device speech model for the locale may not be downloaded yet. iOS downloads it on demand but this can take time on first use.
**How to avoid:** Check `supportsOnDeviceRecognition()` before relying on it. If the model isn't ready, fall back to cloud. Consider calling `androidTriggerOfflineModelDownload()` on Android (not relevant here since iPhone only).
**Warning signs:** Empty transcription results, longer-than-expected startup time on first use.

### Pitfall 6: Memory Leak from Uncleared Listeners
**What goes wrong:** Multiple speech recognition sessions accumulate event listeners, causing memory leaks.
**Why it happens:** Not cleaning up event listeners when the component unmounts or between recording sessions.
**How to avoid:** Use the `useSpeechRecognitionEvent` hook (auto-cleanup on unmount) rather than manual `addListener`. Call `abort()` on unmount if recognition is still active.
**Warning signs:** Increasing memory usage across multiple voice captures, duplicate transcription events.

import {
  ExpoSpeechRecognitionModule,
  useSpeechRecognitionEvent,
} from "expo-speech-recognition";

// State for on-device transcription
const [transcriptText, setTranscriptText] = useState("");

// Check capability once
useEffect(() => {
  const supported = ExpoSpeechRecognitionModule.supportsOnDeviceRecognition();

  setTranscriptText(event.results[0]?.transcript ?? "");

// Listen for audio file URI (for fallback)

  setAudioFileUri(event.uri);
});

  console.warn("Speech recognition error:", event.error, event.message);
  if (audioFileUri) {
    // Fallback: upload the recorded audio to cloud

### Permission Request Pattern
```typescript
// Source: expo-speech-recognition docs
import { ExpoSpeechRecognitionModule } from "expo-speech-recognition";

async function requestSpeechPermissions(): Promise<boolean> {
  const mic = await ExpoSpeechRecognitionModule.requestMicrophonePermissionsAsync();
  if (!mic.granted) return false;

  const speech = await ExpoSpeechRecognitionModule.requestSpeechRecognizerPermissionsAsync();
  return speech.granted;
}
```

## Open Questions

1. **Speech recognition permission prompt UX**
   - What we know: iOS requires separate speech recognition permission (NSSpeechRecognitionUsageDescription). The project currently only requests microphone permission via expo-audio.
   - What's unclear: What's the best moment to request speech recognition permission? On first app launch? On first voice mode entry?
   - Recommendation: Request on first voice mode entry alongside microphone permission. If denied, silently fall back to cloud. This matches the existing pattern (mic permission requested on voice mode entry).

2. **On-device model availability on fresh iOS 26 install**
   - What we know: `SFSpeechRecognizer` on-device models may need to be downloaded on first use. `supportsOnDeviceRecognition()` returns true if the capability exists, but the model may not be installed yet.
   - What's unclear: Does `requiresOnDeviceRecognition: true` fail if the model isn't downloaded, or does iOS auto-download it?
   - Recommendation: Use `supportsOnDeviceRecognition()` as the gate. If on-device mode fails at runtime, the error handler falls back to cloud (the persisted audio file is available). Will's device (iOS 26.3.1) almost certainly has the model already.

3. **HITL voice follow-up with on-device transcription**
   - What we know: CONTEXT.md says voice follow-ups during HITL also use on-device transcription. Currently, voice follow-ups use `sendFollowUpVoice()` which uploads audio to the backend.
   - What's unclear: Should voice follow-ups with on-device transcription use `sendFollowUp()` (text endpoint) instead?
   - Recommendation: Yes. On-device voice follow-up should: transcribe on-device, then call `sendFollowUp()` with the transcribed text. This matches the pattern for initial captures.

4. **expo-audio coexistence**
   - What we know: `expo-audio` and `expo-speech-recognition` both need `AVAudioEngine`. They cannot run simultaneously.
   - What's unclear: Does merely having expo-audio installed (but not actively recording) interfere with expo-speech-recognition?
   - Recommendation: Should be fine -- only active recording causes conflicts. Keep expo-audio for the cloud fallback path. When on-device is active, do NOT call any expo-audio recording methods. Test this in the development build.
