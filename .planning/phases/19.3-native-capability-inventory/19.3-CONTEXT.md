<domain>
## Phase Boundary

Phase 19.2 shipped the transaction-first spine. Observability evolution from here (Phases 19.4 → 22) has one overriding design rule: **Built-In First.** Before any of those phases writes code, we must know what the nine native surfaces we depend on — App Insights, OpenTelemetry, Sentry.IO, Azure AI Foundry, Azure Monitor, Azure Cosmos DB, Azure Container Apps, GitHub Actions, Expo/EAS — already do for us. Without this inventory, every downstream phase re-litigates "should we build this or does Foundry/App Insights/etc. already do it?" in a memo cycle.

Phase 19.3 produces that inventory as a single deliverable document. It also runs three live-evidence checkpoints that will materially reshape downstream phase scope (most likely 19.5 and 21).

**In scope:**
- Audit of 9 native surfaces against how Second Brain uses them today.
- Three live-evidence checkpoints against the deployed system (Foundry trace depth, Foundry evals applicability, Sentry RN status).
- Single deliverable: `docs/superpowers/specs/native-observability-inventory.md`.
- Phase impact addendum: concrete scope deltas for 19.4 / 19.5 / 20 / 21 / 22.
- Two housekeeping todos created via `/gsd:add-todo` (mobile EAS rebuild; Key Vault dup cleanup).

**Out of scope:**
- Any production code changes. This is investigation only.
- Writing plans for 19.4 / 19.5 / 20 / 21 / 22. The inventory *feeds* those plans; it does not pre-decide their content.
- Evaluating third-party (non-built-in) alternatives (Ragas, Promptfoo, etc.).
- Re-investigating surfaces already well-understood from prior phases unless the spec cites them as load-bearing.
</domain>

<decisions>
## Implementation Decisions (LOCKED)

### The 9 Native Surfaces (locked scope)

From the spec, these are inventoried. No additions without a spec revision.

1. **Azure AI Foundry** — Tracing depth, Conversation UI, Evaluations SDK, Prompt-agent versioning
2. **OpenTelemetry** — span attributes, baggage propagation, log correlation
3. **Application Insights / Log Analytics** — auto-instrumentation surface, table schema, retention
4. **Azure Monitor** — Workbooks, Alerts, Action Groups (existing `SecondBrainAlerts`)
5. **Sentry.IO** — Python SDK, RN SDK, tags/breadcrumbs
6. **Azure Cosmos DB** — diagnostic logs (`AzureDiagnostics` + `activityId_g`), Change Feed
7. **Azure Container Apps** — revision metadata, built-in logs
8. **GitHub Actions** — cron schedules, artifact storage, workflow summaries
9. **Expo / EAS** — crash reports, update diagnostics

### Highest-leverage surfaces (per spec)

Three surfaces most likely to collapse downstream phase scope if found capable:
- **Foundry Evaluations SDK** → collapses Phase 21 from "write evaluators" to "configure evaluators"
- **Azure Monitor Workbooks** → may replace part of Phase 19.5's web RCA page with a workbook link
- **Cosmos Change Feed** → may replace Phase 20's per-call-site feedback instrumentation with a single listener

### Deliverable shape (locked)

Inventory document MUST follow this structure:

- Table-first, prose-second. Each row = one capability.
- Columns: *Surface | Capability | How Second Brain uses it today | What's still missing | Downstream phase(s) impacted*.
- At least 15 rows. Every row cites at least one downstream phase or it does not belong.
- Phase impact addendum at the end: one subsection per downstream phase (19.4, 19.5, 20, 21, 22) with explicit scope deltas.
- Three checkpoint findings documented with evidence (screenshot URL, KQL query result, trace link).

### Checkpoint discipline

Three checkpoints are MANDATORY before the inventory is considered complete:

- **Checkpoint A — Foundry trace depth.** Pull one recent capture's Foundry trace for each of the three agents. Inspect: prompt, output, tool args, confidence, conversation/run IDs. Record what's visible vs. absent.
- **Checkpoint B — Foundry evals applicability.** Use the Foundry Evaluations SDK to attempt to score at least one classifier output against a known label. Record: works / partial / doesn't work, with SDK version pinned.
- **Checkpoint C — Sentry RN status on deployed phone.** Trigger one mobile-side error on the installed EAS build. Confirm capture in Sentry with breadcrumb trail. Post-18-03 regression check.

Checkpoint findings are not optional. They are the highest-leverage rows in the inventory.

### Format of phase impact addendum (locked)

For each downstream phase (19.4, 19.5, 20, 21, 22), the addendum must state:
- **Original scope (from spec):** 1-sentence summary
- **Inventory findings that affect this phase:** bulleted list citing specific inventory rows
- **Scope delta:** one of `no change` / `smaller scope — <specifics>` / `larger scope — <specifics>` / `add spike memo — <ambiguity>`
- **Revised plan count estimate:** e.g., "8–10 → 7–8"
</decisions>

<out-of-scope>
## Explicitly Out of Scope

- Writing code against any inventoried surface. This phase is investigation only.
- Building prototypes or proofs of concept beyond what the three checkpoints require.
- Producing a fourth deliverable (e.g., a dashboard, a migration plan). The inventory document + phase impact addendum + three checkpoint findings are the complete output.
- Investigating third-party evaluation frameworks. "Built-In First" means built into platforms we already use.
- Revising downstream phase plans in their own files. Scope deltas live in the 19.3 addendum only. Phase 19.4's plan, when it is written, reads the addendum and incorporates findings.
- Investigating capabilities not on the 9-surface list. Additions require a spec revision.
</out-of-scope>
