## Summary

This phase has two distinct deliverables: (1) closing observability gaps so all three Foundry agents (Classifier, Admin, Investigation) emit detailed traces to App Insights as a single pane of glass, and (2) addressing all findings from the Codex code review plus adding Codex as a CI step.

The observability work has a clear path. The project already calls `enable_instrumentation()` from the Agent Framework SDK and has custom `AuditAgentMiddleware` + `ToolTimingMiddleware` producing OTel spans. What's missing is **Foundry-side visibility** -- the agent's internal thinking time, model inference duration, and tool routing decisions are opaque. The Agent Framework SDK's `enable_instrumentation()` already emits `gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`, and `gen_ai.client.operation.duration` as OTel metrics, and produces `invoke_agent`, `chat`, and `execute_tool` spans per the GenAI semantic conventions. These should already be flowing to App Insights via the `configure_azure_monitor()` exporter -- the key question is whether they're actually visible in the Agents (Preview) view, and whether `ENABLE_SENSITIVE_DATA` needs to be turned on for the detail level Will wants.

The Codex CI work is straightforward: the `openai/codex-action@v1` GitHub Action provides turnkey PR code review with a prompt file, security sandboxing, and output capture.

**Primary recommendation:** Audit what's already flowing to App Insights from `enable_instrumentation()`, fill gaps with env var toggles (`ENABLE_SENSITIVE_DATA`, `AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED`), add targeted alert rules, harden the health endpoint with active Foundry connectivity checks, and fix all Codex findings.

## Architecture Patterns

### Pattern 1: Agent Framework Instrumentation (Already Partially Active)

**What:** The Agent Framework SDK emits OTel spans and metrics when `enable_instrumentation()` is called. These include:
- `invoke_agent <agent_name>` -- top-level span per agent invocation
- `chat <model_name>` -- LLM inference call with token counts
- `execute_tool <function_name>` -- tool execution with timing
- `gen_ai.client.operation.duration` (histogram) -- operation duration
- `gen_ai.client.token.usage` (histogram) -- token counts

**Current state:** `enable_instrumentation()` is called in `main.py` after `configure_azure_monitor()`. These spans should already be flowing to App Insights. The custom `AuditAgentMiddleware` and `ToolTimingMiddleware` add additional spans (`classifier_agent_run`, `tool_{name}`) on top.

**What's missing:**
1. Agent names in spans are likely all "Classifier" (hardcoded in `AuditAgentMiddleware`), not differentiated per agent
2. `ENABLE_SENSITIVE_DATA` is not set -- prompts and responses are not captured
3. No verification that `invoke_agent`/`chat`/`execute_tool` spans actually appear in App Insights
4. The Agents (Preview) view in App Insights portal may not be connected

**How to verify:** Query App Insights for `AppDependencies | where Name startswith "invoke_agent" or Name startswith "chat" or Name startswith "execute_tool"` and check if spans exist.

**Source:** https://learn.microsoft.com/agent-framework/agents/observability

### Pattern 2: Connecting Foundry Portal to App Insights

**What:** AI Foundry can show traces in its portal by connecting to the same App Insights resource. This is a portal-side configuration (Agents > Traces > Connect).

**Current state:** Unknown -- needs verification in the portal.

**Why it matters:** The CONTEXT says "metrics land in App Insights, not Foundry portal" but connecting Foundry to the same App Insights resource lets traces appear in BOTH places simultaneously. This is additive, not a trade-off.

**Source:** https://learn.microsoft.com/azure/foundry/observability/how-to/trace-agent-setup

### Pattern 3: Content Recording for Debugging

**What:** Setting `AZURE_TRACING_GEN_AI_CONTENT_RECORDING_ENABLED=true` (for azure-core) or calling `enable_instrumentation(enable_sensitive_data=True)` (for Agent Framework) enables prompt/response content in spans.

**Decision needed:** Enable in production? For a single-user system with no PII concerns, the benefit (seeing exactly what the LLM received and responded) outweighs the storage cost. But it increases trace payload size.

**Recommendation:** Enable `ENABLE_SENSITIVE_DATA=true` as a Container App env var. The single-user nature of the system makes privacy/compliance a non-issue, and the debugging value is enormous.

### Pattern 4: Enhanced Health Endpoint with Active Foundry Check

**What:** Current `/health` just checks if `foundry_client is not None` (truthy check). It doesn't verify the client can actually reach Foundry. An active check would call `list_agents(limit=1)` (same as startup validation) with a short timeout.

**Implementation approach:**
```python
@router.get("/health")
async def health_check(request: Request) -> dict:
    foundry_client = getattr(request.app.state, "foundry_client", None)
    foundry_status = "not_configured"
    
    if foundry_client is not None:
        try:
            async with asyncio.timeout(5):
                async for _ in foundry_client.agents_client.list_agents(limit=1):
                    break
            foundry_status = "connected"
        except Exception:
            foundry_status = "degraded"
    
    overall = "ok" if foundry_status == "connected" else "degraded"
    return {"status": overall, "foundry": foundry_status, ...}
```

**Pitfall:** This adds latency to every health check. CI health polling hits this endpoint 15 times at 12s intervals. Consider caching the result for 60 seconds to avoid hammering Foundry.

### Pattern 5: Self-Healing Warmup

**What:** Current warmup sends a "ping" message to each agent every N minutes. If the ping fails, it logs a warning but doesn't recreate the client. Self-healing would recreate the `AzureAIAgentClient` on consecutive failures.

**Implementation approach:** Track failure count per agent in the warmup loop. After 2-3 consecutive failures, create a new `AzureAIAgentClient` with the same agent_id and replace `app.state.<agent>_client`. Must be thread-safe (use asyncio lock).

### Pattern 6: Codex GitHub Action for PR Review

**What:** `openai/codex-action@v1` runs Codex CLI in a GitHub Actions job. It installs the CLI, starts the Responses API proxy with your API key, and runs `codex exec` with a prompt.

**Configuration:**
```yaml
- uses: openai/codex-action@v1
  with:
    openai-api-key: ${{ secrets.OPENAI_API_KEY }}
    prompt-file: .github/codex/prompts/review.md
    sandbox: read-only
    safety-strategy: drop-sudo
```

**Trigger decision:** Run on PR events (not every push). Code review on pushes to main is wasteful -- PRs are the review surface.

**Security:** `safety-strategy: drop-sudo` (default) prevents Codex from accessing its own API key. `sandbox: read-only` prevents file changes.

**Source:** https://developers.openai.com/codex/github-action

### Anti-Patterns to Avoid
- **Mixing AIProjectInstrumentor with Agent Framework enable_instrumentation():** The project uses Agent Framework (not azure-ai-projects for agent orchestration). Don't import AIProjectInstrumentor -- it's for the azure-ai-projects SDK pattern.
- **Polling Foundry traces via API:** Don't build a polling mechanism to fetch traces from Foundry. The OTel exporter already sends them directly to App Insights.
- **Enabling content recording without understanding storage impact:** Full prompt/response recording multiplies trace storage. For a single-user system this is fine, but acknowledge the choice explicitly.

| Agent run spans | Custom middleware duplicating invoke_agent | Agent Framework's `enable_instrumentation()` | Already emits invoke_agent/chat/execute_tool spans per GenAI semantic conventions |
| Token usage tracking | Custom token counting | Agent Framework's built-in `gen_ai.client.token.usage` metric | Already tracked when instrumentation is enabled |

**Key insight:** The Agent Framework SDK already does most of the observability work. The gap is configuration (env vars, Foundry portal connection) and verification, not building new instrumentation.

## Common Pitfalls

### Pitfall 1: AuditAgentMiddleware Hardcodes Agent Name
**What goes wrong:** `AuditAgentMiddleware` always sets `agent.name` to `"Classifier"` in the span. All three agents (Classifier, Admin, Investigation) use this middleware, so all spans look like Classifier spans.
**Why it happens:** The middleware was written for the Classifier in Phase 7 and never parameterized for multi-agent use.
**How to avoid:** Parameterize the middleware with agent_name at construction time: `AuditAgentMiddleware(agent_name="investigation")`.
**Warning signs:** All agent spans in App Insights have `agent.name = "Classifier"`.

### Pitfall 2: Health Endpoint Latency from Active Foundry Check
**What goes wrong:** Active Foundry health check adds 1-5s latency per `/health` call. CI deploys poll 15 times at 12s intervals, which is fine, but if external monitoring also polls frequently, it increases Foundry API usage.
**Why it happens:** `list_agents(limit=1)` requires a round-trip to Foundry.
**How to avoid:** Cache the health check result with a TTL (e.g., 60s). Return the cached result for intermediate calls.
**Warning signs:** Health check latency > 5s in AppRequests.

### Pitfall 3: `enable_instrumentation()` Called Before `configure_azure_monitor()`
**What goes wrong:** If instrumentation is enabled before the Azure Monitor exporter is configured, spans are created but have nowhere to go.
**Current state:** main.py correctly calls `configure_azure_monitor()` first, then `enable_instrumentation()`. This is already correct. Don't change the order.

### Pitfall 4: Sentry FallbackRender Type Contract
**What goes wrong:** `Sentry.ErrorBoundary` passes `{ error: unknown, componentStack: string, eventId: string, resetError(): void }` to the fallback. If `ErrorFallback` types `error` as `Error`, TypeScript errors on `unknown is not assignable to Error`.
**How to fix:** Change `ErrorFallback` props to type `error` as `unknown`, then narrow inside the component: `const message = error instanceof Error ? error.message : String(error)`.
**Source:** Verified from `@sentry/react/build/types/errorboundary.d.ts` line 6: `error: unknown`.

### Pitfall 5: Recipe SSRF Tests Coupled to Live DNS
**What goes wrong:** `_is_safe_url()` calls `socket.getaddrinfo()` which fails in sandboxed/offline environments. Tests using `https://example.com/...` URLs fail because DNS resolution is attempted before any test mocks apply.
**How to fix:** Mock `socket.getaddrinfo` in tests to return a safe public IP (e.g., `93.184.216.34`), or extract the resolver into a dependency that can be injected.
**Current impact:** 6 failed tests in test_recipe_tools.py.

### Pitfall 6: Unawaited Coroutine Leak in Errands Tests
**What goes wrong:** `asyncio.create_task` is mocked with a `MagicMock`, but the coroutine passed to it is never awaited, producing a RuntimeWarning.
**How to fix:** Either patch `process_admin_capture` directly, or have the mock close/await the coroutine it receives. Simplest: use `AsyncMock(side_effect=lambda coro: coro.close())` for create_task.

### Example 4: Parameterized AuditAgentMiddleware

```python
class AuditAgentMiddleware(AgentMiddleware):
    def __init__(self, agent_name: str = "Classifier"):
        self._agent_name = agent_name
        self._span_name = f"{agent_name.lower()}_agent_run"

    async def process(self, context, call_next):
        with tracer.start_as_current_span(self._span_name) as span:
            span.set_attribute("agent.name", self._agent_name)
            start = time.monotonic()
            await call_next()
            elapsed_ms = (time.monotonic() - start) * 1000
            span.set_attribute("agent.duration_ms", elapsed_ms)
```
