  **Step 0: Preflight — validate KQL queries actually return data**

  CRITICAL: Before creating any alert rules, validate that the KQL queries work against the real schema. The project has a history of AppTraces vs AppDependencies schema mismatches (see Phase 17 debug notes and 14-03-SUMMARY.md). Run each KQL query manually first:

  ```bash
  # Look up workspace ID
  WORKSPACE_ID=$(az monitor log-analytics workspace show \
    --resource-group shared-services-rg \
    --workspace-name "$(az monitor app-insights component show --app second-brain-insights -g shared-services-rg --query 'workspaceResourceId' -o tsv | xargs -I{} basename {})" \
    --query customerId -o tsv)

  # Test Alert 1 KQL (should return rows if any agent calls >60s exist, or empty if not)
  az monitor log-analytics query \
    --workspace "$WORKSPACE_ID" \
    --analytics-query "AppDependencies | where TimeGenerated > ago(24h) | where Name startswith 'invoke_agent' or Name has '_agent_run' | take 5" \
    -o table

  # Test Alert 3 KQL (verify AppTraces has the expected Message patterns)
  az monitor log-analytics query \
    --workspace "$WORKSPACE_ID" \
    --analytics-query "AppTraces | where TimeGenerated > ago(24h) | where SeverityLevel >= 3 | project TimeGenerated, Message, SeverityLevel | take 5" \
    -o table
  ```

  If the preflight queries return schema errors or unexpected columns, adapt the alert KQL accordingly. If agent spans don't appear in AppDependencies yet (because Plan 02 changes haven't been deployed), note this and create the alerts anyway — they'll start firing once the middleware ships.

  **Step 1: Look up resource IDs**
  ```bash
  SUB_ID=$(az account show --query id -o tsv)
  APP_INSIGHTS_ID=$(az monitor app-insights component show --app second-brain-insights -g shared-services-rg --query id -o tsv)
  ACTION_GROUP_ID=$(az monitor action-group show --name SecondBrainAlerts -g shared-services-rg --query id -o tsv)
  ```

  **Step 2: Check `az monitor scheduled-query create --help`**

  Run `az monitor scheduled-query create --help` and read the actual syntax for `--condition`. The correct format requires a named condition with aggregation type, NOT bare `"count > 0"`. The typical pattern is:

  ```
  --condition "count 'QUERY_PLACEHOLDER' > 0"
  ```

  where `QUERY_PLACEHOLDER` matches a `--condition-query` name. If the CLI version uses a different syntax (e.g., `--condition-query` is the KQL and `--condition` is the threshold expression), adapt accordingly.

  If `az monitor scheduled-query create` proves too brittle, use `az rest` with the Azure Monitor Scheduled Query Rules REST API (2023-03-15-preview or later):

  ```bash
  az rest --method PUT \
    --url "https://management.azure.com/subscriptions/${SUB_ID}/resourceGroups/shared-services-rg/providers/Microsoft.Insights/scheduledQueryRules/Agent-Timeout?api-version=2023-03-15-preview" \
    --body @- <<'EOF'
  {
    "location": "eastus2",
    "properties": {
      "severity": 2,
      "enabled": true,
      "evaluationFrequency": "PT5M",
      "windowSize": "PT5M",
      "scopes": ["${APP_INSIGHTS_ID}"],
      "criteria": {
        "allOf": [{
          "query": "AppDependencies | where Name startswith 'invoke_agent' or Name has '_agent_run' | where DurationMs > 60000",
          "timeAggregation": "Count",
          "operator": "GreaterThan",
          "threshold": 0,
          "failingPeriods": { "numberOfEvaluationPeriods": 1, "minFailingPeriodsToAlert": 1 }
        }]
      },
      "actions": { "actionGroups": ["${ACTION_GROUP_ID}"] },
      "description": "An agent call took longer than 60 seconds"
    }
  }
  EOF
  ```

  Repeat for:
  - **Agent-Slow-Thinking** (severity 3): KQL = `AppDependencies | where Name startswith 'chat' | where DurationMs > 30000`, window PT5M
  - **Agent-Consecutive-Failures** (severity 2): KQL = `AppTraces | where SeverityLevel >= 3 | where Message has 'stream error' or Message has 'timed out' or Message has 'failed' or Message has 'Warmup ping failed' | summarize ErrorCount = count() by bin(TimeGenerated, 10m) | where ErrorCount >= 3`, window PT10M

  The executor should try `az monitor scheduled-query create` first. If it fails on syntax, fall back to `az rest`. Document whichever approach works in docs/instrumentation.md.

  ```markdown
  ## Agent Framework GenAI Spans

  The Agent Framework SDK emits OpenTelemetry spans for every agent interaction when `enable_instrumentation()` is called (in `main.py` after `configure_azure_monitor()`):

  | Span Name | What It Tracks | Where in App Insights |
  |-----------|---------------|----------------------|
  | `invoke_agent <agent_name>` | Full agent run lifecycle | AppDependencies |
  | `chat <model_name>` | LLM inference call (thinking time) | AppDependencies |
  | `execute_tool <function_name>` | Tool execution within agent run | AppDependencies |
  | `{agent_name}_agent_run` | Custom middleware span (classifier/admin/investigation) | AppDependencies (via OTel tracer) |
  | `tool_{function_name}` | Custom tool timing middleware | AppDependencies (via OTel tracer) |

  **Metrics automatically tracked:**
  - `gen_ai.usage.input_tokens` -- input tokens per call
  - `gen_ai.usage.output_tokens` -- output tokens per call
  - `gen_ai.client.operation.duration` -- operation duration histogram

  **Content recording:**
  Set `ENABLE_SENSITIVE_DATA=true` as a Container App env var to include prompt/response content in spans. This is appropriate for this single-user system and provides enormous debugging value.
  ```
