# Plan 17.4-02: Backend Observability Hardening Summary

**Parameterized agent middleware with distinct OTel spans, active Foundry health check with 60s cache, self-healing warmup after 3 consecutive failures, 30s investigation timeout**

## Performance

- **Duration:** 8 min
- **Tasks:** 3
- **Files modified:** 5
- **Files created:** 1

## Accomplishments
- All three agents (classifier, admin, investigation) now emit individually identifiable OTel spans in App Insights
- /health performs a real Foundry API call (list_agents with 5s timeout) instead of a truthy reference check, with 60s cache on app.state
- Warmup loop tracks consecutive failures per agent and recreates the AzureAIAgentClient after 3 failures via factory callbacks
- Investigation stream timeout reduced from 60s to 30s with a user-friendly error message
- 9 new tests in test_observability.py covering cache TTL, degraded status, self-heal, failure count reset, middleware parameterization, and timeout

## Decisions Made
- Used factory callbacks (`client_factories` dict) instead of accessing private SDK attributes for client recreation — cleaner and future-proof
- Cache lives on `request.app.state` not a module global — prevents cross-process leakage in tests
- Tasks 2 and 3 committed together since the test file was written after the implementation (agent ran out of context mid-plan, orchestrator completed)
