# Phase 17.2 Design — Investigation Agent Terminal Client

**Created:** 2026-04-09
**Status:** Approved, ready for planning
**Predecessor:** Phase 17 (Investigation Agent), Phase 17.1 (recent_errors polish)
**Source:** User request for conversational investigation access from Claude Code while Phase 18 (mobile chat) is deferred

---

## Goal

Give Will a natural-language conversational interface to the deployed Investigation Agent, usable from inside a Claude Code session via both:
1. An explicit slash command (`/investigate "question"`)
2. Auto-invocation when his natural-language message contains telemetry vocabulary or is a follow-up to a recent investigation

Multi-turn context is preserved across the conversation with Claude: the first call's `thread_id` is remembered in Claude's working context and reused for follow-ups unless Will starts a new investigation.

This is a small infrastructure phase. It adds a terminal client layer on top of the existing `POST /api/investigate` endpoint without touching backend behavior.

---

## Background

Phase 17 shipped the Investigation Agent SSE endpoint and Phase 17.1 polished the `recent_errors` tool's presentation. But until Phase 18 (mobile chat UI) ships, the only way to ask the agent questions is a raw `curl` with SSE parsing, which is painful for real-time investigation. Will explicitly stated in the brainstorming session that he intends to "try doing some investigations for the next day or so" before starting Phase 18.

The current gap: each investigation requires a hand-crafted curl command, manual Key Vault fetches, and visual parsing of a token-by-token SSE firehose. That workflow is so friction-heavy it discourages exploratory investigation — the opposite of what the agent was built for.

This phase fills that gap with a terminal client that handles:
- Key Vault authentication automatically
- SSE stream parsing into clean text
- Thread-id continuity across conversational turns
- Claude Code integration so Will can ask questions conversationally

## Scope

### In scope

- **`backend/src/second_brain/investigation_client.py`** — new library module with 4 units:
  - `fetch_api_key() -> str` (Key Vault fetch via `azure.identity` + `azure.keyvault.secrets`)
  - `stream_investigation(question, thread_id, api_key) -> InvestigationResult` (httpx POST + SSE parser)
  - `InvestigationResult` dataclass (text, tools_called, thread_id, elapsed_seconds, error, tool_errors, was_continued)
  - `format_response(result) -> tuple[str, str]` (builds stdout/stderr pair)
  - `run_investigation(question, thread_id) -> tuple[str, str, int]` (end-to-end entry point)
  - `_handle_event(event, result)` (internal SSE event dispatcher)
- **`backend/scripts/investigate.py`** — thin CLI wrapper (~30 lines): argparse, calls library, prints output, returns exit code
- **`.claude/commands/investigate.md`** — project-level slash command that instructs Claude how to shell out to the CLI and how to handle the output
- **`.claude/skills/investigate/SKILL.md`** — project-level skill with Rule B auto-invoke logic, Option X ambiguity handling, Sub-option Y status notes, and escape-hatch via `--new` flag / natural-language triggers
- **`backend/tests/test_investigation_client.py`** — 6 narrowly-scoped unit tests covering `format_response` and `_handle_event`
- Documentation: docstring header at the top of `investigation_client.py` explaining the CLI contract, URL, Key Vault secret name, and that the module is NOT imported by backend runtime code

### Out of scope (per Q6 decisions)

- Conversation history / transcript persistence
- Rich terminal formatting, colors, spinners, progress bars
- Export / save-answer commands
- Multiple concurrent threads
- Direct tool invocation bypass
- Cost / token tracking
- Retry on transient failure
- Caching of recent answers
- Correcting the agent's tool choices
- Installation outside this repo (skill and command are project-level, not user-level)
- End-to-end HTTP mock tests (covered by Layer 2 smoke tests instead)
- Unit tests for `fetch_api_key` (would require mocking Azure SDK for marginal value)

### Non-goals

- Replacing Phase 18's mobile chat UI. This is strictly a developer tool.
- Supporting any endpoint other than `POST /api/investigate`
- Any backend change. `investigation_client.py` is a read-only consumer.
- Persisting thread state across Claude Code sessions. Thread_id lives in Claude's conversation memory only.

## Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Helper code lives in `backend/src/second_brain/investigation_client.py` (library) + `backend/scripts/investigate.py` (CLI) | Approach 2 from brainstorming. Library inside the package gives clean imports for tests and future reuse. Script outside the package is a thin CLI wrapper. |
| 2 | Key Vault fetch on every invocation, no caching | Option A from Q2. ~500ms-1s latency is acceptable. Eliminates cache-invalidation bugs and env-var drift. Will be logged in with `az login` as a normal developer prereq. |
| 3 | Thread_id lives in Claude's conversation memory only, not on disk | A1 from Q3. Terminal client is ephemeral by design. If Will wants cross-session persistence, Phase 18's mobile UI covers that. |
| 4 | Implicit thread continuation (reuse remembered thread_id unless told otherwise) | B1 from Q3. Matches natural conversational flow. Escape hatch is `--new` flag and natural-language triggers like "new investigation", "start over", "fresh thread". |
| 5 | Auto-invoke shows one-line status note `[continuing previous thread]` before the agent's answer when thread is reused | Sub-option Y from Q3. Quiet visibility into thread state without a confirmation prompt. |
| 6 | Output is agent text verbatim + one-line status line below the answer | Option B from Q4. Clean answer is primary; tool list + thread state + elapsed time is auxiliary info below. |
| 7 | Errors go to stderr with `ERROR:` prefix; agent text (if any) stays in stdout | Sub-4b from Q4. Distinct format prevents error lines from polluting piped answer streams. |
| 8 | Auto-invoke triggers on explicit telemetry vocabulary OR follow-up pronouns referring to recent agent context | Rule B from Q5. Covers both "show me errors" and "tell me more about step 4" without overreaching into code/dev questions. |
| 9 | Ambiguous questions trigger a confirmation prompt before invoking | Option X from Q5. "Do you want me to check the deployed system for this, or look at the code/tests?" |
| 10 | Slash command `/investigate` is always available as explicit escape hatch | Q5 decision 3. Regardless of natural-language rules, explicit invocation always works. |
| 11 | Machine-readable `[THREAD_ID: ...]` marker on last line of stdout; Claude strips before display | Option Y from Q2 Section 2. Decouples thread extraction from the human-readable status line format. |
| 12 | 6 unit tests scoped to `format_response` and `_handle_event` only | Q3 of Section 3. No tests for Azure SDK or httpx wrappers — they'd test the mocks, not our logic. Layer 2 smoke tests cover the integration surface. |
| 13 | Slash command and skill are project-level (`.claude/commands/`, `.claude/skills/`), not user-level | Scope item #10 from Q6. Tool is bound to the repo it investigates. |
| 14 | 60-second request timeout on the client side, matching the backend's own investigation timeout | Prevents client-side hang if backend or network is broken. Backend usually returns a graceful timeout event first. |
| 15 | Empty agent responses (zero `text` events) are NOT treated as errors | Some legitimate responses have no text (rate-limited, agent confused by prompt). Exit code stays 0, status line still prints. |
| 16 | Tool errors inside a stream keep the agent text in stdout AND write error lines to stderr, exit code 1 | The agent's narrative answer is usually still useful context even when one tool fails. Suppressing it would hide information. |
| 17 | `investigation_client.py` MUST NOT be imported by backend runtime code | Verified in phase completion gate 7 (grep check). Transitive httpx / Key Vault deps on the hot path are unnecessary. |

## Architecture

### Component diagram

```
You type either:
  /investigate "show me errors"                    (explicit slash command)
  "what happened to my last capture?"              (natural language)
                          |
                          v
    +-----------------------------------------+
    | Claude Code dispatches to:              |
    |   .claude/commands/investigate.md       |  (explicit)
    |   .claude/skills/investigate/SKILL.md   |  (auto-invoke)
    +-----------------------------------------+
                          |
                          v
    +-----------------------------------------+
    | I (Claude) build the Bash command:      |
    |                                         |
    |  cd backend && uv run python \          |
    |    scripts/investigate.py \             |
    |    "<question>"                         |
    |    [--thread <last_thread_id>]          |
    |    [--new]                              |
    |                                         |
    | I remember last thread_id from my       |
    | conversation context                    |
    +-----------------------------------------+
                          |
                          v
    +-----------------------------------------+
    | CLI: backend/scripts/investigate.py     |
    | - argparse: question, --thread, --new   |
    | - imports library                        |
    | - calls run_investigation(q, thread_id) |
    | - prints stdout/stderr                  |
    | - exit 0 success / 1 error              |
    +-----------------------------------------+
                          |
                          v
    +-----------------------------------------+
    | Library:                                |
    |   second_brain.investigation_client     |
    |                                         |
    | run_investigation(q, thread_id)         |
    |   ↓                                     |
    |   fetch_api_key() via Key Vault         |
    |   ↓                                     |
    |   stream_investigation()                |
    |     → httpx POST /api/investigate       |
    |     → parse SSE lines                   |
    |     → _handle_event() dispatches to     |
    |       InvestigationResult dataclass     |
    |   ↓                                     |
    |   format_response()                     |
    |     → (stdout_text, stderr_text)        |
    +-----------------------------------------+
                          |
                          v
    +-----------------------------------------+
    | Deployed backend:                       |
    |   POST /api/investigate                 |
    |   Headers: Authorization: Bearer <key>  |
    |   Body: {question, thread_id?}          |
    |   Returns: SSE stream                   |
    +-----------------------------------------+
                          |
                          v  (SSE events flow back)
                          |
                          v
    +-----------------------------------------+
    | Claude reads CLI stdout/stderr:         |
    | - Strips [THREAD_ID: ...] marker line   |
    | - Remembers thread_id for next turn     |
    | - Shows agent text + status line to you |
    | - Shows stderr if any                   |
    +-----------------------------------------+
```

### Library module structure

```python
# investigation_client.py (~180 lines total)

# Constants
INVESTIGATE_URL = "https://brain.willmacdonald.com/api/investigate"
KEY_VAULT_URL = "https://wkm-shared-kv.vault.azure.net/"
SECRET_NAME = "second-brain-api-key"
TIMEOUT_SECONDS = 60.0

# Dataclass
@dataclass
class InvestigationResult:
    text: str = ""
    tools_called: list[str] = field(default_factory=list)
    thread_id: str | None = None
    elapsed_seconds: float = 0.0
    error: str | None = None
    tool_errors: list[dict] = field(default_factory=list)
    was_continued: bool = False

# Four public functions
def fetch_api_key() -> str: ...
def stream_investigation(question, thread_id, api_key, *, url, timeout) -> InvestigationResult: ...
def format_response(result: InvestigationResult) -> tuple[str, str]: ...
def run_investigation(question, thread_id) -> tuple[str, str, int]: ...

# One private helper
def _handle_event(event: dict, result: InvestigationResult) -> None: ...
```

### SSE event handling table

| Event type | Handling |
|---|---|
| `text` | Append `content` to `result.text` |
| `tool_call` | Add `tool` name to `result.tools_called` (deduped) |
| `tool_error` | Append `{tool, error}` dict to `result.tool_errors` |
| `done` | Set `result.thread_id` from event payload |
| `error` | Set `result.error` from event's `message` field |
| `thinking` | Ignored (not meaningful for CLI output) |
| `rate_warning` | Ignored (not meaningful for CLI output) |

### CLI output format

**stdout (on success):**
```
<agent's answer text, verbatim, possibly with markdown tables>

[tools: recent_errors, system_health | thread: thread_abc123de... (continued) | 2.3s]
[THREAD_ID: thread_abc123def456...]
```

**stdout (on success, no tools called):**
```
<agent's answer text>

[no tools | thread: thread_abc123de... (new) | 0.8s]
[THREAD_ID: thread_abc123def456...]
```

**stderr (on tool error):**
```
TOOL ERROR (recent_errors): Log Analytics workspace unreachable
```

**stderr (on Key Vault fetch failure):**
```
ERROR: Failed to fetch API key from Key Vault (second-brain-api-key). Check that you are logged in with 'az login'. Original: <exc>
```

**stderr (on HTTP 5xx):**
```
ERROR: HTTP 503: {"detail":"Investigation agent is unavailable."}
```

### Slash command dispatch

When Will types `/investigate "question"` or `/investigate "question" --new`, Claude Code loads `.claude/commands/investigate.md` which instructs Claude to:

1. Build the Bash command:
   `cd backend && uv run python scripts/investigate.py "<question>"` with optional `--thread <last_id>` (if Claude remembers one and `--new` was NOT passed) or `--new` (if explicitly passed)
2. Run via Bash tool
3. Parse stdout: extract the `[THREAD_ID: ...]` line, strip it before display
4. Show the remaining stdout verbatim (agent answer + status line)
5. Show stderr verbatim if present
6. Remember the extracted thread_id for next turn

### Skill auto-invocation (Rule B)

When Will types a natural-language message (no slash command), Claude Code loads `.claude/skills/investigate/SKILL.md` which instructs Claude to:

1. Check for explicit telemetry vocabulary: "errors", "exceptions", "failures", "crashes", "broken", "captures", "last capture", "most recent capture", "traces", "trace ID", "system health", "latency", "p95", "p99", "success rate", "classifier <behavior>", "admin agent <behavior>", "inbox", "bucket", "how many X this <week/day/hour>", "what happened with Y", "is the system <healthy/broken/slow/degraded>"
2. OR check for follow-up pronouns referring to recent agent context in conversation history: "tell me more about that", "why did THAT happen", "step 4", "go deeper on the first error", "what about yesterday instead"
3. If ambiguous (partial vocabulary but could be code/tests), ask first: "Do you want me to check the deployed system for this, or look at the code/tests?"
4. If clearly not an investigation (read file, run tests, git state, planning, etc.), do NOT invoke
5. On invocation, follow the same Bash + stdout/stderr handling as the slash command
6. Print `[continuing previous thread]` before the agent's answer if thread was reused
7. Handle escape phrases ("new investigation", "start over", "fresh thread", "forget that") by dropping the remembered thread_id and passing `--new`

## Risks & mitigations

| # | Risk | Mitigation |
|---|------|------------|
| 1 | `[THREAD_ID: ...]` marker format changes break Claude's parser | Rigid format with exact prefix, dedicated line at end of stdout. Regex `^\[THREAD_ID: (.+)\]$` is unambiguous. Unit test 3 pins the exact format. |
| 2 | Auto-invoke wrongly triggers on ambiguous questions | Option X mitigation: when in doubt, ask first. Scope #9 from Q6. If still wrong, user can respond "no, check the code" and Claude aborts the investigation path. |
| 3 | Thread_id stale across long conversations (agent forgets context after N turns) | Not a problem we can fix client-side — Foundry Assistants API manages context window limits naturally. If the agent forgets, a fresh thread via `--new` recovers. |
| 4 | Key Vault fetch latency (500ms-1s per invocation) makes investigations feel slow | Accepted per Q2 decision. Optimization is a future concern if Will finds it painful. |
| 5 | httpx or azure-keyvault-secrets missing from local dev dependencies | `azure-keyvault-secrets` is already a main dep (`pyproject.toml` line 14, used by `main.py` Key Vault fetch). `httpx` is listed as a test dep (`pyproject.toml` line 43), but `uv run python` in a standard local dev checkout DOES import it successfully (verified during design, httpx 0.28.1). This is because `uv sync` in dev environments by default installs test extras. The CLI script is never run inside the deployed container — only in local dev — so the dep placement is acceptable. Implementation plan should confirm `cd backend && uv run python -c "import httpx"` works before writing the CLI script; if it fails on a clean machine, promote httpx to main deps. |
| 6 | Importing `investigation_client` at package level risks pulling httpx/azure deps into backend hot path | Enforced by gate 7: grep check ensures no runtime import. Module is only imported by `scripts/investigate.py` and tests. |
| 7 | Library imports fail on fresh machine because `uv sync` didn't install dev deps | Document in the slash command's error handling: if import fails, suggest `cd backend && uv sync` before re-running. |
| 8 | Claude auto-invokes the skill but shells out with an unquoted question containing shell metachars | Always quote `"$ARGUMENTS"` exactly in the Bash command. Note in skill instructions. |
| 9 | Conversation history pollution: agent gets confused by unrelated topics in a long thread | Escape hatch via `--new` flag and natural-language triggers. Will can start a fresh investigation any time. |
| 10 | The skill + command file is loaded by Claude even when Phase 18 ships and makes this redundant | Phase 18 gets its own mobile UI; the terminal skill remains useful for command-line debugging. No conflict. If Phase 18 adds its own project-level skill, we can deprecate this one then. |

## Definition of Done

The phase is complete when ALL SEVEN gates pass:

1. **Unit tests pass** — `cd backend && uv run pytest tests/test_investigation_client.py -v` shows 6 passes, 0 failures
2. **No regressions in backend suite** — `cd backend && uv run pytest -m "not integration"` still shows the current green count (176 + 6 new = 182 expected)
3. **Smoke Test 1 (simple text query)** — `cd backend && uv run python scripts/investigate.py "how is the system doing?"` returns an agent response with `system_health` in the status line, `[THREAD_ID: ...]` marker at end, exit 0
4. **Smoke Test 2 (multi-turn)** — Two sequential CLI calls with `--thread <id>` on the second show the agent's second response references content from the first (proving thread continuity)
5. **Smoke Test 3 (error path)** — CLI with a clearly invalid argument (e.g., empty string) returns stderr with an error, exit 1
6. **Skill/command integration tests A-G pass** — Manual verification by Will after shipping; seven tests cover explicit invoke, pronoun continuation, ambiguity, non-investigation questions, fresh thread triggers, and az login error handling
7. **Grep check** — `grep -rn "investigation_client" backend/src/` returns matches ONLY in `investigation_client.py` itself and NOT in any other `.py` file (confirming no runtime imports)

If any gate fails, the phase is not done.
