### E2E Flows: 7/7 complete

1. **Text Capture to Errand** — mobile → capture → classifier → Admin Agent → Errands container
2. **Admin Processing via Status Screen** — Status tab focus → GET /api/errands triggers processing → delete inbox → poll refresh
3. **HITL Routing for Unrouted Items** — unrouted destination → RoutingPicker → POST /api/errands/{id}/route → AffinityRule auto-save
4. **Recipe URL Extraction** — paste URL → classifier → Admin Agent → fetch_recipe_url → add_errand_items with source attribution → retry mechanism
5. **Admin Notifications** — rule management response → delivery heuristic → notification banners → dismiss endpoint
6. **Mobile Trace ID End-to-End** — generateTraceId → X-Trace-Id header → ContextVar → Cosmos doc → admin processing → App Insights
7. **On-Device Voice Transcription** — SFSpeechRecognizer → real-time transcript → sendCapture with X-Capture-Source → cloud fallback

### Integration Regression

**INT-01: Retry query missing failed/pending conditions**
- **Location:** `backend/src/second_brain/api/errands.py:178-179`
- **Issue:** The processing trigger query only checks `NOT IS_DEFINED(c.adminProcessingStatus) OR IS_NULL(c.adminProcessingStatus)`. Items with `adminProcessingStatus = 'failed'` or `'pending'` are never retried when the user opens the Status screen.
- **Root cause:** Phase 12.1-02 added the fix (commit c9f7e57), but it was likely lost during Phase 12.2 (errands rename) which rewrote the query file.
- **Impact:** Non-critical. Failed items show red "Processing failed" indicator in Inbox but are never auto-retried. User can delete manually.
- **Affected requirement:** CLEAN-01 (partially — delete-on-success works, retry path doesn't)
